## 1. Phase D executor — read-side promotion to row-source execution

### D-E1. Single-table grouped queries on row sources

**Problem:** the narrow `simple_grouped_count_result_from_*` and `simple_grouped_numeric_aggregate_result_from_*` fast paths exist, but the generic `evaluate_select -> build_select_dataset -> evaluate_grouped_select` fallback still builds a full source `Dataset` in memory before grouping.

**Implement:**
1. Add a new safe grouped read helper in `crates/decentdb/src/exec/mod.rs` that executes **single-base-table grouped queries** directly from `VisibleTableRowSource`.
2. Scope:
   - one base table in `FROM`
   - no lateral / no window functions / no recursive CTE / no view or temp table inputs
   - expressions resolving only against the one table binding plus aggregate outputs
3. Run it **after** the existing narrow grouped fast paths miss.
4. Maintain only per-group aggregate state, per-group rewritten output for `HAVING`/`ORDER BY`, and final output rows. **Do not materialize every input row into a `Dataset`.**
5. Keep result-level `ORDER BY`, `LIMIT`, `OFFSET` behavior identical.

**Files:** `crates/decentdb/src/exec/mod.rs`, grouped-query test suites.

**Tests:**
- Reopen-time grouped query on a paged table that currently misses narrow fast paths.
- Same query inside a shared SQL transaction.
- Grouped query with `HAVING` + result `ORDER BY`.
- Mixed aggregate projections not handled by current specialization.

**Exit condition:** tested grouped shapes stay off resident full-table materialization after reopen.

---

### D-E2. Base-table joins on row sources

**Problem:** `evaluate_from_item_in_scope` still evaluates many joins by building left and right `Dataset`s and calling `nested_loop_join(...)`. The indexed join path is good; the fallback is the footgun.

**Implement:**
1. Add a row-source join helper for **base-table non-lateral joins**: `INNER`, `LEFT`, `RIGHT`, `FULL`, with `ON`, `USING`, `NATURAL`.
2. Keep the indexed join fast path first. Use the new helper only when the indexed path misses.
3. Scope to joins whose inputs are already base tables loaded in the snapshot-local runtime clone.
4. Build join output rows incrementally. Materializing the join output is OK; building two full source `Dataset`s first is not.
5. Route the snapshot-local deferred execution path in `db.rs` to this helper before the broad generic fallback.

**Files:** `crates/decentdb/src/exec/mod.rs`, `crates/decentdb/src/db.rs`.

**Tests:**
- Reopen-time paged-table `INNER JOIN`.
- `LEFT JOIN` / `RIGHT JOIN` with `ON` predicates and result ordering.
- `JOIN USING` / `NATURAL JOIN` hidden-column semantics.
- Shared-transaction versions.

**Exit condition:** base-table join queries no longer require full source-table `Dataset` construction from reopened paged tables.

---

### D-E3. Route safe `SELECT` execution to the row-source lane

**Problem:** `db.rs::try_execute_query_with_row_sources_at_snapshot` already clones a snapshot-local runtime and loads only referenced row sources, but it still delegates to the generic executor. After D-E1/D-E2 land, the routing must prefer the new helpers.

**Implement:**
1. Extend row-source safe-query routing so:
   - single-table grouped queries use D-E1
   - base-table joins use D-E2
   - only unsupported features drop to the broad generic resident path
2. Keep recursive CTEs, lateral joins, view expansion, temp tables, window functions on the old executor.
3. Add explicit guards: every query is either handled fully on the row-source lane, or explicitly rejected to the old fallback. No accidental fallthrough.

**Files:** `crates/decentdb/src/db.rs`, `crates/decentdb/src/exec/mod.rs`.

**Tests:**
- Reopen-time query that previously loaded only because routing missed a now-supported path.
- Shared-transaction version.
- Negative coverage: unsupported feature still falls back cleanly.

**Exit condition:** deferred read path reaches row-source helpers instead of the broad resident fallback.

---

### D-E4. Remove post-processing blockers for row-source shapes

**Problem:** `apply_select_distinct`, `sort_dataset`, `project_dataset`, and `evaluate_set_operation` still re-materialize full copies even when the source scan was already row-source safe.

**Implement:**
1. For the query shapes covered by D-E1 and D-E2, keep post-processing bounded:
   - reuse result-level bounded-order helpers where they exist
   - avoid re-materializing a second full copy when the first already exists
2. Do **not** solve full external sort or spill in this slice.
3. Leave recursive CTE and generic set-op internals alone unless they block the new row-source route.

**Files:** `crates/decentdb/src/exec/mod.rs`

**Exit condition:** new grouped and join lanes do not lose their memory win in a second post-processing copy step.

## 2. Phase D mutations — eliminate resident-image fallbacks

### D-M1. Row-id keyed mutation deltas for paged tables

**Problem:** generic `UPDATE`/`DELETE` fallbacks in `exec/dml.rs` still call `table_data_mut(...)` and `mark_table_row_dirty(..., row_index, ...)` when the narrow paged fast path misses (e.g. `RETURNING` with index updates, FK actions). This forces a Paged table to hydrate into a Resident image.

**Implement:**
1. Replace the resident-centric update/delete dirty split with one row-id keyed mutation-delta model for persisted base tables. The delta must distinguish:
   - appended rows
   - rewritten rows by `row_id`
   - deleted `row_id`s
2. `row_index` must stop being the persistence identity for paged tables. It may still be used for resident-table in-memory edits, but paged persistence data must be keyed by `row_id`.
3. Keep temp-table behavior unchanged.

**Files:** `crates/decentdb/src/exec/mod.rs`, `crates/decentdb/src/exec/dml.rs`.

**Exit condition:** paged-table persistence has enough information to rewrite from row-id keyed deltas without consulting a full resident end-state table.

---

### D-M3. Generic DML fallbacks onto the paged mutation model

**Problem:** `execute_update(...)`, `execute_delete(...)`, `apply_conflict_update(...)`, and `render_returning(...)` still require a Resident `TableData` image when the narrow paged helper misses.

**Implement:**
1. Keep existing paged fast paths first.
2. After D-M2 (already done) and D-M1 land, move generic paged `UPDATE`, `DELETE`, and `UPSERT DO UPDATE` branches to the row-id keyed paged mutation model instead of `table_data_mut(...)`.
3. `RETURNING` may still materialize returned rows, but it must do so from the row-change set, not from the full resident table image.
4. Preserve trigger behavior, FK action behavior, index freshness, and re-defer touched persisted tables after autocommit writes exactly as the existing control plane does.

**Files:** `crates/decentdb/src/exec/dml.rs`, `crates/decentdb/src/db.rs`.

**Tests:**
- Reopen-time paged `UPDATE ... RETURNING`.
- Reopen-time paged `DELETE ... RETURNING`.
- Reopen-time paged `INSERT .. ON CONFLICT DO UPDATE RETURNING`.
- Shared-transaction versions where applicable.

**Exit condition:** generic paged DML no longer needs resident `TableData` for persistence or `RETURNING` semantics.

## 3. Phase E cleanup — remove obsolete compatibility paths

### E-C1. Remove deferred-mode broad-load fallbacks

**Problem:** `db.rs` still contains broad-load fallbacks (`refresh_and_ensure_all_tables_loaded`, `ensure_runtime_all_tables_loaded_at_snapshot`) whose only reason to exist was that executor/mutation paths were not fully row-source capable yet.

**Implement:**
1. Audit every deferred-mode caller in `db.rs`.
2. For each caller, either convert it to targeted row-source loading, or leave an explicit comment that it is intentionally unsupported and why.
3. After D-E and D-M land, no "safety" fallback should broad-load all tables for the query and write shapes covered by this document.

**Exit condition:** deferred-mode broad all-table loads are no longer part of normal grouped, join, or paged DML execution.

---

### E-C2. Shrink `EngineRuntime::tables` to post-Phase-D role

**Problem:** `EngineRuntime::tables` stores `TableRowSource`, but too many helpers still conceptually treat it as "the place every base table row lives."

**Implement:**
1. Narrow resident-only helpers so their names and call sites reflect reality.
2. Remove dead compatibility shims that bridged the old `Arc<TableData>` world into the `TableRowSource` world.
3. Keep temp-table helpers intact.

**Files:** `crates/decentdb/src/exec/mod.rs`, `crates/decentdb/src/exec/dml.rs`.

**Exit condition:** the runtime no longer implies that every persisted base table has a resident `TableData` image available by default.

---

### E-C3. Delete old resident-image rewrite helpers and stale comments

**Problem:** once D-M2 and D-M3 are in place, resident-image paged rewrite helpers are dead code.

**Implement:**
1. Delete any remaining old resident-image paged rewrite helpers (verify `rewrite_paged_table_from_resident` is no longer needed for mutations; it is a legitimate legacy-to-paged format conversion, not a mutation helper, but confirm it is only used for that).
2. Delete old row-update/row-delete persistence branches that existed only for the temporary compatibility model.
3. Remove stale comments describing M4, ADR 0141, or ADR 0142 as scaffolds or future work.

**Exit condition:** one paged mutation persistence model exists; dead compatibility branches are gone.

## 5. Definition of done

The full engine-memory task is done only when all of the following are true:

1. Safe grouped and join queries no longer need full resident source-table materialization.
2. Paged `UPDATE` / `DELETE` / `UPSERT DO UPDATE` persistence no longer derives new paged state from a resident end-state table image.
3. Deferred-mode broad-load compatibility fallbacks for those shapes are removed.
4. `python ./scripts/do-pre-commit-checks.py` is green.
5. The memory, benchmark, migration, and crash thresholds above are met.

## 6. Appendix: related but non-blocking follow-ups

These items are not part of the blocking task list but are related findings that should be tracked separately after Phase D/E land:

- **Page-cache LRU bounding** — probe shows `cache_size_mb=4` may not actually cap RSS.
- **Dirty-page residency** — pager may hold thousands of dirty pages in heap between commits.
- **Executor scratch buffers** — hash aggregate / sort / `GROUP BY` still allocate `Vec<...>` proportional to group cardinality. Out of scope for this document; needs its own ADR for spill-to-disk via `DbConfig::temp_dir` and a new `DbConfig::executor_memory_mb` limit.
