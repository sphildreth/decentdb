## 1. Priorities (reminder)

1. Durable ACID writes.
2. Fast reads.
3. Stable, ergonomic multi-language integrations.

This batch does not touch the on-disk format, WAL format, or C ABI. Every new
Dart API call lands on a `ddb_*` symbol that already exists in
`include/decentdb.h`.

## Slice D1 — Expose `ddb_stmt_bind_uuid`

**Severity:** High (feature gap; UUID is a first-class engine type)

### Problem

The Rust engine stores UUIDs as a distinct `Value::Uuid([u8; 16])` variant and
canonicalises text parsing (`crates/decentdb/src/exec/mod.rs:12709-12735`). The
decode path in Dart already handles the UUID tag
(`bindings/dart/dart/lib/src/statement.dart:942-947`), but **the bind path is
missing**:

- `include/decentdb.h:161-164` declares
  `ddb_status_t ddb_stmt_bind_uuid(ddb_stmt_t *stmt, size_t index_1_based, const uint8_t *bytes_16);`
- `bindings/dart/dart/lib/src/native_bindings.dart` has **no**
  `bindUuid`-style FnPtr.
- `bindings/dart/dart/lib/src/statement.dart:336-347` (`bind()`) has no UUID
  dispatch; callers today must fall back to binding UUIDs as TEXT or BLOB,
  forcing the engine to re-parse on every call.

`decent-bench` uses UUIDs liberally and this gap forces a TEXT round-trip on
every UUID bind.

### Files & exact changes

1. `bindings/dart/dart/lib/src/native_bindings.dart`
   - Add typedefs next to existing statement-bind typedefs:
     ```dart
     typedef _StmtBindUuidC = Uint32 Function(
       Pointer<DdbStmt> stmt, IntPtr index, Pointer<Uint8> bytes16);
     typedef _StmtBindUuidDart = int Function(
       Pointer<DdbStmt> stmt, int index, Pointer<Uint8> bytes16);
     ```
   - Add a `late final _StmtBindUuidDart stmtBindUuid;` field.
   - In the lookup block (near lines 633–777 where other bind FnPtrs are
     resolved), add:
     ```dart
     stmtBindUuid = lib.lookupFunction<_StmtBindUuidC, _StmtBindUuidDart>(
       'ddb_stmt_bind_uuid');
     ```

2. `bindings/dart/dart/lib/src/types.dart`
   - Add a `UuidValue` newtype mirroring the shape of `DecimalValue`
     (`:25-40`):
     ```dart
     /// Canonical 16-byte UUID. Bytes are stored in native order (hex byte 0
     /// first). Use [UuidValue.parse] to construct from canonical text
     /// ("xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx").
     final class UuidValue {
       final Uint8List bytes;
       UuidValue(this.bytes) {
         if (bytes.length != 16) {
           throw ArgumentError('UuidValue requires exactly 16 bytes');
         }
       }
       factory UuidValue.parse(String text) { ... }
       String toText() { ... }
       @override bool operator ==(Object other) => ...; // byte-wise equal
       @override int get hashCode => Object.hashAll(bytes);
       @override String toString() => toText();
     }
     ```
   - `parse` must accept the canonical 36-character hyphenated form and reject
     anything else with `FormatException`. `toText` must emit lowercase hex.

3. `bindings/dart/dart/lib/src/statement.dart`
   - Add `void bindUuid(int index, Uint8List bytes)`:
     - validate `bytes.length == 16`; throw `ArgumentError` otherwise;
     - allocate 16 bytes via `malloc<Uint8>(16)` (not `calloc`), copy with
       `setAll(0, bytes)`, invoke `stmtBindUuid`, free in `finally`;
     - on non-zero return code, call the existing error-throwing helper.
   - Extend `bind(int index, Object? value)` (`:336-347`): add a
     `UuidValue` branch **before** the fallback `Uint8List → bindBlob` branch
     so that a bare `Uint8List` still binds as BLOB (back-compat). Also accept
     `UuidValue` in `bindAll` via the same dispatcher.

4. `bindings/dart/dart/lib/src/decentdb.dart`
   - Re-export `UuidValue` from `types.dart` (same line as `DecimalValue`).

### Acceptance criteria

- Compiling `bindings/dart/dart/` produces no analyzer warnings.
- A new unit test binds `UuidValue` values and round-trips them through a
  `UUID` column.
- A `Uint8List` of length 16 still binds as BLOB (back-compat).
- Passing a `UuidValue` whose `bytes.length != 16` throws `ArgumentError`.

### Unit tests to add

New file `bindings/dart/dart/test/uuid_bind_test.dart`:

- `binds UuidValue into a UUID column and round-trips exact bytes`:
  create table `t(id UUID PRIMARY KEY, v UUID)`, bind two distinct
  `UuidValue`s, select them back, assert byte-exact equality.
- `parses and formats canonical text`: `UuidValue.parse('...').toText() == '...'`.
- `rejects malformed UUID text` — `FormatException`.
- `rejects UuidValue with wrong byte length` — `ArgumentError`.
- `bare Uint8List still binds as BLOB` — round-trip a 16-byte `Uint8List`
  into a `BLOB` column and verify it reads back as `Uint8List` (not
  `UuidValue`).

## Slice D2 — Expose `ddb_db_inspect_storage_state_json`

**Severity:** High (required for decent-bench diagnostics)

### Problem

`include/decentdb.h:274` declares
`ddb_status_t ddb_db_inspect_storage_state_json(ddb_db_t *db, char **out_json);`.
The symbol is not bound in Dart. `decent-bench` cannot surface pager/WAL/cache
state without a text-parsed fallback.

### Files & exact changes

1. `bindings/dart/dart/lib/src/native_bindings.dart`
   - Add typedefs:
     ```dart
     typedef _DbInspectStorageStateC = Uint32 Function(
       Pointer<DdbDb> db, Pointer<Pointer<Utf8>> outJson);
     typedef _DbInspectStorageStateDart = int Function(
       Pointer<DdbDb> db, Pointer<Pointer<Utf8>> outJson);
     ```
   - Add `late final _DbInspectStorageStateDart dbInspectStorageStateJson;`
     and resolve it against `'ddb_db_inspect_storage_state_json'`.

2. `bindings/dart/dart/lib/src/types.dart`
   - Add a plain-Dart-class `StorageStateSnapshot` with fields mirroring the
     JSON keys emitted by the engine. Verify current shape by calling the
     function in a test and decoding with `dart:convert`'s `jsonDecode`. At
     minimum capture: `pageCount`, `pageSize`, `walSizeBytes`,
     `walVersions`, `cacheEntries`, `cacheCapacity`, `checkpointSequence`,
     and preserve the raw JSON text as `rawJson` for forward compatibility.
   - Add `factory StorageStateSnapshot.fromJson(Map<String, Object?>)` that
     is lenient: unknown keys are ignored, missing numeric keys default to
     0, but any numeric-typed key that is present and non-numeric throws
     `FormatException`.

3. `bindings/dart/dart/lib/src/database.dart`
   - Add `StorageStateSnapshot inspectStorageState()`:
     - call `dbInspectStorageStateJson`;
     - on non-zero return, throw via the existing error helper;

       through the existing `stringFree` binding, and parse with
       `jsonDecode`.

4. `bindings/dart/dart/lib/src/decentdb.dart` — re-export
   `StorageStateSnapshot`.

### Acceptance criteria

- Calling `db.inspectStorageState()` on an open database returns a snapshot
  whose `pageSize` equals the configured page size and whose `walSizeBytes
  >= 0`.
- `rawJson` is non-empty and round-trips through `jsonDecode` without
  exception.

### Unit tests to add

New file `bindings/dart/dart/test/storage_state_test.dart`:

- `returns page and wal state after a write`: open DB, create table, insert
  rows, assert `pageCount > 0`, `walSizeBytes > 0`.
- `returns zero-allocation state on a fresh :memory: DB`: fields are either
  zero or documented non-zero initial values (assert specifically, no
  `>= 0` loopholes — pin the value).
- `rawJson parses as JSON`: `jsonDecode(snapshot.rawJson)` is a
  `Map<String, Object?>`.

**Severity:** High (parity with other bindings)

### Problem

`bindings/dart/dart/lib/src/database.dart:275-312` exposes only

       _assertValidIdent(name);

## Slice D4 — Prepared-statement cache with `PerformanceWarning` parity

**Severity:** High (performance; mirrors Python B3 which landed in v2.2.2)

### Problem

Python's `PerformanceWarning` + `_stmt_cache` LRU (see
`bindings/python/decentdb/__init__.py:3155-3210`) is missing from Dart. In
`decent-bench`
(`apps/decent-bench/lib/features/workspace/infrastructure/decentdb_bridge.dart:753,768,838`)
the same SQL is re-prepared on every `runQuery` / `fetchNextPage` /
`exportCsv`, wasting an FFI pair (`stmtPrepare` + `stmtFinalize`) per request.
The point-read and email-lookup benchmarks do **not** hit this path (they
prepare once outside the loop) but every real user of the library does.

### Files & exact changes

1. `bindings/dart/dart/lib/src/statement.dart`
   - Extract the "raw handle + metadata" state of `Statement` into a private
     `_CachedStmt` holder (or document a logical separation).
   - Add `void resetForReuse()`: combines `reset()` + `clearBindings()` +
     re-primes streaming execution without re-fetching column metadata
     (see D7).

2. `bindings/dart/dart/lib/src/database.dart`
   - Add a private LRU keyed by SQL text, capacity 128 by default,
     configurable through `Database.open(..., stmtCacheCapacity: N)` with
     `N >= 0` (0 disables). Implementation: a `LinkedHashMap` with
     `remove`-then-`put` on access (O(1) reorder), not `VecDeque::retain`.
   - Change `prepare(String sql)` to:
     1. cache hit → return the cached `Statement` after `resetForReuse()`;
     2. miss → `stmtPrepare`, insert into cache, return; on cache eviction,
        finalize the evicted statement.
   - Add `Map<String, int> get stmtCacheStats` returning
     `{'hits': _h, 'misses': _m, 'size': _lru.length, 'capacity': _cap}`.
   - Expose `void clearStmtCache()` that finalizes every cached statement
     and resets counters.
   - In `Database.close()`, finalize every cached statement before detaching
     the Finalizer.

3. `bindings/dart/dart/lib/src/errors.dart`
   - Add `class PerformanceWarning implements Exception` with a message
     constructor. Keep semantics identical to Python's
     `class PerformanceWarning(UserWarning)`: it is surfaced via a sink,
     not thrown.

4. `bindings/dart/dart/lib/src/database.dart` (continued)
   - Add `void Function(PerformanceWarning)? onPerformanceWarning` setter.
   - After every prepare, if `total >= 100 && hitRate < 0.5`, emit
     `PerformanceWarning('stmt cache hit rate ${pct}% over ${total}
     prepares — consider raising stmtCacheCapacity or reusing Statement
     handles.')` **at most once per Database** (track a `bool
     _perfWarningEmitted`).

5. `bindings/dart/dart/lib/src/decentdb.dart` — re-export
   `PerformanceWarning`.

### Acceptance criteria

- With default capacity, preparing the same SQL 1,000 times runs ≥ 10×
  faster than the current main (measure with a targeted test using
  `Stopwatch`, with a generous headroom to avoid CI flakiness — assert
  ratio ≥ 5×).
- `stmtCacheStats` reports correct counts.
- Overflowing the cache finalizes evicted statements (verified via a
  `SHOW OPEN STATEMENTS` style assertion or by checking
  `inspectStorageState().rawJson` for no handle leak; if neither is
  sufficient, add an internal counter in Dart).
- `onPerformanceWarning` fires exactly once when hit rate stays below 50%
  after 100 prepares, and does not fire when hit rate is ≥ 50%.
- Closing the Database finalizes every cached statement (verified with
  D9's leak test).

### Unit tests to add

New file `bindings/dart/dart/test/stmt_cache_test.dart`:

- `cache hit returns same handle shape` — prepare same SQL twice; second
  call reports `hits == 1`.
- `eviction finalizes the oldest entry` — set capacity=2, prepare three
  distinct SQLs, assert `size == 2`.
- `clearStmtCache drops all entries` — `size == 0`, counters reset.
- `capacity=0 disables caching` — every prepare is a miss.
- `performance warning emits once` — force hit rate < 50% over 100
  prepares, assert sink called exactly once.
- `close finalizes cached statements` — no native leak (cross-checked with
  D9's Finalizer assertion).

## Slice D5 — Replace `calloc` with `malloc` on write-only FFI buffers

**Severity:** High (performance; touches every hot bind/batch path)

### Problem

`bindings/dart/dart/lib/src/statement.dart` zero-fills buffers it
immediately overwrites:

- `:298-316` (`bindBlob`) — `calloc<Uint8>(value.length)`.
- `:427` (`executeBatchInt64`) — `calloc<Int64>(N)`.
- `:459-462` (`executeBatchI64TextF64`) — three `calloc` slabs.
- `:596-609` (`executeBatchTyped`) — `calloc<Int64>`, `calloc<Double>`,
  `calloc<Pointer<Utf8>>`, `calloc<IntPtr>`, `calloc<Uint8>`.

For a 10k-row `executeBatchTyped('itttt', ...)` call this is ~400 KB of
needless `memset`. On `insert_reviews` / `insert_products` paths this
likely contributes to the 15–18% throughput regression.

### Files & exact changes

`bindings/dart/dart/lib/src/statement.dart` — at every site listed above:

- Replace `calloc<T>(N)` with `malloc<T>(N)` (from `package:ffi` — the
  existing `calloc` import already comes from that package; change the
  import alias or use `malloc` directly).
- Keep `calloc` **only** for small out-parameter slots (single-element
  `Pointer<Pointer<Utf8>>` receivers, `stepStatus` scratch) where the
  native side may legitimately leave a field unwritten on an error path.
- Every replaced allocation must be followed by an immediate, deterministic
  initialisation of every byte that will be read by the native side.

For `executeBatchTyped` specifically:

- The `typeFlags` buffer currently zero-fills then overwrites — replace
  with `malloc`; every row/column pair is written exactly once.
- The `blobLens` buffer is per-row per-blob — replace with `malloc`.

### Acceptance criteria

- No `calloc<Int64>`, `calloc<Double>`, `calloc<Uint8>`, or
  `calloc<Pointer<Utf8>>` remain in `statement.dart` except for the
  explicitly-justified small out-param slots.
- A micro-benchmark prepared in the test suite (`test/bench/`) shows
  `executeBatchTyped` with 10k rows at parity or faster than main
  (no statistical regression).

### Unit tests to add

Add to `bindings/dart/dart/test/fast_paths_test.dart`:

- `executeBatchTyped round-trips 10k mixed-typed rows` with a schema that
  hits every type branch (int, text, float, blob); assert row count and
  sampled values.
- `bindBlob round-trips a 1 MiB blob exactly`.

No explicit timing assertions — the micro-benchmark is for developer
reference only; correctness is the test gate.

## Slice D6 — Single-pass `executeBatchTyped` (remove double-buffering)

**Severity:** High (performance; bulk insert path)

### Problem

`bindings/dart/dart/lib/src/statement.dart:529-579` scans the caller's
`List<List<Object?>>` rows and appends to three Dart growable lists (`ints`,
`texts`, `doubles`). Lines `:596-609` then copy those lists into freshly
allocated C buffers. The total size is fully known at entry (`rows.length *
typeFlags.length`) so the intermediate Dart lists are pure overhead.

### Files & exact changes

`bindings/dart/dart/lib/src/statement.dart:529-609`:

- Before the scan, compute per-type totals from `typeFlags` and
  `rows.length`:
  ```dart
  final iTotal = rows.length * _countFlag(typeFlags, 'i');
  final fTotal = rows.length * _countFlag(typeFlags, 'f');
  final tTotal = rows.length * _countFlag(typeFlags, 't');
  final bTotal = rows.length * _countFlag(typeFlags, 'b');
  ```
- Allocate once (via `malloc` per D5): `ints`, `floats`, `texts` (as
  `Pointer<Pointer<Utf8>>`), `blobs`, `blobLens`.
- In the row loop, write directly into the C buffers by advancing
  per-column running indices. Eliminate the three Dart growable lists
  entirely.
- Free TEXT Utf8 pointers in the existing `finally` block; free the
  backing `Uint8List`/`Pointer<Uint8>` copies for blobs in the same
  block.
- The public signature of `executeBatchTyped` does not change.

### Acceptance criteria

- `executeBatchTyped` behaves identically for all existing tests.
- Memory allocator counts (observed via `DECENTDB_NATIVE_LIB`-linked debug
  allocator or by running `valgrind --tool=massif` locally — optional)
  show at most one heap allocation per typed column per call, plus one
  per TEXT/BLOB value (unavoidable).

### Unit tests to add

Add to `bindings/dart/dart/test/fast_paths_test.dart`:

- `executeBatchTyped preserves row order across 50k rows`.
- `executeBatchTyped handles null in TEXT and BLOB columns`.
- `executeBatchTyped handles a mix of short and long strings without
  cross-talk between rows`.
- `executeBatchTyped with rows.length == 0 is a no-op`.

## Slice D7 — Cache column metadata across `reset()`

**Severity:** High (performance; dominates point-read latency)

### Problem

`bindings/dart/dart/lib/src/statement.dart:161-196` (`_loadColumnMetadata`)
calls `stmtColumnNameCopy` + `stringFree` once per column. It is triggered
by `_primeStreamingExecution` (`:198-207`), which runs after every
`_invalidateExecution()`. `_invalidateExecution()` fires on every
`bindAll`/`bind`. For `point_read_p50` (3 columns × 10,000 iterations), this
is **30,000 redundant FFI calls plus 10,000 `stringFree` calls**. This
likely accounts for a large share of the 33% point-read regression.

Column names do not change across reset/bind within a single prepared
statement; they only change on `dispose()`.

### Files & exact changes

`bindings/dart/dart/lib/src/statement.dart`:

- Add `bool _columnMetadataLoaded = false;`.
- In `_loadColumnMetadata`, return early if `_columnMetadataLoaded` is
  true.
- At the end of `_loadColumnMetadata`, set `_columnMetadataLoaded = true`.
- In `_invalidateExecution`, **do not** reset `_columnMetadataLoaded`. Only
  reset the per-execution state (`_currentRows`, `_currentPage`, etc.).
- In `dispose()` (`:904-917`), set `_columnMetadataLoaded = false` just
  before releasing the native handle.
- Audit: the column **count** (from `stmtColumnCount`) similarly does not
  change — cache it with a guard flag or reuse the same flag if the two
  loads are co-located.

### Acceptance criteria

- Column metadata loader fires exactly once per prepared statement per
  lifetime (verify with a test-only counter injected behind an
  `@visibleForTesting` hook or a static counter guarded by an assertion).
- `point_read_p50_ms` regression recovers at least partially (assert by
  comparing `test/bench/point_read_bench_test.dart` before/after — use a
  ratio check rather than absolute numbers to avoid CI flakiness).

### Unit tests to add

Add to `bindings/dart/dart/test/statement_streaming_test.dart`:

- `column metadata is cached across reset/bind/execute cycles` — verify
  via an injected counter or a public `@visibleForTesting` flag that
  `_loadColumnMetadata` is invoked once after 1,000 bind/execute
  iterations.
- `column metadata invalidates on dispose + re-prepare`.

## Slice D9 — `Finalizer` for `Statement`

**Severity:** Medium (memory safety; real leak path observed in
decent-bench)

### Problem

`Database` attaches a `Finalizer` (`:22-38`) and detaches it in `close()`
(`:359`). `Statement` (`bindings/dart/dart/lib/src/statement.dart:92-121`)
does **not**. If a statement is dropped without `dispose()`, the native
`ddb_stmt_t *` leaks. The `decent-bench` cursor map
(`apps/decent-bench/lib/features/workspace/infrastructure/decentdb_bridge.dart:783,799-807`)
has exactly this hazard: an orphaned entry (e.g., isolate shutdown during a
query) leaks the statement.

### Files & exact changes

`bindings/dart/dart/lib/src/statement.dart`:

- Add:
  ```dart
  static final Finalizer<_StatementFinalizable> _finalizer =
      Finalizer((h) => h.bindings.stmtFinalize(h.handle));
  ```
  where `_StatementFinalizable` holds the `NativeBindings` and
  `Pointer<DdbStmt>`.
- In the constructor (`:112`), attach the finalizer:
  `_finalizer.attach(this, _StatementFinalizable(bindings, handle),
  detach: this);`
- In `dispose()` (`:904-917`), call `_finalizer.detach(this)` **before**
  invoking `stmtFinalize` so a double-free cannot happen if GC fires
  concurrently.
- If `Statement` exposes `reset()` or `clearBindings()` after dispose,
  guard them with the existing `_disposed` check.

### Acceptance criteria

- Orphaning a statement (dropping without `dispose()`) eventually
  finalizes it when the GC runs (tested via `await Future.delayed` +
  `forceGC` hook if available, or via a soak test that allocates many
  orphans and asserts RSS does not grow).
- Double-`dispose()` does not call `stmtFinalize` twice.
- The Finalizer is detached in `dispose()` before `stmtFinalize` runs.

### Unit tests to add

New file `bindings/dart/dart/test/statement_finalizer_test.dart`:

- `orphaned statement is finalized eventually` — allocate 10,000
  statements in a tight loop without disposing, `await
  Future.delayed(Duration(seconds: 1))` and then trigger GC (see
  `dart:developer` or the existing `memory_leak_test.dart` pattern);
  assert no native handle leak (via `inspectStorageState` when D2 is
  implemented, or via an internal open-handle counter).
- `dispose + drop does not double-free` — call `dispose()`, set local to
  null, force GC; no crash.

## Slice D10 — Optional async / isolate wrapper

**Severity:** Medium (ergonomics; Flutter relevance)

### Problem

All FFI calls in `database.dart` / `statement.dart` are synchronous and
block the caller's isolate. Flutter UIs that embed DecentDB today must
either run on the UI isolate (bad) or roll their own bridge (as
decent-bench does at `decentdb_bridge.dart`, 1,165 lines).

### Files & exact changes

Add `bindings/dart/dart/lib/src/async_database.dart` (new file):

- `final class AsyncDatabase` owning a worker `Isolate` that hosts a
  synchronous `Database`.
- A request/response protocol over `SendPort`/`ReceivePort` with
  `int requestId` correlation; the message payload is a sealed class
  hierarchy (`_OpenRequest`, `_ExecuteRequest`, `_PrepareRequest`,
  `_StmtBindRequest`, `_StmtStepRequest`, `_StmtQueryRequest`,
  `_StmtDisposeRequest`, `_CloseRequest`) with matching responses.
- Each `AsyncDatabase.execute(...)`, `.prepare(...)`, and related methods
  returns a `Future`.
- `AsyncStatement` handles follow the same shape.
- On worker exit, drain outstanding Futures with a typed
  `AsyncDatabaseClosed` exception.

**Do not** expose the worker protocol as public API — the public surface
is only `AsyncDatabase` and `AsyncStatement`.

**Scope guard:** this slice is Dart-only. It does **not** change the
engine's `Send`/`Sync` behaviour — it relies on the existing thread-safety
of `ddb_db_t *` and confines each handle to the worker isolate. If in the
course of implementation any Rust-side guarantee needs reconsideration,
stop and author an ADR per `AGENTS.md §7` before proceeding.

### Acceptance criteria

- `AsyncDatabase.open(...)` succeeds, `.execute(...)` returns a
  `Future<void>`, `.prepare(...)` returns a `Future<AsyncStatement>`.
- A 100-iteration concurrent workload on the main isolate does not block
  a `Timer.periodic(Duration(milliseconds: 10), ...)` callback by more
  than one tick.
- Worker isolate terminates cleanly on `AsyncDatabase.close()`; no
  zombie isolate.

### Unit tests to add

New file `bindings/dart/dart/test/async_database_test.dart`:

- `AsyncDatabase executes, prepares, and queries` — end-to-end round
  trip.
- `main isolate remains responsive during a long query` — assert a
  `Timer` callback count meets a floor.
- `close terminates the worker isolate`.
- `operations after close throw AsyncDatabaseClosed`.

## Slice D11 — `Stream<Row>` API on `Statement`

**Severity:** Medium (ergonomics)

### Problem

Row iteration requires manual `step()`/`readRow()`
(`bindings/dart/dart/lib/src/statement.dart:805-841`) or manual paging
via `nextPage`. Dart idiom prefers `Stream<T>`.

### Files & exact changes

`bindings/dart/dart/lib/src/statement.dart`:

- Add:
  ```dart
  Stream<Row> rows({int pageSize = 256}) async* {
    RangeError.checkNotNegative(pageSize, 'pageSize');
    if (pageSize == 0) throw ArgumentError.value(pageSize, 'pageSize', 'must be > 0');
    while (true) {
      final page = nextPage(pageSize);
      for (final row in page.rows) { yield row; }
      if (page.isLast) return;
    }
  }
  ```
- Document that starting a new iteration requires `reset()` first.
- Document that the stream is single-subscription.

### Acceptance criteria

- `await for (final row in stmt.rows()) { ... }` produces every row
  exactly once.
- `pageSize = 1` works.
- `pageSize = 0` or negative throws `ArgumentError`.
- Breaking out of the `await for` (via `break`) leaves the statement in
  a resettable state (does not corrupt pending execution).

### Unit tests to add

Add to `bindings/dart/dart/test/statement_streaming_test.dart`:

- `rows() yields the expected sequence for 5k rows`.
- `rows(pageSize: 1) yields the expected sequence`.
- `rows(pageSize: 0) throws ArgumentError`.
- `breaking out of await-for allows subsequent reset + query`.
