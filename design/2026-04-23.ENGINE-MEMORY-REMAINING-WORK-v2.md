## 0. Why this v2 exists

The original remaining-work document now mixes shipped slices with still-open work.
That is no longer useful for execution. This v2 is narrower: it lists only the
unfinished work that still blocks the engine-memory task from being called done.

This document is deliberately prescriptive. The goal is to remove design choice
from the next agent's loop. The remaining work should be executed as the slices
below, in order, without re-deciding the architecture.

Quality checks are a hard completion gate, not a follow-up chore. The work is
**not done** until the required validation commands are green, and any slice
that introduces or preserves failing checks is still unfinished.

## 2. Already landed; do not spend more time here unless a fresh regression proves breakage

1. **Phase B default-on deferred table materialization** is landed.
2. **Phase C persistent PK index** is landed behind its soak flag.
3. **M4 WAL `OnDisk` demotion** is implemented end-to-end.
4. **ADR 0141 spill backend** has a real first implementation; it is not a scaffold anymore.
5. **ADR 0142 allocator plumbing** has a real first implementation; it is not a scaffold anymore.
6. The catastrophic **.NET insert regression** was fixed by removing an accidental
   resident-table `Arc<TableData>` clone before `table_data_mut()` in the hot
   insert path. Do not reopen that investigation unless the benchmark regresses again.

This v2 document therefore starts from the assumption that the remaining work is:
**executor promotion, mutation end-state, cleanup, and proof.**

---

## 3. Hard invariants for every remaining slice

1. **Deferred mode must stay deferred.** Safe reads and writes on reopened paged
   databases must not leave the live runtime broadly materialized after the
   statement or transaction completes.
2. **Final result materialization is allowed; source-table materialization is not.**
   Building the final `QueryResult` rows is fine. Pulling every source row into a
   resident `Dataset` or resident `TableData` just to execute the query is not.
3. **Do not add a second executor.** Extend the existing row-source / snapshot-local
   execution path in `db.rs` and `exec/mod.rs`; do not create a parallel planner or
   a one-off "memory mode" executor.
4. **Row IDs remain stable.** The mutation end-state must preserve `row_id`
   stability and keep persistent PK locator semantics correct across insert,
   update, delete, reopen, checkpoint, and migration.
5. **Checkpoint may compact, but readers stay correct.** Any paged-mutation
   overlay or tombstone design must preserve snapshot-visible bytes until the WAL
   / pager snapshot rules allow compaction.
6. **Non-deferred mode is not the target.** Do not spend cleanup time trying to
   remove the eager-load code paths used when
   `DbConfig::defer_table_materialization = false`; only remove deferred-mode
   fallbacks that are still acting as memory footguns.

---

## 4. Required execution order

Execute the remaining work in this order:

1. **D-E1 through D-E4**: finish read-side executor promotion first.
2. **D-M1 through D-M4**: replace resident-image mutation persistence with the
   streaming paged end-state.
3. **E-C1 through E-C3**: remove the now-dead compatibility paths.
4. **V1 through V4**: run the final proof only after the code work is complete.

Do not start the final proof early. It is too expensive, and the results are not
meaningful until the cleanup slices are merged.

## 5. Phase D executor - exact remaining slices

### D-E1. Finish single-table grouped execution on row sources

**Problem**

The engine already has several grouped fast paths, but when they miss, grouped
queries still fall back to the generic `evaluate_select(...) ->
build_select_dataset(...) -> evaluate_grouped_select(...)` path in
`crates/decentdb/src/exec/mod.rs`. That generic path still builds a full
source `Dataset` in memory before grouping.

The currently relevant code is in:

- `exec/mod.rs::evaluate_select`
- `exec/mod.rs::build_select_dataset`
- `exec/mod.rs::evaluate_grouped_select`
- `exec/mod.rs::apply_select_distinct`
- `exec/mod.rs::sort_dataset`

The existing grouped helpers around `simple_grouped_count_result_from_*` and
`simple_grouped_numeric_aggregate_result_from_*` prove the row-source pattern is
already viable; this slice finishes the broader single-table grouped lane.

**Implement exactly this**

1. Add a new safe grouped read helper in `crates/decentdb/src/exec/mod.rs` that
   executes **single-base-table grouped queries** directly from
   `VisibleTableRowSource`.
2. Scope this helper to:
   - one base table in `FROM`
   - no lateral
   - no window functions
   - no recursive CTE
   - no view / temp table inputs
   - expressions resolving only against the one table binding plus aggregate outputs
3. Run it **after** the existing narrow grouped fast paths miss, not before them.
   The existing specialized paths remain the first choice.
4. Reuse the current grouped expression helpers and rewrite helpers where they
   already exist. Do not invent a new aggregate expression model.
5. Maintain only:
   - per-group aggregate state
   - per-group rewritten output state needed for `HAVING` / `ORDER BY`
   - final output rows
   Do **not** materialize every input row into a `Dataset`.
6. Keep result-level `ORDER BY`, `LIMIT`, and `OFFSET` behavior identical to the
   current generic grouped path.

**Files**

- `crates/decentdb/src/exec/mod.rs`
- tests in the existing grouped-query suites under `crates/decentdb/src/exec/`

**Add tests**

1. Reopen-time grouped query on a paged table that currently misses the simple
   grouped fast path and therefore broad-loads today.
2. The same query inside a shared SQL transaction.
3. One grouped query with `HAVING` plus result `ORDER BY`.
4. One grouped query with mixed aggregate projections that are not handled by the
   current narrow specialization.

**Exit condition**

For the tested grouped shapes, the live runtime and the shared-transaction runtime
both stay off resident full-table materialization after reopen.

### D-E2. Replace generic join fallback with row-source join execution

**Problem**

`exec/mod.rs::evaluate_from_item_in_scope(...)` still evaluates many joins by
first building left and right `Dataset`s and then calling `nested_loop_join(...)`.
That is still a full-table materialization footgun for queries whose inputs are
already available as row sources.

The main hotspots are:

- `exec/mod.rs::evaluate_from_item_in_scope`
- `exec/mod.rs::nested_loop_join`
- `exec/mod.rs::try_indexed_equi_join_with_right_table`
- `exec/mod.rs::evaluate_join_with_lateral_right`

The indexed join path that already exists is good, but the fallback path still
materializes both sides before joining.

**Implement exactly this**

1. Add a new row-source join helper for **base-table non-lateral joins**:
   `INNER`, `LEFT`, `RIGHT`, and `FULL`, with `ON`, `USING`, and `NATURAL`.
2. Keep the current indexed join fast path first. Only use the new row-source
   join helper when the indexed fast path misses.
3. Scope the new helper to joins whose inputs are already base tables loaded in
   the snapshot-local runtime clone. Do **not** try to solve lateral, view,
   subquery, function, or temp-table joins in this slice.
4. Build join output rows incrementally. It is acceptable to materialize the
   join output rows; it is **not** acceptable to first build two full source
   `Dataset`s just to perform the join.
5. Reuse the existing join-column resolution helpers and output-shaping helpers
   where possible. Keep SQL semantics identical.
6. Route the snapshot-local deferred execution path in `db.rs` to this helper
   before the broad generic fallback.

**Files**

- `crates/decentdb/src/exec/mod.rs`
- `crates/decentdb/src/db.rs`

**Add tests**

1. Reopen-time paged-table `INNER JOIN` shape that currently falls back.
2. `LEFT JOIN` and `RIGHT JOIN` cases with `ON` predicates and result ordering.
3. `JOIN USING` or `NATURAL JOIN` regression proving hidden-column semantics stay correct.
4. Shared-transaction versions of the same queries.

**Exit condition**

Base-table join queries that do not need lateral/subquery/view behavior no longer
require full source-table `Dataset` construction from reopened paged tables.

### D-E3. Keep generic safe `SELECT` execution on the row-source lane after D-E1/D-E2 land

**Problem**

`db.rs::try_execute_query_with_row_sources_at_snapshot(...)` already clones a
snapshot-local runtime and loads only referenced row sources, but it still
delegates to the generic executor. After D-E1 and D-E2 land, the routing layer
must prefer those row-source-capable paths and avoid unnecessary broad fallback.

The relevant code is in:

- `db.rs::try_execute_query_with_row_sources_at_snapshot`
- `db.rs::execute_read_in_runtime_state`
- `db.rs::can_execute_statement_with_row_sources_at_latest_snapshot`
- `exec/mod.rs::evaluate_query`
- `exec/mod.rs::evaluate_query_body`

**Implement exactly this**

1. Extend the row-source safe-query routing so that:
   - single-table grouped queries use the D-E1 helper
   - base-table joins use the D-E2 helper
   - only unsupported features drop to the broad generic resident path
2. Keep recursive CTEs, lateral joins, view expansion, temp tables, and window
   functions on the old executor for now. This slice is routing cleanup, not a
   new feature sprint.
3. Add explicit guards so that a query is either:
   - handled fully on the row-source lane, or
   - explicitly rejected from the lane and sent to the old fallback.
   Do not leave "accidental" fallthrough.
4. Keep the trigger/base-table dependency loading rules already established in
   `db.rs::safe_referenced_base_tables_in_runtime(...)`.

**Files**

- `crates/decentdb/src/db.rs`
- `crates/decentdb/src/exec/mod.rs`

**Add tests**

1. Reopen-time query that previously loaded only because routing missed a now-supported path.
2. Shared-transaction version of the same query.
3. Negative coverage: a truly unsupported feature should still fall back cleanly.

**Exit condition**

For safe grouped and join queries, the deferred read path reaches the new
row-source helpers instead of the old broad resident fallback.

### D-E4. Reduce remaining row-source blockers in post-processing helpers

**Problem**

Some helpers still assume "generic query means full dataset" even when the source
scan was already row-source safe. The main remaining memory-sensitive helpers are:

- `exec/mod.rs::apply_select_distinct`
- `exec/mod.rs::sort_dataset`
- `exec/mod.rs::project_dataset`
- `exec/mod.rs::evaluate_set_operation`

Not all of these must become fully streaming in this batch, but the large-table
footguns must be removed for the query shapes in scope.

**Implement exactly this**

1. For the query shapes covered by D-E1 and D-E2, keep post-processing bounded:
   - reuse result-level bounded-order helpers where they already exist
   - avoid re-materializing a second full copy of the result if the first one already exists
2. Do **not** try to solve full external sort or spill in this slice.
3. Leave recursive CTE and generic set-op internals alone unless they are needed
   to preserve the new row-source route for an already-safe query body.

**Files**

- `crates/decentdb/src/exec/mod.rs`

**Exit condition**

The new grouped and join lanes do not immediately lose their memory win in a
second post-processing copy step.

## 6. Phase D mutations - exact remaining slices

### D-M1. Replace resident-centric dirty tracking with row-id keyed mutation deltas

**Problem**

The current dirty tracking in `exec/mod.rs` is still resident-table oriented:

- `mark_table_row_dirty(...)` stores `row_index` plus cloned values
- `mark_table_row_deleted(...)` stores deleted row IDs separately
- `persist_to_db(...)` still branches into resident-image rewrite helpers for
  paged updates and deletes

This is the last big reason paged update/delete persistence still depends on a
resident end-state image.

**Implement exactly this**

1. Replace the resident-centric update/delete dirty split with one row-id keyed
   mutation-delta model for persisted base tables.
2. The mutation delta must distinguish:
   - appended rows
   - rewritten rows by `row_id`
   - deleted `row_id`s
3. `row_index` must stop being the persistence identity for paged tables. It may
   still be used internally for resident-table in-memory edits, but the data
   carried forward to persistence must be keyed by `row_id`.
4. Keep temp-table behavior unchanged.

**Files**

- `crates/decentdb/src/exec/mod.rs`
- `crates/decentdb/src/exec/dml.rs`
- runtime tests in `crates/decentdb/src/exec/runtime_tests.rs`

**Exit condition**

By the end of this slice, paged-table persistence has enough information to
rewrite from row-id keyed deltas without consulting a full resident end-state table.

### D-M2. Replace manifest rewrite-by-resident-image with tombstone-plus-overlay persistence

**Problem**

The current paged mutation helpers in `exec/mod.rs`:

- `apply_paged_row_changes_to_manifest(...)`
- `rewrite_paged_table_from_resident_row_updates(...)`
- `rewrite_paged_table_from_resident_row_deletes(...)`
- the paged branches inside `persist_to_db(...)`

still derive the next persisted paged state by decoding prior chunks and
rebuilding changed chunk payloads from resident rows.

**Do not redesign this again. Implement the chosen end-state now:**

1. Extend the persisted paged-manifest format so each logical chunk can keep:
   - the original base chunk pointer
   - a sorted set/list of tombstoned `row_id`s for that chunk
   - an optional overlay chunk payload containing replacement rows for that chunk
2. **Delete semantics:** mark the row ID tombstoned in its existing chunk. Do not
   regenerate the entire chunk payload.
3. **Update semantics:** tombstone the old row ID in its existing chunk and append
   the replacement row to that chunk's overlay payload, keeping the same `row_id`.
4. **Insert semantics:** keep the existing append-only chunk growth behavior.
5. Read semantics for `TablePageManifest` / `TableRowRef` must become:
   - replacement row wins if present
   - tombstoned base row is skipped
   - untouched base row is read directly from the original chunk
6. Checkpoint compaction must fold tombstones and overlays back into fresh base
   chunk payloads and clear the overlay state.

This is the design. Do not add a second paged-mutation scheme.

**Files**

- `crates/decentdb/src/exec/mod.rs`
  - `TablePageManifest`
  - paged manifest encode/decode helpers
  - `persist_to_db(...)`
  - checkpoint compaction helpers
- any migrate-tool or manifest-version code needed by the format extension

**Add tests**

1. Reopen-time paged `UPDATE` that changes a row in the middle of a large table
   without rewriting untouched chunks.
2. Reopen-time paged `DELETE` with the same guarantee.
3. Multi-update/delete sequence on the same chunk before checkpoint.
4. Checkpoint compaction proving overlays disappear but visible row contents stay identical.

**Exit condition**

`persist_to_db(...)` no longer calls resident-image paged rewrite helpers for
paged update/delete. Paged update/delete persistence is driven by row-id deltas
and manifest overlays instead.

---

### D-M3. Route the remaining DML fallbacks onto the new paged mutation model

**Problem**

Several DML paths in `crates/decentdb/src/exec/dml.rs` still assume resident
table data when the narrow paged helper misses. The biggest remaining paths are:

- `execute_update(...)`
- `execute_delete(...)`
- `apply_conflict_update(...)`
- `render_returning(...)`

Those paths currently use `table_data(...)` / `table_data_mut(...)` heavily.

**Implement exactly this**

1. Keep the existing paged fast paths first.
2. After D-M2 lands, move the remaining generic paged `UPDATE`, `DELETE`, and
   `UPSERT DO UPDATE` branches to the row-id keyed paged mutation model instead
   of falling back to resident `table_data_mut(...)`.
3. `RETURNING` may still materialize the returned rows, but it must do so from
   the row-change set, not by requiring the full resident table image.
4. Preserve trigger behavior, FK action behavior, and index freshness behavior.
5. Re-defer touched persisted tables after autocommit writes exactly as the
   existing row-source write control plane already does.

**Files**

- `crates/decentdb/src/exec/dml.rs`
- `crates/decentdb/src/db.rs` for autocommit / transaction control-plane wiring

**Add tests**

1. Reopen-time paged `UPDATE ... RETURNING`.
2. Reopen-time paged `DELETE ... RETURNING`.
3. Reopen-time paged `INSERT .. ON CONFLICT DO UPDATE RETURNING`.
4. Shared-transaction versions where applicable.

**Exit condition**

Generic paged DML no longer needs resident `TableData` for persistence or
`RETURNING` semantics.

---

### D-M4. Rebuild PK locators, checkpoint compaction, and crash safety around the new mutation model

**Problem**

Once overlays and tombstones exist, the read path, persistent PK locator path,
checkpoint compaction, migrate proof, and crash recovery all need to agree on
the same logical row view.

**Implement exactly this**

1. Rebuild paged-table PK locators from the logical visible row set after
   mutation persistence, not from stale base chunks.
2. Ensure checkpoint compaction rewrites locators to the new compacted chunk
   locations.
3. Ensure crash recovery sees either:
   - the old manifest state, or
   - the fully committed new manifest state,
   never a half-updated mix.
4. Extend crash-injection coverage specifically for the new paged mutation writes.

**Files**

- `crates/decentdb/src/exec/mod.rs`
- locator / persisted-table helpers
- crash-injection tests

**Exit condition**

The new paged mutation format is not just faster; it is durable and restart-safe.

## 7. Phase E cleanup - exact remaining slices

### E-C1. Remove deferred-mode broad-load fallbacks that Phase D made obsolete

**Problem**

`db.rs` still contains broad-load fallbacks whose only reason to exist was that
the executor and mutation paths were not fully row-source capable yet.

The remaining audit targets are:

- `refresh_and_ensure_all_tables_loaded(...)`
- `ensure_runtime_all_tables_loaded_at_snapshot(...)`
- any deferred-mode path that still uses `ensure_all_tables_loaded(...)` because
  a grouped/join/DML gap forced it

**Implement exactly this**

1. Audit every deferred-mode caller in `db.rs`.
2. For each caller, either:
   - convert it to targeted row-source loading, or
   - leave an explicit comment that it is intentionally unsupported and why.
3. After D-E and D-M land, there should be no "safety" fallback that broad-loads
   all tables for the query and write shapes covered by this document.

**Exit condition**

Deferred-mode broad all-table loads are no longer part of normal grouped, join,
or paged DML execution.

---

### E-C2. Shrink `EngineRuntime::tables` to the post-Phase-D role

**Problem**

`EngineRuntime::tables` is already `BTreeMap<String, TableRowSource>`, but too
many helpers still conceptually treat it as "the place every base table row lives".

**Implement exactly this**

1. Narrow resident-only helpers so their names and call sites reflect reality.
   If a helper is resident-only, make that explicit.
2. Remove dead compatibility shims that only existed to bridge the old
   `Arc<TableData>` world into the new `TableRowSource` world.
3. Keep temp-table helpers intact; this cleanup is about persisted base tables.

**Files**

- `crates/decentdb/src/exec/mod.rs`
- `crates/decentdb/src/exec/dml.rs`

**Exit condition**

The runtime no longer implies that every persisted base table has a resident
`TableData` image available by default.

---

### E-C3. Delete the old resident-image paged rewrite helpers and stale comments

**Problem**

Once D-M2 and D-M3 are in place, the resident-image paged rewrite helpers are
dead code and dangerous to keep around.

**Implement exactly this**

1. Delete the old resident-image paged rewrite helpers.
2. Delete the old row-update/row-delete persistence branches that only existed
   for the temporary compatibility model.
3. Remove stale comments that still describe M4, ADR 0141, or ADR 0142 as
   scaffolds or future work.

**Exit condition**

There is one paged mutation persistence model in the codebase, not two.

## 9. Definition of done for the whole engine-memory task

The full task is done only when all of the following are true:

1. Safe grouped and join queries no longer need full resident source-table materialization.
2. Paged `UPDATE` / `DELETE` / `UPSERT DO UPDATE` persistence no longer derives
   new paged state from a resident end-state table image.
3. Deferred-mode broad-load compatibility fallbacks for those shapes are removed.
4. `python ./scripts/do-pre-commit-checks.py` is green.
5. The memory, benchmark, migration, and crash thresholds listed above are all met.

Passing quality checks is mandatory. There is no "code complete but checks still
red" state for this task; red checks mean the task is still in progress.

Until then, the correct active task list remains:

1. `phase-d-executor`
2. `phase-d-mutations`
3. `phase-e-cleanup`
4. `final-validation`
