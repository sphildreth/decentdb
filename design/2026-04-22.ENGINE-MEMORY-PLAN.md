### Outstanding (carried over to the remaining-work document)

- **`defer_table_materialization` default flip blocked on snapshot pinning.** Setting it to `true` triggers `concurrent_writer_reader_overflow_consistency` ("overflow payload length mismatch: expected X, decoded Y"). Until that lands, single-writer users can opt in safely; concurrent-writer users must keep the default.

## 0. Executive summary

The .NET binding benchmark suite revealed that DecentDB peaks at **2.2 GB resident** while loading ~2.75 M rows into a 145 MB on-disk database. Initial assumption was that the .NET layer was the cause; a raw-Rust baseline disproved that — Rust hits the same ceiling. This document records the diagnostic work that pinpointed the real causes and proposes a phased reduction plan.

**Headline number for the user's real workload:** loading 10 M rows of small tuples (~80 B each) into a single DecentDB on a 16 GB host:

| metric | observed (no mitigations) | after Phase 1 (est.) | after Phase 2 (est.) |
|---|---:|---:|---:|
| peak `VmRSS` during load | **2 635 MB** | ~600 MB | ~250 MB |
| peak `VmRSS` during checkpoint | **3 379 MB** | ~900 MB | ~300 MB |
| `VmRSS` after `Db` drop | **2 164 MB** (glibc-held) | <50 MB | <50 MB |
| `VmRSS` after `Db` re-open + tiny query | **2 007 MB** | ~150 MB | ~80 MB |
| DB size on disk | 73 MB | 73 MB | 73 MB |
| peak / disk ratio | **46×** | ~12× | ~4× |

The 16 GB server is **not** in immediate danger for a 10 M-row workload — a single load fits today — but the headroom is much smaller than the on-disk size suggests, and the ratio degrades non-linearly as soon as a query plan needs sort or hash-aggregate buffers. Phases 1 and 2 below collapse this to a defensible footprint.

## 1. Diagnostic methodology

A standalone probe was added at `/tmp/tmp-opus47-decentdb-net-tests/rust-baseline/src/memory_probe.rs`. It:

1. uses the engine's hot path directly: `Db::create()` → `db.transaction()` → cached `prepared.execute_in()` → `txn.commit()` per logical batch;
2. inserts N rows of `events(id BIGINT PRIMARY KEY, kind INT, payload TEXT)` with configurable batch size and `cache_size_mb`;
3. samples **`VmRSS`, `RssAnon`, `RssFile`** from `/proc/self/status` at every 10th batch and at every lifecycle event (`Db::create`, end-of-load, `Db::checkpoint()`, `Db` drop, `malloc_trim(0)`, re-open, `COUNT(*)`, by-id lookup, second drop);
4. snapshots `Db::inspect_storage_state_json()` at end-of-load and post-checkpoint (exposes `wal_versions`, `wal_file_size`, `last_checkpoint_lsn`, `page_count`).

The probe is intentionally a separate `cargo` bin (`memory-probe`) — it does not perturb the existing rust-baseline benchmark.

### 1.1 Reproducer

```bash
cd /tmp/tmp-opus47-decentdb-net-tests/rust-baseline
cargo build --release --bin memory-probe
./target/release/memory-probe --rows 5000000 --batch 50000 --cache-mb 4 --label probe5m
```

### 1.3 What the breakdown proves

| observation | implication |
|---|---|
| `RssAnon ≈ VmRSS`, `RssFile = 2.3 MB` everywhere | The footprint is **anonymous heap**, not page-cache `mmap`. The 4 MB page-cache cap is being honored. The problem is not "the engine is mmap'ing the world". |
| `wal_versions = 47 173` at end-of-load with `last_checkpoint_lsn = 0` | **No automatic checkpoint fired during a 6.5 s, 5 M-row load.** The WAL grew unbounded in memory and on disk (192 MB pre-allocated). |
| `wal_versions × page_size = 47 173 × 4 KB ≈ 184 MB` | The `WalIndex` alone holds ≈ 184 MB of `Arc<[u8]>` page payloads at end-of-load. That is ~14 % of the 1.32 GB heap; the rest comes from per-commit transient allocations the writer keeps reissuing. |
| Checkpoint *increases* RSS (1.32 GB → 1.69 GB) | `wal::checkpoint::checkpoint()` calls `latest_versions_at_or_before(safe_lsn)`, which **clones every `Arc<[u8]>` and the `(PageId, WalVersion)` tuple into a fresh `Vec`** before iterating. For a 47 k-version WAL that is one extra 184 MB allocation on top of everything still resident. |
| `Db` drop releases 1.69 GB → 1.08 GB | The engine *does* free what it allocated; ~600 MB of internal state is dropped cleanly. |
| `malloc_trim(0)` releases 1.08 GB → 8.8 MB | **glibc retains the rest as fragmented heap arenas** until explicitly told to release. This amplifies every transient allocation by the engine into a long-lived RSS number. |
| Re-open + tiny `COUNT(*)` jumps to 1.0 GB on a 36 MB DB | Open path eagerly populates internal state at a size disproportionate to the page cache (4 MB). RSS held even for a single-row by-id lookup. Likely candidates: catalog/runtime materialization, schema cookie expansion, full-DB scan for `COUNT(*)` warming. To be confirmed in slice E1. |
| Per-row growth ≈ 265 B/row, linear in row count | At 5 M rows the engine is allocating ~1.3 GB of *new* heap during the load that doesn't get reused. Confirms per-commit churn rather than per-page bookkeeping. |

### 1.4 What this is *not*

- **Not a leak.** `Db` drop + `malloc_trim(0)` returns the process to baseline (8.8 MB). No reference cycles, no lost `Arc`s.
- **Not the page cache.** `cache_size_mb = 4` (default) is honored; `RssFile` confirms no large `mmap` is in play.
- **Not a binding problem.** The raw-Rust hot path reproduces every number to within ~5 %.
- **Not specific to large rows.** The probe uses ~80 B rows; the per-row amplification is from bookkeeping, not payload size.

## 2. Root causes (ranked by impact)

### C1 — glibc allocator fragmentation amplifies transient writer allocations (impact: **highest**)

DecentDB's commit path issues many short-lived heap allocations per transaction (frame batch buffer, prepared-page payloads cloned into `Arc<[u8]>`, `WalVersion` records, encoded delta scratch). Under sustained writes glibc's `malloc` keeps these in per-arena freelists and almost never returns them to the OS. The 1.08 GB RSS that vanishes on `malloc_trim(0)` is entirely glibc fragmentation.

This is not unique to DecentDB — every Rust workload that does many small allocations against the default global allocator behaves this way on Linux — but DecentDB is unusually exposed because its writer allocates many small `Vec<u8>` and `Arc<[u8]>` per commit and never reuses them.

### C2 — WAL index retains one full page payload per dirtied page until checkpoint (impact: **high**)

`crates/decentdb/src/wal/index.rs` defines

```rust
pub(crate) struct WalIndex {
    pages: HashMap<PageId, Vec<WalVersion>>,
}
pub(crate) struct WalVersion {
    pub(crate) lsn: u64,
    pub(crate) data: Arc<[u8]>,   // full page (4–16 KB) every time
}
```

With no active readers `add_version` keeps the latest version per page (one `Arc<[u8]>` per dirty page); with active readers it appends. Pruning only happens during `checkpoint::checkpoint()`. Result: the in-memory WAL footprint is **proportional to the dirtied-page set**, not to the cache size. For a 5 M-row workload this is 47 173 × 4 KB ≈ 184 MB *before* glibc fragmentation amplifies it.

### C3 — Auto-checkpoint policy never fires during sustained writes (impact: **high**)

`DbConfig::default()` sets `checkpoint_timeout_sec: 30`. ADR 0004 *describes* a size-based threshold, but the running engine did not auto-checkpoint a 192 MB WAL during a 6.5 s load. Whether that is missing implementation, a wall-clock-only trigger, or a threshold set too high, the observable behavior is: sustained writers grow the WAL without bound until the writer voluntarily calls `Db::checkpoint()` or 30 s elapse.

This is the single change with the best leverage: an aggressive size-based or batch-count-based auto-checkpoint would cut steady-state RSS roughly in proportion.

### C4 — Checkpoint copyback materializes the entire WAL in a fresh `Vec` (impact: **medium**)

`wal::checkpoint::checkpoint()`:

```rust
let latest_versions = {
    let index = wal.inner.index.lock()...;
    index.latest_versions_at_or_before(safe_lsn)   // Vec<(PageId, WalVersion)>
};
let page_ids: Vec<_> = latest_versions.iter().map(|(p, _)| *p).collect();
for (page_id, version) in &latest_versions { pager.write_page_direct(...) }
```

`latest_versions_at_or_before` clones every retained `WalVersion` (the `Arc<[u8]>` is cheap, but the surrounding `Vec` and `(PageId, WalVersion)` tuples are not pooled). For a 47 k-page checkpoint that's ~1.5 MB of `Vec` overhead plus a doubled retention window during the loop. Observed effect: RSS spikes from 1.32 GB → 1.69 GB during checkpoint.

### C5 — Re-open path inflates heap on small DBs (impact: **medium, needs measurement**)

Re-opening the 36 MB DB and running a single `COUNT(*)` brings VmRSS to **1.0 GB**, of which 770 MB is still resident after a second drop. The probe didn't differentiate between "open inflates" and "`COUNT(*)` inflates" — the next slice (E1) will. Likely candidates:

- catalog/runtime materialization on first inspection
- `COUNT(*)` performing a full scan that materializes intermediate state
- per-table runtime maps not bounded by `cache_size_mb`

### C6 — Per-commit allocation churn the writer never reuses (impact: **medium**)

`commit_pages()` and `commit_pages_if_latest()` (in `wal/writer.rs`) hold a reused `WalWriteState { page_batch, prepared_pages }` inside the writer mutex — that part is good — but every committed page payload is converted into a fresh `Arc<[u8]>` (`Arc::from(payload)`) and stored in the index. Across 47 k commits this is 47 k `Arc` headers plus 47 k boxed slice allocations. Each is small but the total stresses the allocator.

---

## 3. Reduction plan (phased)

The phases are ordered so each one delivers a measurable RSS reduction independently of the next, and so that no phase requires an on-disk format change (avoids ADR 0001/0033 territory and keeps recovery semantics intact).

### Phase 1 — "stop the bleeding" (no algorithmic changes, no `unsafe`)

Targeted at C1 + C3. These changes are mechanical and small.

| slice | change | expected delta on 5 M-row probe |
|---|---|---|
| **M1** | Implement size-based auto-checkpoint inside `wal::writer::commit_pages*` | end-of-load RSS: 1.32 GB → ~400 MB |
| **M2** | Periodic `malloc_trim(0)` after checkpoint on Linux/glibc (gated, opt-in via cfg) | post-`Db`-drop RSS: 1.08 GB → <50 MB; under sustained load RSS *plateaus* instead of climbing |
| **M3** | Switch the binary's `#[global_allocator]` recommendation to `mimalloc` for embedders that ship a binary; document the option and add a `mimalloc` feature flag on `decentdb-cli` | sustained-load peak RSS: ~30–50 % reduction independent of M2 |

Phase 1 acceptance criteria:

1. `memory-probe --rows 5000000 --batch 50000` shows end-of-load `VmRSS` ≤ 600 MB.
2. `memory-probe --rows 10000000 --batch 100000` completes with peak `VmRSS` ≤ 1 GB on a host without mimalloc, ≤ 700 MB with mimalloc.
3. `Db::inspect_storage_state_json()` reports `wal_versions ≤ 4096` at any point during a sustained write.

### Phase 2 — "bound the working set" (algorithmic, in-memory only)

Targeted at C2 + C4 + C6. These changes touch hot paths and require careful review.

| slice | change |
|---|---|
| **M4** | Replace `WalVersion::data: Arc<[u8]>` with a discriminated `enum WalVersionPayload { OnDiskFrame { offset: u64, len: u32 }, InMemory(Arc<[u8]>) }`. After fsync, drop the in-memory copy and reference the WAL frame on disk. Page cache + WAL `mmap` (ADR 0067) handle the read path. Keep `InMemory` only for the most-recent N pages or for active-reader retention. |
| **M5** | Pool the `Vec<(PageId, WalVersion)>` and `Vec<PageId>` allocations in `checkpoint::checkpoint()` via a `WalCheckpointScratch` field on `WalInner`. Avoids the 184 MB transient spike. |
| **M6** | Pool per-commit `Arc<[u8]>` payloads using a small-object recycler (e.g. `crossbeam-queue::SegQueue<Vec<u8>>` capped at N entries). Re-issue `Arc::from(Vec)` only after the recycler is empty. |
| **M7** | Replace `HashMap<PageId, Vec<WalVersion>>` with `HashMap<PageId, smallvec::SmallVec<[WalVersion; 1]>>` so the no-active-readers fast path (one entry per page) avoids a per-page heap allocation for the inner `Vec`. |

Phase 2 acceptance criteria:

1. Sustained writer steady-state `VmRSS` is **bounded by `cache_size_mb + small constant overhead`**, not by total dirty-page count.
2. Checkpoint does not increase peak `VmRSS` by more than 32 MB regardless of WAL size.
3. `memory-probe --rows 10000000 --batch 100000` shows peak `VmRSS` ≤ 350 MB.

### Phase 3 — "diagnose the open path"

Targeted at C5. Investigation only — the slice scope is sized after evidence is in.

| slice | change |
|---|---|
| **E1** | Extend `memory-probe` to sample `VmRSS` after every named step inside `Db::open()` and after each statement in a known-tiny query suite. Identify which call path inflates a 36 MB DB to 1.0 GB resident. Root-cause goes into Phase 4 if it is structural, or into a hotfix slice if it is an obvious oversight. |

### Phase 4 — "deferred, ADR-required"

| item | ADR |
|---|---|
| Background incremental checkpoint worker | [ADR 0058](adr/0058-background-incremental-checkpoint-worker.md) (Deferred) |
| Switch `WalIndex` to a paged on-disk index with bounded in-memory hot set | [ADR 0141](adr/0141-paged-on-disk-wal-index.md) (Deferred) |
| Per-engine allocator (decoupled from host `#[global_allocator]`) | [ADR 0142](adr/0142-per-engine-allocator.md) (Deferred) |

## 5. Slice details

### Slice M1 — Size-based auto-checkpoint trigger

**Problem.** `commit_pages()` updates `wal_end_lsn` but does not consult any size threshold. `checkpoint_timeout_sec` is the only auto trigger, and it is wall-clock based.

**Verified evidence.** 5 M-row, 6.5 s load produced `wal_versions = 47 173`, `wal_file_size = 192 MB`, `last_checkpoint_lsn = 0`. No checkpoint fired.

**Design decisions.**

- Add `DbConfig::wal_checkpoint_threshold_pages: u32` (default `4096` = 16 MB at 4 KB pages — small enough to bound RSS, large enough to amortize fsync).
- Add `DbConfig::wal_checkpoint_threshold_bytes: u64` (default `64 * 1024 * 1024`).
- Trigger evaluated at the *end* of `commit_pages` / `commit_pages_if_latest`, after `wal_end_lsn.store(offset)`. If either threshold is exceeded *and* `active_reader_count() == 0`, call `checkpoint::checkpoint()` synchronously. If readers are active, skip silently — Phase 4 will revisit.
- Trigger respects existing `checkpoint_pending` flag to avoid re-entry.
- Counters: track `wal_versions_since_last_checkpoint` and `wal_bytes_since_last_checkpoint` on `WalInner` (atomics).

**Implementation steps.**

1. Add the two `DbConfig` fields with defaults; bump `DbConfig` doc comment; add to `metadata.rs` if persisted (it is not — config is per-open).
2. Add atomics on `WalInner`. Increment in `add_version` callers.
3. Helper `wal::writer::should_auto_checkpoint(wal, cfg) -> bool`.
4. At the end of `commit_pages` / `commit_pages_if_latest`, if `should_auto_checkpoint` and no active readers, drop the writer guard, call `checkpoint::checkpoint()`, reset counters.
5. Plumb the values from `DbConfig` into `WalInner` at open.

**Tests.**

- New unit test in `wal/writer.rs`: insert 10 000 4 KB-dirtying rows with threshold = 1024; assert `wal_versions <= 1024` at all times.
- New integration test in `crates/decentdb/tests/`: run the existing benchmark scenario with the new defaults; assert `VmRSS` (via `/proc/self/status`) stays below 600 MB at all sample points for 5 M rows.
- Recovery test: kill mid-batch with auto-checkpoint enabled; assert recovery yields the same row set as without.

**Validation.**

```
cargo fmt --check
cargo clippy --workspace --all-targets --all-features -- -D warnings
cargo test --workspace
./target/release/memory-probe --rows 5000000 --batch 50000 --label m1
```

**Acceptance.**

1. `memory-probe --rows 5000000 --batch 50000` shows `wal_versions <= 4096` at every sampled point.
2. End-of-load `VmRSS` ≤ 600 MB.
3. No throughput regression > 10 % on the rust-baseline `seed_songs` step.

---

### Slice M2 — `malloc_trim(0)` after checkpoint (Linux/glibc)

**Problem.** glibc retains 1.08 GB of freed heap until told to release. Long-lived embedders pay the steady-state cost.

**Design decisions.**

- Compile-time gate: `#[cfg(all(target_os = "linux", target_env = "gnu"))]`. Skipped silently for musl/macOS/Windows/BSD.
- Call site: end of `wal::checkpoint::checkpoint()`, *after* the index is cleared/pruned and the writer guard is dropped, so we don't hold any locks across the syscall.
- Behavior: `unsafe extern "C" { fn malloc_trim(pad: usize) -> i32; }`. Best-effort; ignore return value.
- Throttle: at most once per checkpoint. No additional timer.
- `DbConfig::release_freed_memory_after_checkpoint: bool` (default **`true`** on Linux/glibc, **`false`** elsewhere — the field exists everywhere so embedders have a uniform knob).

**Implementation steps.**

1. Add the `DbConfig` field + doc.
2. Helper `mod platform { #[cfg(...)] pub fn release_freed() { unsafe { libc::malloc_trim(0); } } #[cfg(not(...))] pub fn release_freed() {} }`.
3. Call it from `checkpoint::checkpoint()` under `if cfg.release_freed_memory_after_checkpoint`.
4. No new dep — call `malloc_trim` via `extern "C"`; do not pull in `libc` if not already used.

**Tests.**

- Linux-only test that loads 1 M rows, checkpoints, asserts `VmRSS` < 100 MB after checkpoint.
- Non-Linux test that simply confirms the call compiles and is a no-op.

**Acceptance.**

1. On Linux, `memory-probe` post-checkpoint `VmRSS` is within 50 MB of pre-load baseline.
2. No regression on a `--no-trim` run (passes the existing benchmarks).

---

### Slice M3 — `mimalloc` feature flag on `decentdb-cli`

**Problem.** Even after M2, sustained-write workloads pay glibc fragmentation cost between checkpoints. mimalloc reduces both fragmentation and small-allocation cost.

**Design decisions.**

- New optional dep `mimalloc = { version = "0.1", default-features = false }` on `decentdb-cli` (not on `decentdb` — embedders pick their own allocator).
- Feature flag `mimalloc` on `decentdb-cli` (off by default, opt-in via `cargo build --features mimalloc`).
- Document recommended build for production CLI usage in README and `design/BENCHMARKING_GUIDE.md`.

**Implementation steps.**

1. Add optional dep + feature.
2. In `crates/decentdb-cli/src/main.rs`, gated `#[cfg(feature = "mimalloc")] #[global_allocator] static GLOBAL: mimalloc::MiMalloc = mimalloc::MiMalloc;`.
3. README + BENCHMARKING_GUIDE update.

**Tests.**

- CI smoke job builds with `--features mimalloc` and runs the existing CLI smoke test.

**Acceptance.**

1. `cargo build -p decentdb-cli --features mimalloc` succeeds.
2. The same `memory-probe` rerun (using the CLI for inserts) shows ≥ 30 % lower steady-state `VmRSS`.

### Slice M4 — `WalVersion` discriminated payload

**Problem.** Every retained version holds a 4–16 KB `Arc<[u8]>` in heap.

**Design decisions.**

- `enum WalVersionPayload { Resident(Arc<[u8]>), OnDisk { wal_offset: u64, frame_len: u32, encoding: FrameEncoding } }`.
- Default: write path inserts `Resident(Arc::from(payload))` first (current behavior), then a background-or-inline post-fsync step (same writer thread, after `sync_for_mode`) demotes entries older than the most-recent K to `OnDisk`. K = small (e.g. 16) to keep the hot snapshot fast.
- Read path (`latest_visible`): if `OnDisk`, decode from the WAL `mmap` view (ADR 0067 already provides this); otherwise return the `Arc<[u8]>` directly.
- Active-reader retention: any version visible to a reader stays `Resident` until the reader's snapshot LSN advances past it.

**Why it works.** ADR 0067 already mmaps the WAL for reads; this slice just stops shadow-storing every page in heap.

**Risks / open questions.**

- Decode path needs to be fast enough that a single `latest_visible` lookup against a recently-promoted `OnDisk` version doesn't regress reads measurably. Bench under N17/N18 in the .NET plan.
- Delta-encoded frames (ADR 0132) must be reconstructed against their base; that's already the read-path requirement, so no new logic.

**Tests, validation, acceptance.** Drafted at slice-pickup time (out of scope for this document).

---

### Slice M5 — Pooled checkpoint scratch buffers

Trivial: hoist the `Vec<(PageId, WalVersion)>` and `Vec<PageId>` produced by `latest_versions_at_or_before` and the page-id collection into a `WalCheckpointScratch { versions: Vec<...>, page_ids: Vec<PageId> }` field on `WalInner`, behind the same writer mutex used for checkpoint coordination. `clear()` instead of allocate.

Acceptance: zero new allocations attributable to checkpoint per `cargo test --features dhat-heap` profile.

---

### Slice M6 — Per-commit payload recycler

**Problem.** Each commit issues `Arc::from(Vec<u8>)` per page, never reused.

**Design.** Add `WalInner::payload_recycler: Mutex<SmallVec<[Vec<u8>; 64]>>`. After a `WalVersion` is dropped (Phase 2 demotion or checkpoint pruning), reclaim its `Vec<u8>` (via `Arc::try_unwrap(...).ok()`) and push back. Writer pulls from recycler before allocating.

**Caveat.** `Arc::try_unwrap` only succeeds if the count is 1. Active readers may keep a clone alive — that is correct: we don't want to recycle a buffer a reader still holds. So the recycler is opportunistic, not guaranteed.

---

### Slice M7 — `SmallVec` in `WalIndex`

One-line change: `pages: HashMap<PageId, SmallVec<[WalVersion; 1]>>`. The no-active-readers steady state (one entry per page) avoids one heap allocation per dirty page — at 47 k pages, that's 47 k fewer `Vec` headers.

**Scope.** Extend `memory-probe` with sampling around:

2. `Db::execute("SELECT COUNT(*) FROM events")` (vs `Db::execute("SELECT 1")`)

## 6. What this plan deliberately does not change

- **No on-disk format change.** Phases 1–3 leave the WAL frame format, page format, and recovery contract untouched. Phase 4 items requiring format changes are explicitly tagged ADR-required.
- **No concurrency-model change.** The writer mutex / reader registry semantics are unchanged.
- **No new public API.** New `DbConfig` fields are additive with defensible defaults; existing call sites compile unchanged.
- **No `unsafe` outside the existing FFI boundaries.** `malloc_trim` is a single FFI call wrapped in a one-line helper.
- **Re-open path RSS is investigated, not yet fixed.** Slice E1 produces the diagnosis. The fix becomes a Phase 4 slice once we know the cause.

---

## 7. Risks and unknowns

1. **Auto-checkpoint with active readers.** Phase 1 skips the trigger entirely if `active_reader_count() > 0`. That preserves correctness but means a workload with one long-lived reader degrades to today's behavior. Phase 4 ADR 0058 (background worker) is the structural answer.
2. **Throughput sensitivity to checkpoint cadence.** Aggressive checkpointing increases fsync count. The default thresholds (16 MB / 4096 pages) need benchmark validation against ADR 0135 (async commit / group commit). If throughput regresses > 10 %, raise defaults and document the trade-off.
3. **mimalloc on Windows.** `mimalloc` works on Windows but increases binary size meaningfully. The feature stays opt-in.
4. **`malloc_trim` on heavily threaded workloads.** Calling `malloc_trim(0)` from the writer is fine; doing it from many threads concurrently can stall. The single-writer model means we never call it from more than one thread at a time.
5. **`Arc::try_unwrap` recycler effectiveness.** Active readers prevent recycling. In write-heavy workloads with brief readers, expect 80–95 % hit rate; in read-heavy mixed workloads, expect 20–40 %. Either way it's a strict improvement over no recycling.
