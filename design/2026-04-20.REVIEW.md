# DecentDB Performance & Memory Review — 2026-04-20

**Reviewer:** Copilot CLI (Claude Opus 4.7)
**Scope:** DecentDB engine core (`crates/decentdb`) and all first-party language bindings (`bindings/{python,dotnet,go,java,node,dart}`).
**Goal of this document:** Capture concrete, actionable performance and memory-management findings as **implementation slices** that a coding agent can execute without making product decisions. Each slice states *what* to change, *where* to change it, *how* to verify it, and what tests to add.

The repository's standing priorities are honored throughout:

1. Durable ACID writes
2. Fast reads
3. Stable, ergonomic multi-language integrations

No on-disk format, WAL frame format, or C ABI signature changes are proposed. All changes are internal refactors plus one Go binding correctness fix and a small Python documentation/observability change.

---

## How to use this document

- Each slice is independently mergeable.
- Slices are ordered by impact. Implement in order unless noted otherwise.
- Every slice has an explicit **Acceptance Criteria** section. Do not declare a slice done until every criterion passes.

## Engine — Slice E1: Make WAL page versions cheaply shareable (`Arc<[u8]>`)

**Severity:** High. Hot-path read allocation on every WAL-resident page read.

### Problem

`WalVersion::data` is `Vec<u8>` and `WalIndex::read_page_at_snapshot()` returns it by clone. Every reader that resolves a page through the WAL allocates and copies a fresh page (4 KB by default, configurable) instead of sharing a reference-counted buffer. A 1000-page query on WAL-resident data allocates ~4 MB that is immediately discarded.

### Files & exact changes

1. `crates/decentdb/src/wal/index.rs`
   - Change the field type:
     ```rust
     pub(crate) struct WalVersion {
         pub(crate) lsn: u64,
         pub(crate) data: std::sync::Arc<[u8]>,
     }
     ```
   - `WalVersion` is currently `#[derive(Clone, Debug, Eq, PartialEq)]`. Keep the derives. `Arc<[u8]>` implements all of these.
   - `latest_versions_at_or_before` (line 47) returns `Vec<(PageId, WalVersion)>`. With `Arc<[u8]>`, the `entry.clone()` call on line 51 becomes a refcount bump — keep the call as-is.

2. `crates/decentdb/src/wal/recovery.rs`
   - Line 53: change `pending` to hold `Arc<[u8]>` payloads:
     ```rust
     let mut pending = Vec::<(PageId, std::sync::Arc<[u8]>, u64)>::new();
     ```
   - Line 64 (Page frame): wrap once: `pending.push((frame.page_id, std::sync::Arc::from(frame.payload), next_offset));`
   - Line 71/72 (PageDelta path): the base read currently calls `pager.read_page(...)?.to_vec()`. After applying the delta, wrap the resulting `Vec<u8>` in `Arc::from(data)` before pushing. (The base-page clone optimization is handled in slice **E2**; do not couple them — the slice E1 change here is purely the type change.)
   - Line 81 (`WalVersion { lsn, data }`): `data` is now `Arc<[u8]>`. No call-site change beyond the type flowing through.

3. `crates/decentdb/src/wal/writer.rs`
   - Lines 91 and 180: `WalVersion { ... data: ... }` constructions. Change the right-hand side to `Arc::from(<existing Vec<u8>>)`. Do this *once*, at construction, not at each push.
   - Line 316: `.map(|v| v.data.clone())` is now an Arc clone (refcount bump). Update the surrounding type if it was `Vec<u8>`-typed; otherwise leave the `.clone()` call.

4. `crates/decentdb/src/wal/checkpoint.rs`
   - Line 56: `pager.write_page_direct(*page_id, &version.data)?;` — `&version.data` becomes `&Arc<[u8]>`, which derefs to `&[u8]` automatically. Confirm `write_page_direct` accepts `&[u8]` (it does today). No change needed unless the borrow check flags it; in that case use `&version.data[..]`.

5. `crates/decentdb/src/wal/mod.rs`
   - Lines 92–105 (`read_page_at_snapshot`): change return type and avoid the byte clone:
     ```rust
     pub(crate) fn read_page_at_snapshot(
         &self,
         page_id: PageId,
         snapshot_lsn: u64,
     ) -> Result<Option<std::sync::Arc<[u8]>>> {
         let index = self
             .inner
             .index
             .lock()
             .expect("wal index lock should not be poisoned");
         Ok(index
             .latest_visible(page_id, snapshot_lsn)
             .map(|version| std::sync::Arc::clone(&version.data)))
     }
     ```

6. **Propagate the return-type change to all callers** of `read_page_at_snapshot`. Use ripgrep to enumerate them and update each one. Most callers will only need `&snapshot_page[..]` instead of `&snapshot_page` if they were borrowing `&Vec<u8>`. Where a caller previously consumed the `Vec<u8>` by value, change it to take `Arc<[u8]>`; if a `Vec<u8>` is genuinely needed downstream, use `Arc::try_unwrap(arc).unwrap_or_else(|a| a.to_vec())` only at the call site that truly needs ownership of bytes.

### Acceptance criteria

- `cargo build -p decentdb` and `cargo clippy --workspace --all-targets -- -D warnings` pass.
- All existing tests pass (`cargo test --workspace`), in particular `recovery_large_wal_replay_keeps_single_version_per_page` in `crates/decentdb/src/wal/recovery.rs`.
- A new unit test (see below) demonstrates that `read_page_at_snapshot` returns Arcs that share a backing allocation across calls.

### Unit tests to add

Add the following test inside an existing `#[cfg(test)]` module in `crates/decentdb/src/wal/mod.rs` (or create one if none exists for that file):

```rust
#[test]
fn read_page_at_snapshot_shares_backing_allocation() {
    // Arrange: open an in-memory DB, write one page, take two snapshot reads.
    // Assert: Arc::ptr_eq on the returned Arc<[u8]> values is true.
    //         (Use Arc::strong_count >= 2 as a secondary signal.)
}
```

The test must:

- Use `crate::vfs::mem::MemVfs`.
- Write at least one page through the public engine API so it lives in the WAL (no checkpoint between write and read).
- Read the page twice via `read_page_at_snapshot` at the same snapshot LSN.
- Assert `Arc::ptr_eq(&first, &second)`.

Also add a regression test in `crates/decentdb/src/wal/index.rs` that constructs two `WalVersion` clones from the same `Arc<[u8]>` and asserts `Arc::ptr_eq(&a.data, &b.data)`.

## Engine — Slice E2: Eliminate per-delta base-page clones during WAL recovery

**Severity:** High. Causes `open()` RSS to scale with uncheckpointed WAL size (matches prior memory probe data: 64 MB DB + 128 MB WAL → ~134 MB RSS delta on open).

### Problem

`crates/decentdb/src/wal/recovery.rs:66-75` (the `FrameType::PageDelta` arm) clones the entire base page on every delta frame:

```rust
let base = if let Some(version) = index.latest_visible(frame.page_id, u64::MAX) {
    version.data.clone()
} else {
    pager.read_page(frame.page_id)?.to_vec()
};
let data = apply_page_delta(&base, &frame.payload)?;
pending.push((frame.page_id, data, next_offset));
```

For N deltas to the same page, this performs N full page copies during open, even though only the latest result is retained per page (see comment at lines 78–80).

### Files & exact changes

`crates/decentdb/src/wal/recovery.rs` only. Inside `initialize_or_recover`:

1. Introduce a local cache that maps `PageId -> Vec<u8>` of the most recent reconstructed bytes for **uncommitted, pending** frames. This cache is in addition to `pending` and lives only for the duration of `initialize_or_recover`. Use `std::collections::HashMap<PageId, Vec<u8>>` (small N, simple hot path).

2. In the `FrameType::PageDelta` arm, resolve the base in this priority order:
   1. If the per-page pending cache has bytes for this `page_id`, use a mutable borrow into the cache and apply the delta in place using a scratch buffer (do not clone). After applying, write the result back into the cache entry.
   2. Else if `index.latest_visible(...)` returns a version, take **one** owned `Vec<u8>` copy from it (this is the unavoidable transition from shared/recovered state into the mutable scratch). Insert into the cache.
   3. Else read the page from the pager (`pager.read_page(...)?.to_vec()`) and insert into the cache.

3. In the `FrameType::Page` arm, replace any cache entry for `frame.page_id` with `frame.payload.clone()` (or `frame.payload.to_vec()` if `frame.payload` is already a `Vec<u8>` and you want to avoid the clone — choose the variant that compiles without warnings under `-D warnings`).

4. In the `FrameType::Commit` arm, **drain both** structures:
   - Drain `pending` into `index.add_version(...)` exactly as today, but the data values pushed should now come from the cache, not from re-cloning. The simplest approach: in steps (2) and (3) above, push `(page_id, /* placeholder */, next_offset)` into `pending` and store the bytes only in the cache; then on Commit, look up final bytes in the cache by page_id, `take` them out of the cache, and pass them to `index.add_version`.
   - After draining, clear the cache.

5. In the `FrameType::Checkpoint` arm, also clear the cache (in addition to the existing `pending.clear()`).

6. **Add a hard upper bound on `pending.len()` to defend against malformed/adversarial WAL files.** Cap at `1_000_000` entries. On overflow, return `DbError::corruption(...)` with a message like `"WAL recovery aborted: more than 1,000,000 uncommitted page frames before commit"`. Apply the same cap to the per-page cache size — if it exceeds 1,000,000 distinct pages between commits, return the same corruption error.

### Acceptance criteria

- All existing tests pass.
- The pre-existing `recovery_large_wal_replay_keeps_single_version_per_page` test still passes unchanged.
- New tests (below) pass.
- Recovery memory growth is no longer linear in delta-frame count for the same page.

### Unit tests to add

Add to `crates/decentdb/src/wal/recovery.rs` `mod tests`:

1. `recovery_many_deltas_to_same_page_does_not_clone_base_per_delta`
   - Construct a WAL with 1 Page frame followed by 1024 PageDelta frames against the same page, then a single Commit frame.
   - Run `initialize_or_recover`.
   - Assert that the resulting `WalIndex` contains exactly one version for that page.
   - Assert the final reconstructed bytes match a reference computed by sequentially applying the deltas independently.

2. `recovery_rejects_pending_overflow`
   - Build a synthetic WAL that has 1,000,001 distinct page frames before any Commit frame.
   - Assert `initialize_or_recover` returns `DbError::corruption(...)`.
   - This test may be marked `#[ignore]` if generating the WAL is too slow for default `cargo test`; if so, gate it behind a `cfg(feature = "slow-tests")` and document the gating in `crates/decentdb/Cargo.toml`. Default behavior is to keep it unignored if it runs in under a few seconds.

## Engine — Slice E3: Stop deep-cloning catalog/runtime per statement

**Severity:** Medium. Per-statement allocation churn on autocommit workloads.

### Problem

`EngineRuntime`'s `Clone` implementation (`crates/decentdb/src/exec/mod.rs:504-537`) deep-clones `tables`, `temp_table_data`, `indexes`, and related state on every statement execution. For an autocommit single-row `SELECT` against a 1 MB catalog, this is ~1 MB of allocation/free per call.

### Files & exact changes

`crates/decentdb/src/exec/mod.rs` only.

1. Identify the fields on `EngineRuntime` whose values are read-mostly across statements. Wrap each of those fields in `Arc<...>`:
   - `tables`, `temp_table_data`, `indexes` are the named offenders. Audit the surrounding struct to confirm no other field is also a per-statement deep-clone offender (anything `> ~1 KB` typical size, cloned on every statement).

2. For each wrapped field, expose a copy-on-write helper:
   ```rust
   fn tables_mut(&mut self) -> &mut TablesMap {
       std::sync::Arc::make_mut(&mut self.tables)
   }
   ```
   Use `Arc::make_mut` so reads stay zero-copy and only mutations trigger a clone of the affected map.

3. Update mutation sites to call the new `*_mut()` accessors. Read sites should be left as `&self.tables` etc. (they will deref through `Arc` automatically).

4. The existing `Clone` impl becomes a refcount bump. If `derive(Clone)` was used, keep it. If `Clone` was hand-written specifically to deep-copy these fields, simplify to the derived impl.

5. **Do not** change the externally observable transactional semantics. Specifically:
   - The fork/merge points where the runtime is cloned for transaction support must continue to produce a snapshot that does not see post-fork mutations of the original. With `Arc::make_mut`, this is preserved automatically because writers clone-on-write before mutating.
   - If any mutation site reaches into nested structures via `&mut`, audit that the outer `Arc::make_mut` is called first; otherwise the mutation will be applied to shared state.

### Acceptance criteria

- `cargo test --workspace` passes.
- A new microbench-style unit test (see below) shows that `EngineRuntime::clone()` does not allocate when no mutations have occurred.
- No transaction-isolation tests regress.

### Unit tests to add

Add to `crates/decentdb/src/exec/mod.rs` `#[cfg(test)] mod tests`:

1. `engine_runtime_clone_is_shallow_when_unmutated`
   - Construct an `EngineRuntime` populated with a few tables.
   - Clone it.
   - Assert `Arc::strong_count(&runtime.tables) == 2` (and similar for the other Arc-wrapped fields).

2. `engine_runtime_cow_isolates_mutations`
   - Clone an `EngineRuntime`, mutate the clone's tables, and assert the original is unchanged.
   - Assert the strong count for the original's tables drops back to 1 after the clone is dropped.

3. `engine_runtime_autocommit_select_does_not_grow_strong_counts_unboundedly`
   - Run N (e.g., 100) autocommit SELECTs through the public engine API on a small DB.
   - Assert `Arc::strong_count` for the catalog Arc stays bounded (≤ small constant) at the end. This guards against accidental Arc retention by held clones.

## Engine — Slice E5: Bound `cached_payloads` and `TempSchemaState` clones

**Severity:** Low–Medium. Per-session memory growth that is not bounded by `config.cache_size_mb`.

### Problem

- `crates/decentdb/src/exec/mod.rs:990` — `cached_payloads` map grows unbounded across statements.
- `crates/decentdb/src/db.rs:237` (`apply_to_runtime`) and `db.rs:245` (`update_from_runtime`) deep-clone all temp tables, indexes, and views on every statement.

### Files & exact changes

1. **`cached_payloads` cap.** In `crates/decentdb/src/exec/mod.rs` at the `cached_payloads` declaration site (line 990 and surrounding struct):
   - Add a configurable maximum. Source the value from `DbConfig` — add a new field `cached_payloads_max_entries: usize` to `crates/decentdb/src/config.rs` with default `1024`. This is an internal cap, not a stable API addition, but expose a setter for tests.
   - Replace the bare `HashMap` with a simple LRU. Use a `linked_hash_map`-style structure built from `std::collections::HashMap` + `std::collections::VecDeque<Key>` to keep the dependency footprint at zero (do not add a new crate). On each access, move the key to the back of the deque. On insertion, if `len() > cap`, pop from the front and remove from the map.
   - Document the cap in a `///` comment on the field: "Per-session cache; capped at `cached_payloads_max_entries`. Eviction is LRU."

2. **`TempSchemaState` Arc-ification.** In `crates/decentdb/src/db.rs` around lines 230–250:
   - Wrap the temp-table collections in `Arc<...>`.
   - In `apply_to_runtime` and `update_from_runtime`, replace `.clone()` with `Arc::clone(&...)` for read-mostly fields. Use `Arc::make_mut` at mutation points.
   - The semantics are identical to slice E3 but for the `TempSchemaState` rather than `EngineRuntime`.

- New unit tests (below) pass.

## Engine — Slice E6: Document C ABI handle-leak responsibilities

**Severity:** Low. Documentation/clarity improvement for FFI consumers.

### Problem

`crates/decentdb/src/c_api.rs:761` returns a `Box::into_raw` `StmtHandle` from `ddb_db_prepare`. If a host process crashes between prepare and `ddb_stmt_free`, the handle leaks (which can hold a cloned Db plus a result set). This is expected FFI behavior, but is not documented in `include/decentdb.h` next to the function declaration.

### Files & exact changes

1. `include/decentdb.h` — Above the `ddb_db_prepare` declaration, add a comment block explicitly stating:
   - Ownership transfers to the caller.
   - The caller **must** call `ddb_stmt_free` exactly once for every successful return.
   - The handle retains references to internal database state; failing to free it will leak that state until process exit.
   - It is not safe to call `ddb_stmt_free` on the same handle from multiple threads concurrently.

2. Apply the same comment treatment to `ddb_db_open`, `ddb_db_close`/`ddb_db_free`, `ddb_txn_*` constructors, and any other `Box::into_raw`-returning entry point in `crates/decentdb/src/c_api.rs`. Audit by ripgrepping `Box::into_raw` in `c_api.rs`.

3. Mirror these notes in the rustdoc above the corresponding `extern "C" fn` declarations in `crates/decentdb/src/c_api.rs` so they appear in `cargo doc` output.

### Acceptance criteria

- No code behavior changes.
- `cargo doc --no-deps -p decentdb` succeeds and the new doc comments render.
- `include/decentdb.h` continues to be a valid C header (no syntax regressions). If the project has a header-validation step (search for one in `scripts/` and `tests/`), it must still pass.

### Unit tests to add

None — documentation-only slice.

## Bindings — Slice B1: Fix Go cgo blob-pointer rule violation

**Severity:** High (silent corruption risk; will crash under `GODEBUG=cgocheck=2`).

### Problem

`bindings/go/decentdb-go/driver.go:840`:

```go
C.ddb_stmt_bind_blob(s.stmt, idx, (*C.uint8_t)(unsafe.Pointer(&v[0])), C.size_t(len(v)))
```

Passing `&v[0]` of a Go-managed byte slice into C violates cgo pointer-passing rules, even for byte slices, because the Go runtime is allowed to relocate the backing array. The current C side copies synchronously, but cgo cannot prove that and `GODEBUG=cgocheck=2` will trip.

### Files & exact changes

`bindings/go/decentdb-go/driver.go` only.

1. Replace the call site with one of:
   - **Preferred (Go ≥ 1.21):** Use `runtime.Pinner` to pin the slice for the duration of the call. This avoids a copy and is the modern idiomatic fix.
     ```go
     var pinner runtime.Pinner
     defer pinner.Unpin()
     if len(v) > 0 {
         pinner.Pin(&v[0])
     }
     // ... existing C call
     ```
   - **Fallback if module `go.mod` targets < 1.21:** Use `C.CBytes(v)` and `C.free` via `defer`. This adds one copy per blob bind but is universally safe.
     ```go
     if len(v) == 0 {
         C.ddb_stmt_bind_blob(s.stmt, idx, nil, 0)
     } else {
         cbuf := C.CBytes(v)
         defer C.free(cbuf)
         C.ddb_stmt_bind_blob(s.stmt, idx, (*C.uint8_t)(cbuf), C.size_t(len(v)))
     }
     ```
   Check `bindings/go/decentdb-go/go.mod` first. If `go 1.21` or newer, use `runtime.Pinner`. Otherwise use `C.CBytes`. Do not raise the `go` directive in `go.mod` as part of this slice.

2. Apply the same pattern audit to **every** `unsafe.Pointer(&slice[0])` site in the Go binding. Ripgrep for `unsafe.Pointer(&` under `bindings/go/`. Each hit must either be pinned or replaced with `C.CBytes`.

### Acceptance criteria

- `go test ./...` from `bindings/go/decentdb-go/` passes.
- The same test suite passes under `GODEBUG=cgocheck=2 go test ./...`.
- A new test (below) explicitly exercises a large-blob bind to ensure the path is covered.

### Unit tests to add

In `bindings/go/decentdb-go/`:

1. `TestBindLargeBlob_NoCgoPointerViolation`
   - Open a DB, prepare an `INSERT INTO t(b BLOB) VALUES (?)`, bind a 1 MiB byte slice, execute, then read the row back.
   - Assert the round-tripped bytes are byte-equal.
   - Document in a comment that this test is the regression guard for the cgo pointer rule fix.

2. Add a CI invocation (in the same test file or a `Makefile`/`go test` flag in `bindings/go/decentdb-go/Makefile` if it exists) to run with `GODEBUG=cgocheck=2`. If no Makefile exists, document the command in `bindings/go/decentdb-go/README.md`.

## Bindings — Slice B2: Eliminate Go `SetFinalizer` double-free race

**Severity:** Medium.

### Problem

`bindings/go/decentdb-go/driver.go:191` registers a finalizer that calls `Close()`. The explicit `Close()` method (around line 196) does not clear the finalizer. If a user calls `Close()` and the wrapper is later collected, the finalizer runs again and calls `ddb_db_free` on an already-freed handle.

### Files & exact changes

`bindings/go/decentdb-go/driver.go` only.

1. In `Close()` (around line 196):
   - At the very start, attempt to atomically transition a `closed uint32` field on the wrapper from `0` to `1` using `sync/atomic.CompareAndSwapUint32`. If the swap fails, return `nil` (already closed — idempotent close).
   - On success, immediately call `runtime.SetFinalizer(d, nil)` to detach the finalizer.
   - Then perform the existing `ddb_db_free` work.

2. Add the `closed uint32` field to the `DB` struct.

3. The finalizer body should also check the same `closed` flag (defensively) and short-circuit if already closed:
   ```go
   runtime.SetFinalizer(wrapper, func(d *DB) {
       if atomic.LoadUint32(&d.closed) == 1 {
           return
       }
       _ = d.Close()
   })
   ```

4. Apply the same pattern to any other binding type that uses `runtime.SetFinalizer` in `bindings/go/`. Ripgrep `runtime.SetFinalizer` under `bindings/go/`.

### Acceptance criteria

- `go test ./...` passes from `bindings/go/decentdb-go/`.
- New tests (below) pass.

### Unit tests to add

In `bindings/go/decentdb-go/`:

1. `TestDoubleClose_IsIdempotent`
   - Open a DB. Call `Close()` twice. Assert no error and no panic.

2. `TestExplicitCloseDetachesFinalizer`
   - Open a DB, call `Close()`, force `runtime.GC()` and `runtime.GC()` again (twice to flush finalizers).
   - The test should not panic and should not double-free. This is a behavioral assertion via instrumentation: add a package-level `atomic.Uint32` counter incremented by a test-only hook in the close path (gate behind a build tag if necessary), and assert it equals 1 after the GC cycles.

## Bindings — Slice B3: Document and observe Python statement-cache hit rate

**Severity:** Low.

### Problem

`bindings/python/decentdb/__init__.py:3186` evicts FIFO from `_stmt_cache`. Applications that build SQL with embedded literals (instead of parameters) churn `ddb_db_prepare`/`ddb_stmt_free` on every call, defeating the cache silently.

### Files & exact changes

`bindings/python/decentdb/__init__.py` only.

## Out of scope for this batch

The following were considered and intentionally excluded to keep this batch a single patch release:

- **ADR-required changes** (per `AGENTS.md` § 7): any WAL format change, any change to `Send`/`Sync` boundaries, any new dependency, any C ABI signature change. Specifically, do **not** introduce a new crate (e.g., `linked-hash-map`) for the LRU in slice E5 — implement it with `std` collections.
- **Engine-wide allocator instrumentation.** Worth doing as a follow-up for evidence-based perf work, but is its own design exercise.
- **Cross-binding pooled-connection semantics.** Each binding's connection model is currently consistent with the engine; a unified pool is a feature, not a fix.
