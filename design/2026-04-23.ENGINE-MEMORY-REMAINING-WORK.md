# Engine Memory — Remaining Work to Fully Resolve the 16 GB / 10 M-row Concern
**Date:** 2026-04-23
**Status:** Design / hand-off document
**Companion:** [`design/2026-04-22.ENGINE-MEMORY-PLAN.md`](2026-04-22.ENGINE-MEMORY-PLAN.md), [`design/adr/0143-on-disk-row-scan-executor.md`](adr/0143-on-disk-row-scan-executor.md)

---

## 0. Why this document exists

The user's headline workload — **10 M rows of small tuples on a 16 GB host** —
must run with stable resident memory roughly proportional to `cache_size_mb`,
not to total row count. Today (2026-04-23) the engine peaks at ~2.6 GB on a
145 MB DB and re-opens a 36 MB DB at ~1 144 MB RSS. The work landed in the
2026-04-23 batches (Phase 1 + M5 + M6 + M7 + ADR 0058 + 0141/0142 scaffolds +
ADR 0143 Phase A + Phase B opt-in) reduces the slope but does **not** change
the ceiling.

This document enumerates **every remaining work item** required to bring the
ceiling down to the ADR 0143 acceptance criteria:

- 5 M-row re-open: < **200 MB RSS** before any query (today: 1 144 MB)
- 10 M-row load with `cache_size_mb=64`: peak < **1 GB** (today: 2.6 GB)

Each item is sized, scoped, and called out with the exact files, structs,
and invariants involved so that a follow-up agent can pick it up without
re-deriving the analysis.

---

## 1. The single root cause that must be fixed

`crates/decentdb/src/exec/mod.rs:472`:

```rust
pub(crate) struct EngineRuntime {
    pub(crate) tables: Arc<BTreeMap<String, Arc<TableData>>>,
    // ...
}

// line 130
pub(crate) struct TableData {
    pub(crate) rows: Vec<StoredRow>,    // every row, every table, in RAM
}

pub(crate) struct StoredRow {
    pub(crate) row_id: i64,
    pub(crate) values: Vec<Value>,
}
```

`Db::open` walks the on-disk overflow chain for every persisted table and
decodes the entire payload into `Vec<StoredRow>`. The disk format is a
checkpoint of an in-memory snapshot, not a queryable on-disk row layout.

Every other memory concern (page cache, WAL retention, allocator overhead,
delta scratch, SmallVec) is secondary. **Until `tables` no longer holds
every row, the user's 10 M-row workload cannot fit in 16 GB.**

## 2. Outstanding work, in dependency order

### 2.1 Phase B (continuation): per-table on-demand load — **REQUIRED**

**Status:** opt-in scaffold only. Today's `defer_table_materialization=true`
flag works but is misleading: the *first* SQL statement on any deferred
table calls `ensure_deferred_tables_loaded`, which materializes **every
still-deferred table in one shot**. So peak RSS during that first statement
is unchanged.

**Acceptance criterion:** 5 M-row re-open + a `SELECT * FROM small WHERE id=42`
on a small (≤10 k-row) table must allocate ≤ 200 MB RSS, even when there
are other huge tables present.

**Files & change shape:**

1. `crates/decentdb/src/db.rs::ensure_deferred_tables_loaded` (line 2615)
   - Currently signature `fn ensure_deferred_tables_loaded(&self) -> Result<()>`
     loads everything in `self.deferred_tables`.
   - Change to `fn ensure_tables_loaded(&self, names: &[&str]) -> Result<()>`
     that loads only the intersection of `names` with `deferred_tables`.
   - Add `fn ensure_all_tables_loaded(&self)` for paths that genuinely
     scan everything (`PRAGMA integrity_check`, `list_tables`, etc.).

2. `crates/decentdb/src/exec/mod.rs::materialize_deferred_tables_with_store`
   (line 790) — accept a `filter: Option<&BTreeSet<String>>` parameter.
   When `Some`, only materialize matching tables. When `None`, current
   "load everything" behavior.

3. **Statement-target analysis:** every caller of
   `ensure_deferred_tables_loaded` (~12 sites in `db.rs`) needs a
   pre-execute pass that walks the parsed/prepared statement and
   collects referenced base table names. The cleanest hook is in
   `crates/decentdb/src/sql/plan.rs` (or wherever `LogicalPlan` lives)
   adding `LogicalPlan::referenced_tables() -> Vec<&str>`.

4. **Index materialization tie-in:** `materialize_deferred_tables_with_store`
   currently calls `rebuild_indexes(page_size)?` which rebuilds every index
   in the catalog. With per-table loading, we must rebuild only indexes
   whose `table_name` matches a freshly-loaded table. This already lines
   up with `rebuild_stale_indexes` (line 1397) — that is the entry point.
   Drop the unconditional `rebuild_indexes` call and rely on
   `mark_indexes_stale_for_table` + `rebuild_stale_indexes` post-load.

5. **Test:** copy `defer_table_materialization_skips_eager_load_at_open`
   and extend: create two tables (small, large), open with the flag,
   query only the small one, assert `rows_in_memory_count` equals only
   the small table's row count.

**Risk:** statement-target analysis must handle CTEs, subqueries, JOINs,
view expansion, and triggers. Missing a referenced table will surface as
"unknown table" at runtime. Mitigation: when in doubt, fall back to
`ensure_all_tables_loaded` (correctness over performance). Add a
diagnostic counter on the conservative-fallback path so we can measure
how often it triggers.

**Estimated impact:** alone, this brings the 5 M-row re-open + targeted
query under the 200 MB acceptance criterion **only when the user's
working set is actually small**. For OLAP-style scans of huge tables,
Phase D is also required.

### 2.2 Phase C: persisted primary-key index — **REQUIRED**

**Status:** not started.

**Why required:** Phase B reduces `Db::open` to nearly zero allocation
for unused tables, but the moment a query touches a 10 M-row table it
re-materializes that whole table to satisfy a `WHERE id = ?`. The PK
lookup path in `TableData::row_index_by_id` (line 135) requires the
`Vec<StoredRow>` to exist.

**Files & change shape:**

1. **On-disk B+Tree.** A new persistent btree keyed by `row_id` →
   `(page_id, slot)` for every base table. The existing
   `crates/decentdb/src/btree/` infrastructure already provides
   page-backed B+Tree primitives — reuse it instead of rolling new code.

2. **Catalog change (versioned, requires migration).** Each
   `TableSchema` gains `pk_index_root: Option<PageId>`. Per AGENTS.md
   §7 + ADR 0131, this is a file-format change and requires:
   - bump the format version in `decentdb-migrate`
   - add a read-only forward parser for the previous version
   - new ADR (suggested number 0144) covering the on-disk shape

3. **Write path.** Every INSERT/UPDATE/DELETE updates the new index in
   the same WAL transaction that updates the row payload. The existing
   `persist_to_db` (`exec/mod.rs:828`) is the orchestrator.

4. **Read path.** `TableData::row_by_id` becomes a wrapper that:
   - if the table is `Resident`, uses today's `Vec<StoredRow>` lookup;
   - if the table is `Paged` (introduced in Phase D), reads
     `pk_index_root`, descends the on-disk btree, fetches the page,
     decodes the slot, returns a `Cow<StoredRow>`.

**Acceptance criterion:** `SELECT * FROM huge WHERE id = ?` against a
10 M-row table allocates ≤ a few KB, regardless of table size.

**Risks:**
- File format migration is a one-way door. Get review on the ADR before
  cutting in.
- Index recovery must survive crashes mid-checkpoint. Reuse existing
  WAL recovery semantics; do not invent new persistence.

### 2.3 Phase D: on-disk row format + paged TableRowSource — **REQUIRED**

**Status:** not started. This is the largest remaining slice.

**Why required:** Phase C makes single-row lookups cheap, but a
sequential scan (`SELECT * FROM huge`) and a hash aggregate
(`SELECT a, COUNT(*) FROM huge GROUP BY a`) still need to walk every
row. With today's format that means decoding the entire overflow chain
into `Vec<StoredRow>`. With a paged format the executor can stream
rows page-by-page, bounded by the page cache.

**Files & change shape:**

1. **New row-source enum:**
   ```rust
   pub(crate) enum TableRowSource {
       Resident(Arc<TableData>),         // small tables (≤ DbConfig::table_resident_max_rows)
       Paged { manifest: Arc<TablePageManifest> },
   }
   ```
   Replaces `tables: BTreeMap<String, Arc<TableData>>` with
   `tables: BTreeMap<String, TableRowSource>`.

2. **`TablePageManifest`** — list of leaf-page IDs for the table,
   read against a snapshot. Iterating yields `(row_id, &[Value])`
   without materializing intermediate `Vec<StoredRow>`.

3. **Executor scan path.** Every site that today does
   `&runtime.tables[name].rows` (see `grep -n '\.rows'` in `exec/mod.rs`
   — **139 hits**, the largest single refactor in this plan) must
   switch to `for row in source.iter()`. The iterator yields a
   short-lived `RowRef<'_>` borrowing into a page in the cache; the
   page remains pinned for the duration of one row.

4. **Write path.** New on-disk format must support per-row update without
   rewriting the whole table. Two options:
   - (a) Append-only with tombstones + periodic compaction (simple,
     write-amp friendly, complicates VACUUM).
   - (b) Slotted pages with in-place update (closer to PostgreSQL,
     more complex, fewer write-amp surprises).
   Pick (a) initially; (b) can be a follow-up. Decision belongs in a new
   ADR (suggested number 0145).

5. **Compatibility.** Read-only support for the legacy
   `Vec<StoredRow>`-checkpoint format must remain in
   `decentdb-migrate` so existing on-disk databases can be migrated
   forward.

**Acceptance criterion:** 10 M-row INSERT load with `cache_size_mb=64`
peaks < 1 GB RSS. A `SELECT a, COUNT(*) FROM huge GROUP BY a` over the
same data allocates O(distinct values), not O(rows).

**Risks:**
- 139 call sites in `exec/mod.rs` reference `.rows`. Burn it down with
  a `RowSource::iter()` adapter that returns the same shape current
  code expects, then incrementally migrate hot paths to the borrowing
  form. Land in many small PRs.
- Sequential scan performance can degrade 10× without prefetching.
  Mandatory: a `SnapshotPageStore::prefetch(start, end)` hint that the
  page cache honors before iteration begins.
- Hash aggregate / sort spill paths (`exec/mod.rs` aggregate executor)
  also allocate per-row and can themselves blow memory; out of scope
  for Phase D but tracked separately as the **executor scratch
  buffer** finding from the original Engine Memory Plan.

---

### 2.4 Phase E: remove `tables` BTreeMap entirely — **CLEANUP**

After Phases B + C + D, `EngineRuntime::tables` reduces to a small
cache for residency-promoted tables and DROP TABLE state. Deprecate
`materialize_deferred_tables_with_store`. This is mechanical cleanup
and does not move the memory needle further; it removes a footgun.

---

### 2.5 Executor scratch — **PARALLEL TRACK**

Even after the row-source becomes paged, hash aggregate, sort, and
GROUP BY allocate `Vec<...>` proportional to group cardinality. Today
`SELECT artist_id, COUNT(*) FROM songs GROUP BY artist_id` over 5 M
rows can allocate ~80 MB of hash-table state alone. This is **not a
Phase B/C/D problem** but is on the same critical path for the user's
real workloads.

**Files:**
- `crates/decentdb/src/exec/mod.rs` aggregate path (search for
  `HashMap<.*Vec<Value>>`)
- Sort: search for `.sort_by` over per-row vectors

**Mitigation:** spill-to-disk via a temp-file row-store keyed by
`DbConfig::temp_dir`. Bounded by a new `DbConfig::executor_memory_mb`.
Track as a follow-up ADR after Phase D lands.

---

### 2.6 ADR 0141 — paged WAL index spill — **OPTIONAL POST-PHASE D**

Today's `WalIndex` holds `(page_id → SmallVec<[WalVersion; 1]>)` for
every page touched between checkpoints. With Phase 1 (M1) capping
`wal_versions` at 4096 the practical ceiling is ~16 MB, which is
acceptable. ADR 0141's spill backend remains scaffolded; only land it
if real-world workloads show the WAL index growing without bound under
sustained writer pressure (e.g. long-running readers preventing
checkpoint per ADR 0019). Tracked separately.

---

### 2.7 ADR 0142 — per-engine allocator — **OPTIONAL POST-PHASE D**

Scaffold landed (`crates/decentdb/src/alloc.rs`). Threading
`EngineAlloc` through the engine is a wide refactor with marginal
return until Phases B+C+D are done. Re-evaluate then.

## 3. Suggested PR sequencing

1. **PR 1 — Phase B continuation (per-table on-demand).** Statement
   target analysis + filtered materialization. Land behind the
   existing `defer_table_materialization` flag so it ships dark until
   the working-set assumption is validated against the .NET binding
   benchmark suite.

2. **PR 2 — ADR 0144 (on-disk PK index format).** Pure design doc.
   Get review.

3. **PR 3 — Phase C implementation.** Persistent PK index, write-path
   wiring, recovery test. Behind a new `DbConfig::persistent_pk_index`
   flag (default off until 1 release of soak time).

4. **PR 4 — ADR 0145 (paged on-disk row format).** Pure design doc.

5. **PRs 5..N — Phase D implementation.** Land
   `TableRowSource::Paged` for one table at a time using the existing
   `Resident` path as the fallback. The migration tool reads the old
   format and writes the new one in-place during a one-shot upgrade.

6. **PR final — Phase E cleanup.** Delete dead code paths. Flip the
   default for `defer_table_materialization` and
   `persistent_pk_index` to `true`. Bump major version per semver.
