# DecentDB Performance Improvement Plan

**Date:** 2026-05-18
**Status:** Draft implementation plan
**Scope:** Recover benchmark regressions, improve fair SQLite comparisons, and keep durability guarantees intact.

## Objective

Improve DecentDB performance in a fair, durable, and repeatable way. The immediate target is not to make SQLite look worse. The target is to make DecentDB's measured hot paths closer to what SQLite already does well: compiled prepared plans, tight B+Tree/index lookups, minimal value materialization, and efficient durable WAL writes.

This plan covers two separate regressions observed in `benchmarks/rust-baseline` and the broader engine work needed for README-visible SQLite comparisons.

## Non-Negotiables

- Do not weaken ACID durability for any metric labeled durable.
- Do not compare DecentDB `AsyncCommit` against SQLite `synchronous=FULL` as a durable apples-to-apples claim.
- Do not hide losing metrics by changing charts or cherry-picking axes.
- Keep default-profile results separate from tuned-profile results.
- Record exact command lines, commit SHA, engine version, host, and benchmark profile for every published result.
- Add an ADR before on-disk format, WAL format, C ABI, major dependency, or concurrency contract changes.

## Regression 1: Read Path Slowdown From Deferred Row Sources

### Hypothesis

The Apr 29 read regression was caused by paged row storage and deferred materialization changes that reduce resident memory but force repeated row-source loading, index rebuild, or re-deferral work across autocommit read statements. Historical data supports this: read steps became about `2x` to `5x` slower while peak RSS dropped from about `2.2 GB` to about `566 MB` at full scale.

### Investigation

1. Run `benchmarks/rust-baseline` with default `DbConfig`.
2. Add a temporary local-only benchmark switch or code patch for:
   - `defer_table_materialization=false`
   - `paged_row_storage=false`
   - `defer_table_materialization=true`, `paged_row_storage=true`, no re-deferral after reads
   - `defer_table_materialization=true`, `paged_row_storage=true`, retain row sources only while memory budget allows
3. Measure `smoke`, `medium`, and `full` for:
   - total runtime
   - per-step query time
   - peak RSS
   - database size and WAL size
4. Use `perf record -g` on the `full` query steps if the config toggles do not isolate the cost.
5. Add narrow counters under an internal benchmark feature for:
   - row-source materializations
   - re-deferrals
   - index rebuilds
   - table payload reads
   - deferred paged locator cache hits and misses

### Fix Direction

1. Split memory-saving behavior from hot read behavior:
   - Keep deferred materialization for open-time memory.
   - Do not automatically re-defer row sources after every autocommit read when the same handle is actively running repeated hot reads.
   - Add a bounded row-source residency policy instead of an unconditional drop.
2. Preserve memory wins with explicit pressure management:
   - retain recently used row sources up to a configurable memory budget
   - evict least-recently-used row sources when the budget is exceeded
   - keep `release_freed_memory_after_checkpoint` behavior for checkpoint cleanup
3. Keep index state stable for resident row sources:
   - avoid marking indexes stale on harmless re-deferral cycles
   - avoid rebuilding indexes repeatedly for unchanged persisted tables
4. Add tests that prove:
   - repeated read-only statements do not repeatedly materialize the same row source
   - schema changes still invalidate prepared statements
   - memory-constrained mode can evict row sources and still returns correct results
   - concurrent readers with checkpoints do not regress into the old overflow mismatch bug

### Success Criteria

- `rust-baseline full` query steps return near the Apr 27-28 timing regime without returning to unbounded RSS.
- `query_aggregate_durations` target: within `1.25x` of the best early historical run.
- `query_top10_artists_by_songs` target: within `1.25x` of the best early historical run.
- `query_top10_albums_by_songs` target: within `1.25x` of the best early historical run.
- Peak RSS target: materially below the old `2.2 GB` full-scale peak, unless the benchmark profile is explicitly labeled `resident-hot-read`.

## Regression 2: Current Insert/Write Throughput Slowdown

### Hypothesis

The current `2.5.0` branch has a separate insert/write regression. The raw Rust baseline already uses prepared statements and explicit transactions, so the slowdown is likely in row validation, value coercion, semantic type handling, constraint checks, index maintenance, row encoding, WAL serialization, or commit-time persistence work.

### Investigation

1. Run current `rust-baseline` for `smoke`, `medium`, and `full` as the initial write baseline.
2. Compare against latest historical `2.3.1` reports:
   - `seed_artists`
   - `seed_albums`
   - `seed_songs`
3. Profile `seed_songs` first because it is the largest write path:
   ```bash
   perf record -g -- cargo run --manifest-path benchmarks/rust-baseline/Cargo.toml --release --bin rust-baseline -- --scale medium --out-dir .tmp/rust-baseline-current --db-path .tmp/rust-baseline-current/run-rust-medium.ddb
   perf report
   ```
4. Trace syscalls for durable write path:
   ```bash
   strace -c -e fsync,fdatasync,pwrite64 cargo run --manifest-path benchmarks/rust-baseline/Cargo.toml --release --bin rust-baseline -- --scale medium --out-dir .tmp/rust-baseline-current --db-path .tmp/rust-baseline-current/run-rust-medium.ddb
   ```
5. Compare hot functions before and after recent semantic data type work.
6. Add local counters for:
   - row validation calls
   - value coercions
   - generated column checks
   - index update preparation
   - row encoding allocations
   - WAL frame allocations and bytes written

### Fix Direction

1. Fast-path prepared inserts where the statement shape is stable:
   - cache table schema resolution
   - cache column index mapping
   - cache value coercion plan
   - cache required constraint checks
   - cache index update plan
2. Avoid repeated generic semantic coercion work when prepared parameter types are stable.
3. Reuse row encoding buffers for prepared insert loops.
4. Reduce allocation churn in index update preparation.
5. Verify WAL writes are batched per explicit transaction and not accidentally doing per-row durable work.
6. Keep semantic type correctness tests in place while adding microbenchmarks for binding-native values.

### Success Criteria

- Current `seed_songs` returns to within `1.25x` of latest historical throughput.
- No regression in semantic data type tests.
- No regression in crash recovery, WAL replay, or binding smoke tests.
- Native insert throughput in `crates/decentdb/benches/embedded_compare.rs` improves without changing benchmark fairness.

## Engine Work

### 1. Prepare-Time Physical Read Plans

Biggest win: prepare-time physical read plans.

Today `execute_read_statement` walks a chain of runtime pattern matchers, including benchmark-shaped fast paths in `crates/decentdb/src/exec/mod.rs:3711`. Prepared statements should cache resolved table IDs, column offsets, index handles, parameter slots, and a physical operator enum. Point reads and indexed joins should bypass AST matching on every execution.

Implementation outline:

- Add a prepared physical plan enum for read statements.
- Compile eligible statements at `Db::prepare` / transaction `prepare` time.
- Store schema cookie and index epoch with the prepared plan.
- Fall back to current execution if schema/index state has changed.
- Start with:
  - primary-key point projection
  - single-column indexed projection
  - simple indexed nested-loop join
  - simple range scan
  - simple aggregate count/sum

### 2. Generalize Benchmark Fast Paths

Generalize the benchmark fast paths.

Functions like `try_execute_benchmark_history_query` and `try_execute_benchmark_report_query` should become normal planner capabilities: indexed nested-loop join, grouped aggregate, projection-aware materialization. Keeping benchmark-named SQL recognizers is hard to defend as "fair".

Implementation outline:

- Replace benchmark-specific recognizers with planner rules.
- Name rules by physical capability, not workload.
- Add shape tests that prove the benchmark SQL uses the generic rules.
- Add non-benchmark SQL tests that use the same rules with different table and column names.

### 3. Reduce Result Materialization Cost

Reduce result materialization cost.

SQLite's hot prepared lookups are extremely tight. DecentDB should avoid cloning/converting full rows internally when a query projects one or two columns. Add internal borrowed row/value views and allocate only at the public API boundary.

Implementation outline:

- Introduce internal borrowed projection rows for executor hot paths.
- Delay `Value` cloning until constructing public `QueryResult`.
- Avoid building intermediate `Dataset` rows for simple projection, point lookup, and indexed join paths.
- Add allocation-count benchmarks or counters for point lookup and join hot paths.

### 4. Fair Index Parity For Python Benchmarks

Add fair index parity to Python benchmarks.

Some benchmark queries imply indexed workloads but the schema lacks matching indexes, for example range scans by `items.price` and aggregates by `orders.total_amount`. Add the same indexes for SQLite and DecentDB, then measure. This is fair if the workload is labeled "indexed OLTP".

Implementation outline:

- Add `idx_items_price` for range scans by `items.price`.
- Add `idx_orders_total_amount` for aggregate filters by `orders.total_amount`.
- Keep existing foreign-key and join indexes.
- Label the profile as indexed OLTP in output metadata.
- Preserve an unindexed profile only if it is explicitly useful.

### 5. Documented Tuned DecentDB Profile

Tune DecentDB config in a documented profile.

Python SQLite uses a 64 MB cache in the comparison driver; DecentDB default cache is 4 MB. A fair "tuned durable" DecentDB profile can use a larger `cache_size_mb`, and possibly opt-in `persistent_pk_index=true` once we are comfortable treating it as release-ready. Keep default-profile numbers too.

Implementation outline:

- Add profile names:
  - `decentdb_default_durable`
  - `decentdb_tuned_durable`
  - `sqlite_wal_full`
  - `sqlite_wal_normal`
- Set tuned DecentDB cache to match the benchmark's SQLite cache budget.
- Include all config values in benchmark metadata.
- Never replace default numbers with tuned numbers without labeling.

### 6. Durable Commit Profiling

Durable commit path needs profiling, not guessing.

Use `perf` plus `strace -c -e fsync,fdatasync,pwrite64` on the native durable commit benchmark. Likely fair targets are fewer allocations in WAL frame assembly, reusable commit buffers, WAL preallocation where correct, and ensuring checkpoint work is never on the single-commit hot path.

Implementation outline:

- Profile `durable_commits` in `crates/decentdb/benches/embedded_compare.rs`.
- Count sync calls and write calls.
- Verify one logical durable commit maps to the expected WAL sync behavior.
- Optimize allocation and buffer reuse before changing WAL semantics.
- Keep crash-injection and recovery tests green after every change.

### 7. Insert Throughput Work

Insert throughput likely needs row-encoding and constraint/index update work.

Prepared inserts are already used in `crates/decentdb/benches/embedded_compare.rs:680`, so the remaining gap is probably in row validation, value coercion, index maintenance, and commit-time serialization.

Implementation outline:

- Cache prepared insert coercion plans.
- Cache index update plans.
- Reuse row encoding buffers.
- Reduce temporary `Value` cloning.
- Add microbenchmarks for row encoding, index update preparation, and WAL frame assembly.

## Implementation Sequence

1. Generate current rust-baseline comparison summaries.
2. Run `cargo bench -p decentdb --bench embedded_compare`.
3. Run Python complex benchmark with DecentDB and SQLite.
4. Save all raw output under `.tmp/perf-baselines/<timestamp>/`.

1. Add fair index parity for Python indexed OLTP profile.
2. Add profile metadata for default/tuned/durable/relaxed modes.
3. Update chart generation to avoid misleading partial radar data.
4. Add documentation for how README charts should be interpreted.

### Phase 2: Recover Read Regression

1. Add row-source residency policy or skip unconditional re-deferral in hot read profiles.
2. Add counters/tests for materialization and re-deferral behavior.
3. Verify `rust-baseline` read steps move toward Apr 27-28 performance.
4. Validate memory remains bounded under a memory-constrained profile.

### Phase 3: Recover Insert Regression

1. Profile `seed_songs`.
2. Add prepared insert runtime caches for coercion, constraints, and index update plans.
3. Reuse row encoding buffers.
4. Verify `rust-baseline` seed throughput returns near historical levels.

### Phase 4: Prepared Physical Read Plans

1. Implement point lookup physical plan.
2. Implement simple indexed projection physical plan.
3. Implement simple indexed nested-loop join physical plan.
4. Move benchmark-specific fast paths into generic planner capabilities.
5. Add tests for schema invalidation and plan fallback.

### Phase 5: Durable Commit Optimization

1. Profile native durable commit path with `perf` and `strace`.
2. Remove avoidable allocations and per-commit setup.
3. Add reusable WAL commit buffers where safe.
4. Confirm crash-injection and recovery tests still pass.
