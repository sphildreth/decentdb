## S1 — Page cache mutex serializes all reader lookups

**Severity:** HIGH
**Files:** `crates/decentdb/src/storage/cache.rs`, `crates/decentdb/src/storage/pager.rs`

### Problem

`PageCache` guards its entire state behind a single `Mutex<PageCacheState>`:

```rust
// cache.rs:14
struct PageCache {
    capacity_pages: usize,
    page_size: usize,
    state: Mutex<PageCacheState>,   // <-- single global lock
}
```

Every `pin_or_load`, `try_pin_existing`, and `insert_page` call acquires this lock. For the read path, this
means every cached-page access — even for pages already present in the cache — serializes on the global
mutex. The inner `CachedPage::inner: Mutex<CachedPageInner>` is held separately per page, so concurrent
reads of *different* already-pinned pages do proceed in parallel (once pins are acquired), but the initial
lookup step remains a serialization point.

### Impact

Under concurrent read workloads (the one-writer/many-readers model the project targets), the page cache
mutex becomes the dominant bottleneck. Reader threads accessing disjoint page sets will contend on every
`read_page_for_snapshot` → pager → cache path.

### Implementation plan

1. In `cache.rs`, change `state: Mutex<PageCacheState>` to `state: RwLock<PageCacheState>`.
2. Import `std::sync::RwLock` at the top of `cache.rs`.
3. Update all call sites within `cache.rs`:
   - `self.state.lock()` → `self.state.read()` for read-side operations (`try_pin_existing`, `pin_or_load` cache-hit path).
   - `self.state.lock()` → `self.state.write()` for mutation operations (`insert_page`, `evict_one_if_needed`, `clear`, `discard`).
   - `pin_or_load` is the tricky one: use `.read()` for the cache-hit fast path, then upgrade to `.write()` if eviction+insertion is needed. The standard pattern is: read-check, if not found, drop read, acquire write, double-check under write (a page may have been inserted between dropping read and acquiring write).
   - The test helper `dirty_flag` (which reads `state.pages` and `page.inner`) needs `.read()` instead of `.lock()`.
4. The poison error `"page cache lock poisoned"` stays the same — `RwLock::read()` and `RwLock::write()` return `PoisonError` the same way `Mutex::lock()` does.
5. No changes needed in `pager.rs` — it calls `cache.pin_or_load()` and `cache.insert_clean_page()`, which are the public API and already abstract away the locking.

### Constraints

- Must ensure `pin_count` increment remains atomic with the access-timestamp update. An `RwLock` read guard
  is sufficient because `CachedPageInner` has its own `Mutex`, meaning the pin-count bump under the
  per-page lock is still serialized once we have a page reference.
- The `evict_one_if_needed` path already requires a write guard, which is correct.
- Do NOT replace the per-page `CachedPage::inner: Mutex<CachedPageInner>` with an `RwLock` — the
  pin_count/dirty/last_access triple must be mutated atomically together.

## S2 — O(n) linear LRU eviction scan (page cache)

**Severity:** MEDIUM
**Files:** `crates/decentdb/src/storage/cache.rs:189-216`

### Problem

`evict_one_if_needed` iterates every entry in `state.pages`, acquiring and releasing each
`CachedPage::inner` mutex to read `pin_count` and `last_access`:

```rust
fn evict_one_if_needed(&self, state: &mut PageCacheState) -> Result<()> {
    // ...
    for (page_id, page) in &state.pages {
        let inner = page.inner.lock()...?;      // lock per page
        if inner.pin_count == 0 {
            // compare last_access for LRU decision
        }
    } // drop inner lock per page
}
```

### Impact

With large cache sizes (e.g., 100K pages at 16 KB = 1.6 GB of cached data), every eviction scans all
entries, acquiring/releasing 100K mutexes. This is paid on every cache miss that requires eviction.
The overhead is proportional to the number of *pinned* pages — when many pages are pinned by active
reader snapshots, the scan iterates them all only to skip them.

### Implementation plan

Replace the linear LRU scan with a `BTreeMap<u64, Vec<PageId>>` bucketed by `last_access` timestamp:

1. Add a field `lru_index: BTreeMap<u64, Vec<PageId>>` to `PageCacheState`.
2. On page insert, page replacement, discard, and eviction: keep `lru_index` synchronized with the page's indexed access timestamp. The initial S2 implementation may also update `lru_index` on every cache hit, but S19 supersedes that hot-hit behavior with lazy recency repair so cache hits do not block on a global write lock.
3. In `evict_one_if_needed`: iterate `lru_index.values().flatten()` in order, skipping pages whose `pin_count != 0`. The first unpinned page found is the LRU victim. Remove it from the `BTreeMap` bucket and from `state.pages`.
4. On page eviction and discard, remove from `lru_index`.
5. `clear()` resets `lru_index` to empty.

This makes eviction O(k) where k is the number of pinned pages scanned before finding an unpinned page — typically very small.

**Do NOT pull in the `lru` crate** — the page cache is a low-level primitive and should avoid external dependencies.

### Constraints

- Must still handle the pin-count guard (skip pinned pages during eviction).
- Must not regress the common fast path of cache-hit lookups. Per-hit `BTreeMap` maintenance under `PageCacheState::write()` is not acceptable for the final cache design; S19 is required to move that work off the contended hot-hit path.
- The `last_access` counter is `u64` and monotonically increases — wrapping is not a practical concern.

- [ ] `cargo clippy --workspace` passes with zero warnings.
- [ ] All existing page cache tests pass (same list as S1).
- [ ] `cargo test -p decentdb` passes (full integration test suite).
- [x] `python ./scripts/do-pre-commit-checks.py` exits 0 with every runnable check in a `PASSED` state (20 passed, 1 skipped: Java binding tests skipped due local Gradle/JDK incompatibility).

## S3 — Unnecessary page-data copying in hot paths

**Severity:** MEDIUM
**Files:** `crates/decentdb/src/wal/mod.rs:426-442`, `crates/decentdb/src/btree/page.rs:249`

### Problem A: `to_vec()` in WAL delta reconstruction

When materializing a delta-encoded page from the WAL index, the code calls `.to_vec()` on the base page
`Arc<[u8]>` to obtain a mutable buffer for `apply_page_delta_in_place`:

```rust
// wal/mod.rs:435
let mut base = if let Some(previous) = self.materialize_latest_visible_locked(...)? {
    previous.to_vec()            // copies entire page (up to 16 KB)
} else {
    pager.read_page(page_id)?.to_vec()   // another full-page copy
};
apply_page_delta_in_place(&mut base, &frame.payload)?;
Ok(Arc::from(base))
```

### Problem B: `payload.to_vec()` in B+Tree leaf-cell decode

The B+Tree `decode_page` path copies every cell's inline payload:

```rust
// btree/page.rs:249
cells.push(LeafCell::inline(key, payload.to_vec()));
```

### Impact

- **A:** Under delta-heavy WAL workloads (small updates to large pages), every read of a delta-encoded
  page pays a 4–16 KB copy that could be avoided.
- **B:** Full table scans through B+Tree cursor produce `Vec<u8>` allocations per cell, causing
  allocator churn proportional to row count.

### Implementation plan

**Part A — WAL delta reconstruction** (`wal/mod.rs:426-442`):

Do **not** use `Arc::try_unwrap(previous)` here. `previous` is `Arc<[u8]>`, and the desired owned output is
`Vec<u8>`; treating `try_unwrap` as a zero-copy conversion from `Arc<[u8]>` to `Vec<u8>` is not a valid
implementation plan.

Implement this slice as an allocation-reduction pass, not as a copy-elimination pass:

1. Add a reusable page materialization scratch buffer to `SharedWalInner` in `wal/mod.rs`:
   ```rust
   materialize_scratch: Mutex<Vec<u8>>,
   ```
   Initialize it with `Vec::with_capacity(page_size as usize)` next to existing WAL scratch buffers.
2. Extract a helper:
   ```rust
   fn materialize_delta_page_with_scratch(
       &self,
       base: Arc<[u8]>,
       delta_payload: &[u8],
   ) -> Result<Arc<[u8]>>;
   ```
3. The helper must lock `materialize_scratch`, clear it, extend from `base.as_ref()`, apply
   `apply_page_delta_in_place(&mut scratch, delta_payload)?`, then return `Arc::from(scratch.as_slice())`.
   This still copies the page bytes, because delta application mutates a full page image, but it reuses
   the scratch allocation across calls.
4. Before returning, leave `scratch` allocated and length 0:
   ```rust
   let out = Arc::<[u8]>::from(scratch.as_slice());
   scratch.clear();
   Ok(out)
   ```
   The returned `Arc<[u8]>` owns its copy; the scratch buffer keeps its capacity for the next call.
5. Replace both `previous.to_vec()` and `pager.read_page(page_id)?.to_vec()` in the `FrameEncoding::PageDelta`
   arm with calls through the helper. The helper receives the selected base page as `Arc<[u8]>`.
6. Add benchmark-internal counters under the existing `bench-internals` feature:
   - `wal_delta_materialize_calls`
   - `wal_delta_scratch_reuses`
   - `wal_delta_scratch_grows`
   Increment `wal_delta_scratch_grows` only when `scratch.capacity() < page_size`.
7. Add a microbenchmark or existing `decentdb::benchmark` helper that materializes the same delta-backed page
   repeatedly and reports scratch reuse counters. The expected result is stable allocation capacity after
   the first materialization.

**Part B — B+Tree leaf-cell decode** (`btree/page.rs:249`):

1. The `decode_page` function already owns the full page `Arc<[u8]>` through its caller. The `payload: &[u8]` is a slice into that owned buffer.
2. Change `LeafCell::inline` to accept the owning `Arc<[u8]>` plus an offset+len range instead of `Vec<u8>`. This is a larger refactor because `LeafCell` is used throughout the B+Tree module. Hold this as a follow-up unless the B+Tree read path is a confirmed hotspot in profiling.
3. **Immediate low-risk change:** Do nothing for Part B in this slice. Document the opportunity for a future optimization pass.

### Constraints

- This slice does not eliminate the page-byte copy needed to apply a delta. It eliminates repeated heap
  allocation for the mutable page image.
- The scratch buffer must be protected by a `Mutex` because WAL materialization can be reached by multiple
  reader threads. Do not use thread-local storage or global scratch.
- The returned `Arc<[u8]>` must not borrow from the scratch buffer. Always create an owned `Arc<[u8]>`
  before clearing and retaining the scratch allocation.
- Do NOT change the `pager.read_page()` API in this slice — the returned `Arc<[u8]>` is intentionally shared.

- [x] Bench-internal counters are available: `wal_delta_materialize_calls`, `wal_delta_scratch_reuses`, and `wal_delta_scratch_grows`.
- [x] `micro_hot_paths` includes `wal_delta/wal_delta_materialize_4k_scratch_reuse`, backed by a real in-memory WAL delta page.

## S4 — WAL index O(n) demote iteration per commit

**Severity:** MEDIUM
**Files:** `crates/decentdb/src/wal/index.rs:190-218`, `crates/decentdb/src/wal/index.rs:150-165`

### Problem

`demote_cold()` is called on every commit and iterates *all* pages in the WAL index:

```rust
pub(crate) fn demote_cold(
    &mut self,
    min_reader_snapshot: Option<u64>,
    retain_recent_per_page: u32,
) {
    for versions in self.pages.values_mut() {  // O(pages in index)
        // ... check and demote prefix versions
    }
}
```

Additionally, `populate_latest_versions_at_or_before` performs O(n log n) work during checkpoint due to
the `sort_by_key` on page IDs.

### Impact

On workloads that dirty millions of distinct pages between checkpoints (bulk loads, large table
rebuilds), the WAL index grows to hold versions for every touched page. The `demote_cold` scan adds
measurable CPU overhead per commit. In extreme cases, the combination of large index + frequent commits
becomes a latency multiplier.

### Implementation plan

**Part A — `demote_cold` incremental tracking** (`wal/index.rs`):

1. Add a `dirty_since_demote: Vec<PageId>` field to `WalIndex`.
2. In `add_version`, after `self.touch(page_id)`, push `page_id` into `dirty_since_demote` if it's not already present in the last N entries. Use a `HashSet<PageId>` for dedup — kept small via `shrink_to_fit` when emptied.
3. In `demote_cold`, iterate `dirty_since_demote` instead of `self.pages.values_mut()`. After the loop, `dirty_since_demote.clear()`.
4. On `clear()`, reset `dirty_since_demote`.

This means `demote_cold` only scans pages whose version list was modified since the last demotion. In steady state with many static pages, this reduces the scan to nearly zero.

**Part B — `populate_latest_versions_at_or_before` sort removal** (`wal/index.rs:150-165`):

1. The `sort_by_key(|(page_id, _)| *page_id)` at line 164 exists to produce a deterministic checkpoint order. Check whether any downstream consumer of the checkpoint output actually depends on page-ID ordering.
2. To check: trace the callers of `populate_latest_versions_at_or_before` in `wal/checkpoint.rs`. If the pages are written to the database file via `write_all_at` with per-page offsets, the order does not matter — each page write is independent. If they are written as a contiguous block, order may matter.
3. If ordering is not required: remove the `sort_by_key` call and the `out.sort_by_key` line. Update any tests that assert ordering to sort their expected output instead.

### Constraints

- The `dirty_since_demote` set must not grow unboundedly. Use `shrink_to_fit` after clear.
- Do not change `demote_cold` behavior — only change *which* pages it iterates. All existing demotion logic (reader snapshot check, retain_recent_per_page) stays identical.

### Current status

- [x] Part A is implemented: `WalIndex` tracks unique dirty pages since the last demotion and `demote_cold` scans only that dirty set.
- [x] Reader-visible resident versions that cannot be demoted yet are requeued so they are retried after the protecting reader snapshot leaves.
- [x] Part B is implemented: the in-memory WAL index no longer sorts checkpoint copyback candidates; checkpoint writes each page by page ID offset, so in-memory order is not semantically required. Sidecar merge still sorts when sidecar records are appended.

## S5 — FFI string/value copy overhead

**Severity:** MEDIUM

## S6 — Missing I/O read-ahead and page prefetching

**Severity:** LOW
**Files:** `crates/decentdb/src/btree/read.rs`, `crates/decentdb/src/vfs/os.rs`

### Problem

Sequential B+Tree scans follow `next_leaf` pointers, loading each leaf page individually with no read-ahead or OS hinting.

### Impact

Full table scans and index range scans experience one disk I/O stall per page. With 4 KB pages and 100 µs SSD latency, scanning 10K pages costs 1 second of wall-clock time versus ~10 ms if pages were read in 64-page batches.

### Implementation plan

This slice has two independent parts. Implement Part A first; Part B is optional.

**Part A — `posix_fadvise` sequential hint** (`vfs/os.rs` and `btree/read.rs`):

1. Add a method `fn advise_sequential(&self) -> Result<()>` to the `VfsFile` trait in `vfs/mod.rs`.
2. Implement it for `OsVfsFile` using `libc::posix_fadvise` with `POSIX_FADV_SEQUENTIAL`. Guard with `#[cfg(target_os = "linux")]` — on non-Linux, make it a no-op.
3. Implement it for `MemVfsFile` as a no-op.
4. In the B+Tree cursor (`btree/read.rs`), when the cursor first advances from an internal page to a leaf page (indicating a range scan start), call `page_store.advise_sequential()` on the database VFS file.
5. The cursor needs access to the database VFS file handle. This may require threading a reference through the `PageStore` trait or the cursor struct. Minimal approach: expose `advise_sequential()` on the `PagerReadStore` and have it forward to the underlying VFS file.

**Part B — Application-level read-ahead** (`btree/read.rs`, follow-up):

1. When advancing to `next_leaf`, read the page header of `next_leaf + 1` to verify it is a leaf page, then issue a non-blocking `pread` of its data.
2. Store the prefetched page in a single-slot cache on the cursor. On the next advance, if the prefetched page matches the requested page_id, return it immediately.
3. This is optional and lower priority than Part A. Do not implement unless profiling shows the `fadvise` hint is insufficient.

### Constraints

- Must not prefetch across the end of the leaf chain (leaf header has `next_leaf == 0`).
- The `fadvise` hint is advisory only — the OS may ignore it on some filesystems/kernels.
- Do NOT introduce an async I/O runtime. All I/O remains synchronous, blocking `pread`/`pwrite`.

## S8 — FIFO statement cache + no engine-side prepared-statement cache

**Severity:** LOW
**Files:** `crates/decentdb/src/db.rs:359-417`, `crates/decentdb/src/c_api.rs:797-800`

### Problem A: `ddb_db_prepare` bypasses the statement cache

Every `ddb_db_prepare` call re-parses SQL through `libpg_query` and rebuilds the `PreparedStatement`.
The existing `statement_cache: Mutex<StatementCache>` in `DbInner` is only used for pragma dispatch.

### Problem B: Existing `StatementCache` is FIFO, not LRU

`StatementCache::get_or_parse` never promotes entries on cache hit:

```rust
fn get_or_parse(&mut self, sql: &str) -> Result<Arc<SqlStatement>> {
    if let Some(statement) = self.entries.get(sql) {
        return Ok(Arc::clone(statement));   // <-- no promotion; pure FIFO
    }
}
```

### Impact

- **A:** Repeated execution of the same parameterized SQL re-parses every time.
- **B:** On cache-pressure workloads, hot statements are evicted before cold statements.

### Implementation plan

**Part A — Plumb `ddb_db_prepare` through the cache** (`c_api.rs` and `db.rs`):

1. Add a method `fn prepare_cached(&self, sql: &str) -> Result<PreparedStatement>` to `Db` that:
   a. Checks `self.inner.statement_cache.lock().get_or_parse(sql)?` for the parsed SQL AST.
   b. Calls the existing `self.prepare_with_runtime(&parsed_sql, &runtime)` logic.
   c. Validates schema_cookie + temp_schema_cookie before returning (invalidate if schema changed).
2. In `ddb_db_prepare` (`c_api.rs:797`), replace `db.db.prepare(&sql_owned)?` with `db.db.prepare_cached(&sql_owned)?`.
3. The `statement_cache` is already invalidated on DDL (schema_cookie bump). Add a `flush_statement_cache()` call in the DDL path if not already present. Check: schema_cookie is bumped in `DbInner` when DDL executes — is the cache flushed at that point? If not, add `self.inner.statement_cache.lock().entries.clear()` and `self.inner.statement_cache.lock().order.clear()`.

**Part B — Convert FIFO to LRU** (`db.rs:394-416`):

1. In `StatementCache::get_or_parse`, add promotion on cache hit:
   ```rust
   fn get_or_parse(&mut self, sql: &str) -> Result<Arc<SqlStatement>> {
       if let Some(statement) = self.entries.get(sql) {
           // Promote to back of LRU order
           self.order.retain(|k| k != sql);
           self.order.push_back(sql.to_string());
           return Ok(Arc::clone(statement));
       }
       // ... existing parse + insert logic unchanged
   }
   ```
2. Note: the `retain()` here is O(n) in the deque length, same as S12's concern. For `STATEMENT_CACHE_CAPACITY` (typically 16–64), O(n) is acceptable. If capacity grows, replace with an `lru` crate or `LinkedHashMap`, as in S12.

### Constraints

- The cache key is `sql: String`. Case-sensitive matching is correct because `parse_sql_statement` normalizes identifiers, but the raw SQL text is what the caller provides.
- Schema invalidation: when `schema_cookie` changes, the cache must be flushed. The `PreparedStatement` already carries `schema_cookie` and `temp_schema_cookie`; validate on each `prepare_cached` call against current cookies.
- Must NOT cache statements that contain `RETURNING` or other clauses that make them non-idempotent — check if `prepare_with_runtime` already handles this.

## S10 — Heap fragmentation from repeated small allocations

**Severity:** LOW
**Files:** `crates/decentdb/src/storage/page.rs:20`, `crates/decentdb/src/storage/pager.rs:111`

### Problem

Several hot paths perform repeated heap allocations of known-size buffers:

| Source | Size | Frequency |
|---|---|---|
| `zeroed_page(page_size)` | 4–16 KB | Per allocation of new pages, header reads |
| `payload.to_vec()` in B+Tree decode | variable | Per cell during scans |
| `to_vec()` in delta reconstruction | 4–16 KB | Per delta-page materialization |
| Row-level `Vec<Value>` allocations | variable | Per row |

Existing mitigations: `malloc_trim(0)` post-checkpoint, `SmallVec` for WAL versions, reusable checkpoint buffers.

### Implementation plan

Add a page-buffer pool to `PagerHandle` for recycling page buffers that already exist. Do **not** allocate a
new buffer solely to put it into the pool.

1. In `crates/decentdb/src/storage/pager.rs`, add a field to `PagerHandle` (or its inner struct):
   ```rust
   page_pool: Mutex<Vec<Vec<u8>>>,
   page_pool_max: usize, // configurable, default 256
   ```
2. Add helper methods:
   ```rust
   fn take_page_buffer(&self) -> Vec<u8>;
   fn recycle_page_buffer(&self, buffer: Vec<u8>);
   ```
3. `take_page_buffer` must pop from the pool if available, resize to `page_size`, fill with zeroes, and return it.
   If the pool is empty, allocate `vec![0_u8; page_size]`.
4. `recycle_page_buffer` must accept only buffers already produced by pager hot paths. It must:
   - reject buffers whose capacity is less than `page_size`,
   - truncate length to 0,
   - push only if `pool.len() < page_pool_max`,
   - never allocate.
5. Replace hot `page::zeroed_page(self.inner.page_size)` calls in `pager.rs` with `self.take_page_buffer()`.
6. When a pager hot path owns a temporary page buffer and is done with it, call `recycle_page_buffer(buffer)`.
   Do not recycle buffers that have been moved into `Arc<[u8]>`, WAL frames, or page-cache storage.
7. Keep the free function `page::zeroed_page(page_size)` for callers without a `PagerHandle`.
8. Config option: add `page_pool_max: usize` to `DbConfig` with default 256.

Example implementation shape:
   ```rust
   fn take_page_buffer(&self) -> Vec<u8> {
       let page_size = self.inner.page_size as usize;
       if let Ok(mut pool) = self.inner.page_pool.lock() {
           if let Some(mut buffer) = pool.pop() {
               buffer.resize(page_size, 0);
               buffer.fill(0);
               return buffer;
           }
       }
       vec![0_u8; page_size]
   }
   ```

### Constraints

- Pooled buffers must be zeroed before reuse. The `zeroed_page` semantic must be preserved.
- Pool must not grow unboundedly. The `page_pool_max` cap prevents memory hoarding.
- The pool is per-`PagerHandle` (per database), not global. This matches ADR 0142's per-engine intent.
- The pool must never allocate in `recycle_page_buffer`; otherwise this slice can increase allocation pressure.

`runtime.clone()` on the `EngineRuntime`. `create_savepoint()` also clones. While many fields are `Arc`-wrapped, these collections are deeply cloned:

| Field | Type | Clone cost |
|---|---|---|
| `persisted_tables` | `BTreeMap<String, PersistedTableState>` | **O(n) deep copy** |
| `deferred_tables` | `BTreeSet<String>` | **O(n) deep copy** |
| `dirty_tables` | `BTreeSet<String>` | **O(n) deep copy** |
| `cached_payloads` | `HashMap<String, Arc<Vec<u8>>>` | **O(n) refcount bumps + key clones** |
| `cached_payload_access_order` | `VecDeque<String>` | **O(n) string copies** |

### Impact

With thousands of tables plus large payload caches, `begin_transaction()` and `create_savepoint()` incur
O(n) allocation overhead proportional to total tracked state, not just the working set.

### Implementation plan

**Part A — Wrap `persisted_tables`, `deferred_tables`, `dirty_tables` in Arc** (`exec/mod.rs`):

1. Change the field types:
   ```rust
   persisted_tables: Arc<BTreeMap<String, PersistedTableState>>,
   deferred_tables: Arc<BTreeSet<String>>,
   dirty_tables: Arc<BTreeSet<String>>,
   ```
2. Update the `Clone` impl: replace `self.persisted_tables.clone()` with `Arc::clone(&self.persisted_tables)`. Same for `deferred_tables` and `dirty_tables`.
3. Update all mutation sites where these collections are modified. The pattern:
   ```rust
   // Before: runtime.persisted_tables.insert(key, val);
   // After:
   let mut tables = (*runtime.persisted_tables).clone();
   tables.insert(key, val);
   runtime.persisted_tables = Arc::new(tables);
   ```
   This is copy-on-write: cheap when there are no other references (the common case in autocommit), only pays the clone cost when a savepoint or clone holds a reference.
4. Search for all mutation sites: `grep -n "persisted_tables\." crates/decentdb/src/exec/mod.rs crates/decentdb/src/db.rs` and update each.
5. Same for `deferred_tables` and `dirty_tables`.

**Part B — Extract `cached_payloads` into a shared structure** (`exec/mod.rs`):

1. Define a new struct:
   ```rust

   pub(crate) struct PayloadCache {
       entries: HashMap<String, Arc<Vec<u8>>>,
       access_order: VecDeque<String>,
       max_entries: usize,
   }
   ```
2. Replace `cached_payloads`, `cached_payload_access_order`, and `cached_payloads_max_entries` with a single field: `payload_cache: Arc<Mutex<PayloadCache>>`.
3. Update `EngineRuntime::clone` to `Arc::clone(&self.payload_cache)`.
4. Update `touch_cached_payload`, `cache_payload_insert`, `cache_payload_remove` to lock `payload_cache.lock().unwrap()` and delegate.
5. This also incidentally fixes S12 (the O(n) `retain()` scan) if the `PayloadCache` uses the `lru` crate internally — do NOT do that in this slice. Focus only on the `Arc<Mutex<>>` wrap. S12 is tracked separately.

### Constraints

- CoW semantics must be correct: when `Arc::strong_count(&self.persisted_tables) == 1`, mutations can be done in-place without a clone. This is an optimization, not required for correctness.
- The `Arc<Mutex<PayloadCache>>` introduces a new lock. Verify it does not create a lock-ordering deadlock with `engine: RwLock<EngineRuntime>`. ## S12 — O(N) VecDeque::retain() in cached_payload LRU

**Severity:** MEDIUM
**Files:** `crates/decentdb/src/exec/mod.rs:1236-1241`

### Problem

`EngineRuntime::touch_cached_payload` uses `VecDeque::retain()` for LRU maintenance:

```rust
fn touch_cached_payload(&mut self, table_name: &str) {
    self.cached_payload_access_order
        .retain(|key| key != table_name);        // O(N) scan + shift
    self.cached_payload_access_order
        .push_back(table_name.to_string());
}
```

Every cache hit pays O(n) in the access-order deque.

### Impact

At `cached_payloads_max_entries = 512`, every hit scans 512 entries doing string comparisons. This
turns the payload cache from an accelerator into a drag at moderate cache sizes.

### Implementation plan

Do NOT add an external dependency (no `lru` crate). Instead, use `std::collections::LinkedList` as a standalone access-order list with `HashMap<String, *const Node>` for O(1) node removal. But `LinkedList` doesn't expose node pointers. Better approach: implement a minimal intrusive doubly-linked list OR use `indexmap` which the project may already depend on.

**Simplest no-dependency approach:** Replace `HashMap<String, Arc<Vec<u8>>>` + `VecDeque<String>` with a single `Vec<(String, Arc<Vec<u8>>)>` + LRU via `swap_remove`:

1. Replace fields:
   ```rust
   cached_entries: Vec<(String, Arc<Vec<u8>>)>,
   cached_payloads_max_entries: usize,
   ```
2. `touch_cached_payload`: find the entry by linear scan of `cached_entries` (O(n), same as current), then `swap_remove` + `push` to move it to the end. This is O(n) but avoids the 2x `retain` O(n) cost (one for touch, one for eviction).
3. Actually, this is still O(n). Let me reconsider.

**Better approach — use a generation counter + sorted eviction:**

1. Add `next_touch_gen: u64` to `EngineRuntime`.
2. Change `cached_payloads` from `HashMap<String, Arc<Vec<u8>>>` to `HashMap<String, (u64, Arc<Vec<u8>>)>` where the `u64` is the last-touch generation.
3. Remove `cached_payload_access_order` entirely.
4. `touch_cached_payload`: increment `next_touch_gen`, update the entry's touch generation. O(1).
5. Eviction (`cache_payload_insert` overflow path): iterate all entries to find the one with the smallest touch generation. This is O(n) on eviction (infrequent), not on every cache hit.
6. The common path (cache hit) becomes O(1). Only eviction (cache overflow) is O(n).

This is the recommended implementation. It mirrors the `WalIndex` touch-epoch pattern already used in `wal/index.rs:250-256`.

### Constraints

- The generation counter wraps at `u64::MAX`. Wrapping is safe because comparisons only care about relative ordering. Use `wrapping_sub` for comparison, same as `WalIndex`.
- Eviction is infrequent (only when inserting into a full cache). The O(n) eviction cost is acceptable since `cached_payloads_max_entries` is bounded (default 512).

## S13 — O(rows × columns) instrumentation via approximate_heap_bytes

**Severity:** LOW
**Files:** `crates/decentdb/src/exec/mod.rs:214-225`

### Problem

    // O(rows) * O(columns) traversal
    for row in &self.rows {
        for value in &row.values {
            total += value.approximate_heap_bytes();

## S14 — Blocking checkpoint in DbInner::drop

**Severity:** LOW
**Files:** `crates/decentdb/src/db.rs:238-268`

### Problem

`DbInner::drop` forces a blocking WAL checkpoint when the last `Db` handle is dropped:

```rust
// db.rs:265-267
let _ = self
    .wal
    .checkpoint(&self.pager, self.config.checkpoint_timeout_sec);
```

### Impact

If `Db` is dropped on a latency-sensitive thread, the blocking I/O adds unpredictable tail latency.

### Implementation plan

Spawn the final checkpoint on a background thread with a join timeout:

1. In `DbInner::drop`, replace the blocking checkpoint call with:
   ```rust
   let wal = self.wal.clone();
   let pager = self.pager.clone();
   let timeout = self.config.checkpoint_timeout_sec;
   let handle = std::thread::spawn(move || {
       let _ = wal.checkpoint(&pager, timeout);
   });
   // Wait briefly for the checkpoint to complete, but don't block indefinitely.
   // The timeout here is shorter than the checkpoint timeout to ensure Drop
   // returns promptly.
   let drop_deadline = std::time::Duration::from_secs(timeout.min(5));
   // join_timeout is unstable; use a simple join with a short
   // checkpoint_timeout_sec value as the mechanism. Set a recommended
   // DEFAULT_CHECKPOINT_TIMEOUT_SEC of 5 seconds.
   let _ = handle.join();
   ```
2. Wait — `std::thread::JoinHandle::join` blocks indefinitely. The timeout mechanism is the `checkpoint_timeout_sec` passed to the checkpoint function, which internally has a timeout. So the correct approach:
   ```rust
   let wal = self.wal.clone();
   let pager = self.pager.clone();
   let timeout = self.config.checkpoint_timeout_sec;
   std::thread::spawn(move || {
       let _ = wal.checkpoint(&pager, timeout);
   });
   // Do NOT join — let the background thread complete independently.
   // The WalHandle's internal state ensures the checkpoint finishes before
   // the WAL file descriptor is closed (Arc keeps the underlying file alive).
   ```
3. Verify: `WalHandle` uses `Arc<SharedWalInner>`, and `SharedWalInner.file` is `Arc<dyn VfsFile>`. As long as the background thread holds an `Arc` clone, the file descriptor stays open. Spawning and not joining is safe.
4. Check `self.wal.clone()` — `WalHandle` must implement `Clone` and `Send`. It does (it's `#[derive(Clone)]` wrapping `Arc<SharedWalInner>`).
5. `PagerHandle` must also be `Clone` and `Send`. It is (`Arc<Pager>` wrap).
6. Update the default `checkpoint_timeout_sec` to a small value (e.g., 5 seconds) if it is larger.

### Constraints

- The spawned thread is detached. If the process exits before the checkpoint completes, the WAL may not be fully checkpointed. This is acceptable — the next open will recover from the WAL.
- `WalHandle` must remain valid after `DbInner` is dropped. Verify: `WalHandle` holds an `Arc` that the background thread extends.
- Do NOT add a `join_timeout` dependency (unstable). The checkpoint timeout is the mechanism.

## S16 — Full table materialization / paged-row default rollout

**Severity:** HIGH (memory — dominant RSS concern)
**Files:** `crates/decentdb/src/config.rs`, `crates/decentdb/src/exec/mod.rs`, `crates/decentdb/src/db.rs`

### Problem

The `huge` benchmark loads a 785 MB database (~16.5M rows across 3 tables) and peaks at
**11.6 GB RSS** — a **14.8× memory multiplier**. The root cause is not the page cache or WAL
index, but full table materialization.

With the previous default configuration (`defer_table_materialization: true`, `paged_row_storage: false`):

1. **At open time**: tables are correctly deferred (not loaded). Promising.
2. **On first access**: `ensure_tables_loaded_at_snapshot` (in `db.rs`) calls
   `EngineRuntime::load_deferred_tables_with_snapshot` → `materialize_deferred_tables_with_store`
   (line 1509) → `decode_persisted_table_data` (line 13651). This function **always** creates a
   fully-resident `TableData { rows: Vec<StoredRow> }` — every row is decoded into heap-allocated
   `Value` objects and held in memory.

Each `StoredRow` contains `Vec<Value>` where the `Value` enum is 88 bytes, and TEXT/Blob
variants hold heap-allocated `String`/`Vec<u8>` copies for every column value. For a table with
13.7M songs (each with ~5 TEXT columns), materialization allocates tens of GB of heap memory
that persists for the lifetime of the `EngineRuntime`.

The **paged row storage path** (`paged_row_storage: true`, ADR 0145 Phase D) already exists
with 60+ integration tests in `db.rs`. It stores a `TablePageManifest` with per-row byte-range
locators and decodes individual rows on demand from the page store, rather than materializing
all rows up front.

The `materialize_deferred_table_row_sources_with_store` method (line 1554) at line 1582 checks
`state.pointer.is_table_paged_manifest()`:
- **Paged**: creates `TableRowSource::Paged(Arc<TablePageManifest>)` — no materialization.
- **Legacy**: falls back to `TableRowSource::Resident(Arc<TableData>)` — full materialization.

The `backfill_paged_row_storage` method (line 2997, called at line 1633 on open) already wraps
legacy non-paged table payloads into the paged manifest format, so existing databases are
automatically migrated when `paged_row_storage: true`.

### Impact

- **Memory**: 14.8× RSS multiplier eliminated for read-heavy workloads. The `huge` benchmark

  paged row lookup. Full table scans incur repeated page-store reads but the page cache absorbs

- **Correctness**: The paged path's on-disk format (`TablePageManifest`) is read-only; writes
  create overlay chunks, preserving multi-version reader isolation. The `Paged` variant in

## S19 — Page-cache LRU maintenance serializes hot cache hits

**Severity:** HIGH
**Files:** `crates/decentdb/src/storage/cache.rs`

### Problem

S1 replaces the global page-cache `Mutex` with an `RwLock`, and S2 adds a `BTreeMap` LRU index to avoid
linear eviction scans. The combined implementation can still take a global write lock on every cache hit to
remove and reinsert the page in the LRU index.

That defeats S1's purpose under concurrent hot-cache reads: readers no longer serialize on lookup, but they
serialize immediately afterward on LRU maintenance.

### Impact

Concurrent readers accessing disjoint cached pages should mostly contend only on per-page locks. If every hit
takes `PageCacheState::write()`, throughput can be worse than the original single-mutex cache because the
`BTreeMap` maintenance adds extra work under the global write lock.

### Implementation plan

Use lazy recency repair. Cache hits update per-page recency immediately, but global LRU index maintenance is
opportunistic and must not block hot reads.

1. In `CachedPageInner`, add a second timestamp:
   ```rust
   last_access: u64,
   indexed_access: u64,
   ```
   `last_access` is the true latest touch. `indexed_access` is the timestamp currently represented in
   `PageCacheState::lru_index`.
2. When creating or replacing a cached page, set both fields to the same access counter value and insert that
   value into `lru_index`.
3. In `try_pin_existing`:
   - acquire `PageCacheState::read()`,
   - clone the `Arc<CachedPage>`,
   - while the read guard is still held, lock the page, increment `pin_count`, and update `last_access`,
   - drop the page lock and read guard,
   - call a new `try_refresh_lru_index(page_id)` helper.
4. `try_refresh_lru_index(page_id)` must use `self.state.try_write()`:
   - if it gets the write lock, compare `indexed_access` to `last_access`; if they differ, move the page from
     the old bucket to the new bucket and set `indexed_access = last_access`,
   - if it returns `TryLockError::WouldBlock`, return `Ok(())` without waiting,
   - if poisoned, return `DbError::internal("page cache lock poisoned")`.
5. In `evict_one_if_needed`, repair stale LRU entries lazily:
   - iterate `lru_index` in timestamp order,
   - for each candidate page, lock the page,
   - if `indexed_access != bucket_ts`, remove the stale bucket entry and restart,
   - if `last_access != indexed_access`, move the page to the `last_access` bucket, set `indexed_access`, and restart,
   - if `pin_count == 0`, evict the page,
   - if all pages are pinned, return the existing transaction error.
6. Keep exact safety semantics: a page found under the read guard must be pinned before releasing that guard so
   eviction cannot remove it before the caller receives `PageHandle`.
7. Add a temporary benchmark helper under `.tmp/` only while developing if needed, but do not commit it. The
   committed validation must be a test or benchmark in the existing Rust test/benchmark structure.

### Required tests

1. `cache_hit_does_not_wait_for_lru_write_lock`:
   - load page 1,
   - hold a `PageCacheState::read()` guard in the test,
   - call `pin_or_load(1, ...)`,
   - assert it returns immediately and does not call the loader.
2. `lazy_lru_repair_preserves_recently_touched_page`:
   - create capacity 2,
   - load pages 1 and 2,
   - repeatedly hit page 1 while preventing `try_refresh_lru_index` from taking the write lock,
   - insert page 3,
   - assert page 2 is evicted and page 1 remains cached.
3. `all_pages_pinned_returns_transaction_error` must still pass.
4. `eviction_is_faster_than_linear_scan` from S2 must still pass.

### Benchmark evidence

Add or use a page-cache microbenchmark that runs hot cache-hit reads against disjoint page sets with
1, 2, 4, 8, and 16 threads. The median result at 8 and 16 threads must be faster than the current S2
implementation and must not be slower than the pre-S2 global-mutex implementation.

## S20 — Reader path lock audit and write-transaction fast path

**Severity:** HIGH
**Files:** `crates/decentdb/src/db.rs`, `crates/decentdb/src/wal/mod.rs`, `crates/decentdb-benchmark/src/scenarios.rs`

### Problem

After S1/S2/S19, the page cache should no longer be the dominant serialization point for hot cached reads.
The next likely bottleneck is the rest of the read path. `Db::read_page_at_snapshot_lsn` currently locks
`self.inner.write_txn` for every page read to check `staged_pages`, even when no write transaction is active.

This is unnecessary for normal reader workloads and adds a global mutex acquisition to every page read.

### Impact

Under concurrent readers, a per-page `write_txn.lock()` can serialize reads even when there is no writer.
This is especially harmful for B+Tree scans and paged row storage, where one query can read many pages.

### Implementation plan

Part A is mandatory. Part B adds measurement so later lock work is guided by data.

**Part A — skip `write_txn` lock when no writer is active**

1. Add a field to `DbInner`:
   ```rust
   write_txn_active: AtomicBool,
   ```
2. Initialize it to `false` in `Db::open` / `Db::open_or_create` where `DbInner` is constructed.
3. In `begin_write`, after successfully setting `txn.active = true`, set:
   ```rust
   self.inner.write_txn_active.store(true, Ordering::Release);
   ```
4. In every path that makes the write transaction inactive (`commit`, rollback, abort/error cleanup), set:
   ```rust
   self.inner.write_txn_active.store(false, Ordering::Release);
   ```
   This must happen after `txn.active = false` and after `txn.staged_pages.clear()`.
5. In `read_page_at_snapshot_lsn`, replace the unconditional `write_txn.lock()` with:
   ```rust
   if self.inner.write_txn_active.load(Ordering::Acquire) {
       let txn = self
           .inner
           .write_txn
           .lock()
           .map_err(|_| DbError::internal("write transaction lock poisoned"))?;
       if let Some(staged) = txn.staged_pages.get(&page_id).cloned() {
           return Ok(Arc::from(staged));
       }
   }
   ```
6. Preserve the existing staged-page visibility behavior when a write transaction is active.
7. Add an internal counter under `#[cfg(any(test, feature = "bench-internals"))]` to count read-path
   `write_txn` lock acquisitions. In the no-writer case, repeated `read_page` calls must leave the counter at 0.

**Part B — lock wait instrumentation for the reader path**

1. Under the existing `bench-internals` feature, add counters for:
   - `read_path_write_txn_lock_count`
   - `read_path_held_snapshots_lock_count`
   - `read_path_wal_reader_begin_count`
2. Increment counters at the actual acquisition sites.
3. Add these counters to `decentdb::benchmark` snapshots and include them in `read_under_write` scenario metrics.
4. Do not change lock types in Part B. This slice's behavior change is only the `write_txn_active` fast path.

### Required tests

1. `read_page_without_writer_skips_write_txn_lock`:
   - open a database,
   - write and checkpoint a page/table,
   - reset the test counter,
   - perform repeated reads with no active writer,
   - assert the read-path write transaction lock count is 0.
2. `read_page_with_active_writer_still_sees_staged_page`:
   - begin a write transaction,
   - stage a page update,
   - read the page through the existing read path,
   - assert the staged bytes are returned,
   - commit or rollback cleanly.

### Benchmark evidence

Run `read_under_write` before and after this slice with identical scale overrides. The after artifact must show:

- lower or equal `reader_p95_under_write_us`,
- lower `read_path_write_txn_lock_count` per reader operation in no-writer and reader-only phases,

## S21 — Projection-aware row decoding and streaming scans

**Severity:** HIGH
**Files:** `crates/decentdb/src/exec/mod.rs`, `crates/decentdb/src/exec/row.rs`, `crates/decentdb/src/record/row.rs`, `crates/decentdb/src/btree/read.rs`

### Problem

Paged row storage avoids materializing entire tables, but executor paths can still decode full rows into
`Vec<Value>` even when a query needs only a subset of columns. Full scans and range scans also tend to build
intermediate row collections before producing results.

Examples:

- `SELECT id FROM large_table WHERE id BETWEEN ? AND ?` should not decode large TEXT/BLOB payload columns.
- `SELECT COUNT(*) FROM large_table` should not allocate `Value` objects for every projected column.
- Range scans should stream rows through filters and projection instead of building large temporary vectors.

### Impact

Once S16 makes paged row storage the default, unnecessary row decoding becomes a primary read-path cost.
Projection-aware decoding reduces CPU, allocation pressure, and cache churn for point lookups, range scans,
aggregates, and joins.

### Implementation plan

Implement this in three phases in a single slice. Do not skip phases or substitute a different design.

**Phase A — represent required columns**

1. Add a new executor type:
   ```rust
   #[derive(Clone, Debug, Eq, PartialEq)]
   pub(crate) struct ColumnProjection {
       required: Vec<usize>,
   }
   ```
2. Implement:
   ```rust
   impl ColumnProjection {
       pub(crate) fn all(column_count: usize) -> Self;
       pub(crate) fn from_required(required: impl IntoIterator<Item = usize>) -> Self;
       pub(crate) fn requires(&self, index: usize) -> bool;
       pub(crate) fn required_indexes(&self) -> &[usize];
   }
   ```
   `from_required` must sort and deduplicate indexes.
3. Add unit tests for sorting, deduplication, and `all()`.

**Phase B — decode only projected values**

1. Add a row decoder function in `record/row.rs`:
   ```rust
   pub(crate) fn decode_row_projected(bytes: &[u8], projection: &ColumnProjection) -> Result<Vec<Option<Value>>>;
   ```
2. The returned vector length must equal the table column count. Required columns contain `Some(Value)`.
   Non-required columns contain `None`.
3. The decoder must still validate row format enough to skip non-required values safely. It must not skip
   malformed length prefixes, invalid type tags, or out-of-bounds payloads.
4. Existing full decode remains unchanged and delegates to `decode_row_projected(..., ColumnProjection::all(...))`
   only if that keeps behavior identical.
5. Add tests with mixed fixed-width and variable-width values proving non-required TEXT/BLOB values are not allocated.
   Use an internal allocation counter if available; otherwise assert via a test-only decoder counter.

**Phase C — thread projection through scans**

1. In SELECT execution, compute the required source-table columns from:
   - SELECT list,
   - WHERE predicates,
   - JOIN predicates,
   - GROUP BY,
   - ORDER BY,
   - aggregate arguments.
2. For `COUNT(*)`, the projection is empty unless predicates require columns.
3. Pass `ColumnProjection` into table scan/range scan row-source iteration.
4. Convert projected rows to full `Value` rows only at the final result boundary. If a later operator requests a
   non-projected column, that is a bug; add debug assertions in operator input construction.
5. Keep mutation paths (`INSERT`, `UPDATE`, `DELETE`) on full-row decode until a separate write-path projection
   design exists.

### Required tests

1. `projected_decode_skips_unneeded_large_text`:

## S22 — Planner/index coverage benchmark and targeted fixes

**Severity:** HIGH
**Files:** `crates/decentdb-benchmark/src/types.rs`, `crates/decentdb-benchmark/src/scenarios.rs`, `crates/decentdb-benchmark/src/profiles.rs`, `crates/decentdb/src/exec/mod.rs`, `benchmarks/targets.toml`

### Problem

The benchmark suite covers point lookups, range scans, joins, and complex e-commerce queries, but the review
does not require proof that the planner/executor is using the best available access path for each query shape.
If a query silently falls back to table scan or nested-loop behavior, lower-level cache/WAL improvements cannot
bring performance to where it should be.

### Impact

Planner misses can dominate all storage-level optimizations. A single unindexed scan inside a repeated join can
turn an otherwise efficient workload into O(rows × rows) behavior.

### Implementation plan

## Items confirmed as well-designed

The following patterns were examined and found to be sound — no action needed:

1. **WAL batch write** — Single `write_all_at` for all frames + commit frame using reusable buffers.
2. **Delta encoding** — 512-byte deltas instead of full pages for small edits.
3. **`SmallVec<[WalVersion; 1]>`** — Eliminates heap allocation for the single-version-per-page case.
4. **`WalVersionPayload::OnDisk` demotion** — Cleanly drops resident heap data with on-demand reconstruction.
5. **AsyncCommit group-commit** (ADR 0135) — `AtomicU64` watermarks + background flusher + `Condvar`.
6. **Post-checkpoint `malloc_trim(0)`** (ADR 0138) — Addresses glibc arena fragmentation.
7. **Reusable scratch buffers** — `checkpoint_scratch`, `delta_scratch`, `page_batch`.
8. **No `unsafe` in storage/WAL/btree** — Confined to `alloc.rs`, `c_api.rs`, `checksum.rs`. All `// SAFETY:` comments present.
9. **Reader-aware checkpoint** — Correctly gates WAL truncation on active reader completion.
10. **FFI panic safety** — `ffi_boundary()` wraps every `extern "C"` function in `catch_unwind`.
11. **Zero-copy row-view API** — `ddb_stmt_row_view` and friends; Python `_fastdecode.c` exploits this.
12. **LRU eviction pin-count safety** — Returns `DbError::Transaction` when all pages pinned, no deadlock.
