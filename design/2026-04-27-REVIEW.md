## S1 — Page cache mutex serializes all reader lookups

**Severity:** HIGH  
**Files:** `crates/decentdb/src/storage/cache.rs:14`, `crates/decentdb/src/storage/pager.rs`

### Problem

`PageCache` guards its entire state behind a single `Mutex<PageCacheState>`:

```rust
// cache.rs:14
struct PageCache {
    capacity_pages: usize,
    page_size: usize,
    state: Mutex<PageCacheState>,   // <-- single global lock
}
```

Every `pin_or_load`, `try_pin_existing`, and `insert_page` call acquires this lock. For the read path, this
means every cached-page access — even for pages already present in the cache — serializes on the global
mutex. The inner `CachedPage::inner: Mutex<CachedPageInner>` is held separately per page, so concurrent
reads of *different* already-pinned pages do proceed in parallel (once pins are acquired), but the initial
lookup step remains a serialization point.

### Impact

Under concurrent read workloads (the one-writer/many-readers model the project targets), the page cache
mutex becomes the dominant bottleneck. Reader threads accessing disjoint page sets will contend on every
`read_page_for_snapshot` → pager → cache path.

### Recommended action

Replace `Mutex<PageCacheState>` with `RwLock<PageCacheState>`. The majority of cache operations are
read-side lookups (`try_pin_existing`, `pin_or_load` with a cache hit). Only eviction and insertion need
write access. This allows concurrent cache-hit pinning across reader threads.

A subsequent optimization would be a lock-free or sharded concurrent hash map (e.g., `dashmap`), but
`RwLock` is the lowest-risk first step.

### Constraints

- Must ensure `pin_count` increment remains atomic with the access-timestamp update. An `RwLock` read guard
  is sufficient because `CachedPageInner` has its own `Mutex`, meaning the pin-count bump under the
  per-page lock is still serialized once we have a page reference.
- The `evict_one_if_needed` path already requires a write guard, which is correct.

## S2 — O(n) linear LRU eviction scan

**Severity:** MEDIUM  
**Files:** `crates/decentdb/src/storage/cache.rs:189-216`

### Problem

`evict_one_if_needed` iterates every entry in `state.pages`, acquiring and releasing each
`CachedPage::inner` mutex to read `pin_count` and `last_access`:

```rust
fn evict_one_if_needed(&self, state: &mut PageCacheState) -> Result<()> {
    // ...
    for (page_id, page) in &state.pages {
        let inner = page.inner.lock()...?;      // lock per page
        if inner.pin_count == 0 {
            // compare last_access for LRU decision
        }
    } // drop inner lock per page
}
```

### Impact

With large cache sizes (e.g., 100K pages at 16 KB = 1.6 GB of cached data), every eviction scans all
entries, acquiring/releasing 100K mutexes. This is paid on every cache miss that requires eviction.
The overhead is proportional to the number of *pinned* pages — when many pages are pinned by active
reader snapshots, the scan iterates them all only to skip them.

### Recommended action

Replace the flat `HashMap` + linear LRU scan with a proper LRU data structure. A `LinkedHashMap` (or a
custom intrusive doubly-linked list keyed by `PageId`) would make eviction O(1). Alternatively, a
`BTreeMap<u64, PageId>` sorted by `last_access` with a separate `HashMap<PageId, ...>` for O(log n)
eviction.

### Constraints

- Must still handle the pin-count guard (skip pinned pages during eviction).
- Must not regress the common fast path of cache-hit lookups.

## S3 — Unnecessary page-data copying in hot paths

**Severity:** MEDIUM  
**Files:** `crates/decentdb/src/wal/mod.rs:435`, `crates/decentdb/src/btree/page.rs:249`

### Problem A: `to_vec()` in WAL delta reconstruction

When materializing a delta-encoded page from the WAL index, the code calls `.to_vec()` on the base page
`Arc<[u8]>` to obtain a mutable buffer for `apply_page_delta_in_place`:

```rust
// wal/mod.rs:435
let mut base = if let Some(previous) = self.materialize_latest_visible_locked(...)? {
    previous.to_vec()            // copies entire page (up to 16 KB)
} else {
    pager.read_page(page_id)?.to_vec()   // another full-page copy
};
apply_page_delta_in_place(&mut base, &frame.payload)?;
Ok(Arc::from(base))
```

Every delta-encoded read incurs a full-page heap copy.

### Problem B: `payload.to_vec()` in B+Tree leaf-cell decode

The B+Tree `decode_page` path copies every cell's inline payload:

```rust
// btree/page.rs:249
cells.push(LeafCell::inline(key, payload.to_vec()));
```

For a leaf page with many small rows, this creates one allocation per value.

### Impact

- **A:** Under delta-heavy WAL workloads (small updates to large pages), every read of a delta-encoded
  page pays a 4–16 KB copy that could be avoided.
- **B:** Full table scans through B+Tree cursor produce `Vec<u8>` allocations per cell, causing
  allocator churn proportional to row count.

### Recommended action

**For A:** Use `Arc::try_unwrap(previous)` when the refcount is 1 to get the owned `Vec<u8>` without a copy.
Fall back to `to_vec()` only when the data is shared with another reader. Alternatively, maintain a
reusable scratch buffer per reader context, memcpy the base page into it, apply the delta, then return
a new `Arc<[u8]>` from the scratch buffer (see S10 for scratch-buffer patterns).

**For B:** Consider whether `LeafCell` can store `Arc<[u8]>` (or `Bytes`) instead of `Vec<u8>`, so that
the decode path can slice into the already-owned page buffer rather than copying. The `decode_page`
function receives `payload: &[u8]` that lives as long as the owning `Arc<[u8]>` from the pager; if
callers hold that `Arc`, cells can reference byte ranges within it.

## S4 — WAL index O(n) demote iteration per commit

**Severity:** MEDIUM  
**Files:** `crates/decentdb/src/wal/index.rs:190-218`

### Problem

`demote_cold()` is called on every commit and iterates *all* pages in the WAL index:

```rust
pub(crate) fn demote_cold(
    &mut self,
    min_reader_snapshot: Option<u64>,
    retain_recent_per_page: u32,
) {
    for versions in self.pages.values_mut() {  // O(pages in index)
        // ... check and demote prefix versions
    }
}
```

Additionally, `populate_latest_versions_at_or_before` performs O(n log n) work during checkpoint due to
the `sort_by_key` on page IDs.

### Impact

On workloads that dirty millions of distinct pages between checkpoints (bulk loads, large table
rebuilds), the WAL index grows to hold versions for every touched page. The `demote_cold` scan adds
measurable CPU overhead per commit. In extreme cases, the combination of large index + frequent commits
becomes a latency multiplier.

### Recommended action

**demote_cold:** Track a dirty-generation counter or a list of pages touched since last demotion.
Only iterate pages whose version lists potentially grew since the prior demotion. Most pages' version
lists do not change between demote calls.

**populate_latest_versions_at_or_before:** The page-ID sort is inherently O(n log n), but if the
checkpoint path can avoid sorting (e.g., by iterating in `HashMap` order and letting the pager write
pages in arbitrary order), the sort cost disappears. Check whether the page-ID ordering is required by
the checkpoint consumer or is an artifact of the original implementation.

## S5 — FFI string/value copy overhead

**Severity:** MEDIUM  
**Files:** `crates/decentdb/src/c_api.rs:245-250`, `c_api.rs:289`, `c_api.rs:349`

### Problem A: Every inbound string is copied

`utf8_arg()` always calls `.to_string()`:

```rust
fn utf8_arg(ptr: *const c_char, name: &str) -> Result<String> {
    c_string_arg(ptr, name)?
        .to_str()
        .map(|value| value.to_string())  // allocates every time
        ...
}
```

SQL strings (short-lived), bind parameter values, and metadata names all incur a copy even when the
Rust side immediately consumes them.

### Problem B: Every TEXT bind copies

```rust
// c_api.rs:289
Ok(Value::Text(text.to_string()))
```

Large text values bound as parameters are always heap-copied, even for insert paths where the data
could theoretically be written to the WAL or B+Tree without an intermediate copy.

### Problem C: Every TEXT/Blob copy-out allocates

```rust
// c_api.rs:349
out.data = owned_bytes(inner.clone());
```

The `owned_bytes` → `Box<[u8]>` → raw pointer path allocates a fresh buffer for every TEXT and Blob
column in every result row.

### Impact

For high-throughput OLTP workloads with many small queries, the per-call string allocations add up.
For large TEXT/Blob values, the copy cost is proportional to value size. The zero-copy `ddb_*row_view`
API (S5-mitigation) already exists for read-heavy workloads, but write-heavy and general-purpose paths
still pay the copy tax.

### Recommended action

**A (inbound strings):** For `ddb_db_execute`, accept the copy (SQL is short-lived and must be owned by
the prepared statement). For `ddb_db_prepare`, the `StmtHandle` already stores `sql: String`, so the
copy is necessary. The remaining short-lived string copies (table names, etc.) are low-impact.

**B (TEXT binds):** Accept the copy. Engine semantics require owned `Value` types for WAL durability.
The copy cost is inherent to the copy-in ownership model.

**C (TEXT copy-out):** The zero-copy row-view API (`ddb_stmt_row_view`, `ddb_stmt_step_row_view`) already
addresses this. Ensure all bindings use the row-view API as their primary read path. Document the tradeoff
in the C header: `_value` functions are convenience/simplicity; `_row_view` functions are the performant
path.

**No code changes required** for S5 beyond potentially auditing that host-language bindings prefer the
row-view API. The current design has the right escape hatches already.

## S6 — Missing I/O read-ahead and page prefetching

**Severity:** LOW  
**Files:** `crates/decentdb/src/btree/read.rs`, `crates/decentdb/src/vfs/os.rs`

### Problem

Sequential B+Tree scans follow `next_leaf` pointers, loading each leaf page individually:

```rust
// btree/read.rs (cursor logic)
// When cells exhausted: load next_leaf page fresh, no prefetch
```

The OS VFS layer uses synchronous `pread`/`pwrite` with no I/O hints, no `fadvise`, and no application-level
read-ahead buffer.

### Impact

Full table scans and index range scans experience one disk I/O stall per page, rather than streaming
consecutive pages. With 4 KB pages and 100 µs SSD latency, scanning 10K pages costs 1 second of
wall-clock time versus ~10 ms if pages were read in 64-page batches.

### Recommended action

1. Add a simple read-ahead hint in the B+Tree cursor: when advancing to `next_leaf`, immediately issue
   an asynchronous (or pread-in-advance) read of `next_leaf + 1` page (the subsequent leaf). The cost
   is minimal (one extra pending read) and can hide one disk-latency round trip.

2. Consider `posix_fadvise(POSIX_FADV_SEQUENTIAL)` on the database file when a sequential scan is
   detected (cursor moves monotonically forward through leaf pages). This lets the OS page cache
   perform read-ahead automatically.

### Constraints

- Must not prefetch across the end of the leaf chain.
- Prefetching should be optional/configurable to avoid regressing random-access workloads.

## S7 — EngineByteBuf allocator boundary is dead code

**Severity:** LOW  
**Files:** `crates/decentdb/src/alloc.rs`

### Problem

The `EngineByteBuf` and `EngineAlloc` scaffold (ADR 0142) is gated behind `#![allow(dead_code)]`:

```rust
#![allow(dead_code)]     // Entire alloc module
```

The `EngineByteBuf` is only exercised in its own unit tests. No production hot path routes allocations
through `EngineAllocHandle`. The `WalWriteState` holds an `EngineByteBuf` via `EngineAllocHandle::default()`,
but the WAL writer currently uses the global allocator through it — zero functional difference.

### Recommended action

Either:
1. **Plumb it through.** Wire `EngineAllocHandle` into `Pager`, `WalIndex`, and `TableData` so that
   each `Db` instance can be individually metered. This is tracked as ADR 0142 follow-up work.
2. **Remove it.** If per-engine accounting is not on the near-term roadmap, remove the scaffold to
   reduce dead-code maintenance burden and the `unsafe` surface area.

Option 1 is the intended path per ADR 0142.

## S8 — FIFO statement cache + no engine-side prepared-statement cache

**Severity:** LOW  
**Files:** `crates/decentdb/src/db.rs:359-417`, `crates/decentdb/src/c_api.rs:797-800`

### Problem A: `ddb_db_prepare` bypasses the statement cache

Every `ddb_db_prepare` call re-parses SQL through `libpg_query` and rebuilds the `PreparedStatement`:

```rust
// c_api.rs:797-800
let sql_owned = sql;
let stmt = db.db.prepare(&sql_owned)?;
```

Statement caching is done entirely in host-language bindings (Python LRU slots, Go prepared statement
map, Dart `LinkedHashMap`). The engine has no statement cache for the FFI path.

`DbInner` does have a `statement_cache: Mutex<StatementCache>` field at `db.rs:233`, but it is not
consulted by `ddb_db_prepare`. It is used only for SQL pragma dispatch (`handle_pragma`).

### Problem B: Existing `StatementCache` is FIFO, not LRU

The `StatementCache::get_or_parse` method never promotes entries on cache hit:

```rust
// db.rs:394-416
fn get_or_parse(&mut self, sql: &str) -> Result<Arc<SqlStatement>> {
    if let Some(statement) = self.entries.get(sql) {
        return Ok(Arc::clone(statement));   // <-- no promotion; pure FIFO
    }
    // ... parse, then push_back on insert
}
```

If a workload bursts many unique queries, frequently-hot prepared statements can be evicted simply
because they were inserted earliest, reverting to parse-on-every-use.

### Impact

- **A:** Moderate overhead for workloads with repeated execution of the same parameterized SQL.
- **B:** On cache-pressure workloads, hot statements are evicted before cold statements that arrived
  later, worsening the parse ratio.

### Recommended action

1. Plumb `ddb_db_prepare` through `DbInner.statement_cache` with schema-cookie invalidation.
2. Promote accessed entries to the back of `order` on cache hit, converting it from FIFO to LRU:

   ```rust
   if let Some(statement) = self.entries.get(sql) {
       self.order.retain(|k| k != sql);
       self.order.push_back(sql.to_string());
       return Ok(Arc::clone(statement));
   }
   ```

## S9 — Large monolithic exec/mod.rs (28K+ lines)

**Severity:** LOW (maintenance)  
**Files:** `crates/decentdb/src/exec/mod.rs`

### Problem

`exec/mod.rs` is over 28,000 lines. It contains the `EngineRuntime` struct, table-data encoding/decoding,
row execution, operator implementations, and test helpers, all in a single file.

### Impact

- Rust compile times: changing any line in `exec/mod.rs` triggers a full rebuild of the executor
  translation unit, which is CPU-intensive due to the amount of generic and macro code.
- Code review and navigation: finding relevant logic within a 28K-line file is difficult compared to
  a well-factored module tree.

### Recommended action

Split `exec/mod.rs` into smaller modules that mirror the existing `exec/` sub-modules (dml, ddl,
operators, etc.). Move test-only helpers into `#[cfg(test)]` sub-modules or separate test files. This
is a mechanical refactor with no behavioral change, but it will improve iteration speed and
maintainability.

## S10 — Heap fragmentation from repeated small allocations

**Severity:** LOW  
**Files:** `crates/decentdb/src/storage/page.rs:20`, `crates/decentdb/src/btree/page.rs:249`,
`crates/decentdb/src/storage/pager.rs:111`

### Problem

Several hot paths perform repeated heap allocations of known-size buffers:

| Source | Size | Frequency |
|---|---|---|
| `zeroed_page(page_size)` | 4–16 KB | Per allocation of new pages, header reads |
| `payload.to_vec()` in B+Tree decode | variable (row-sized) | Per cell during scans |
| `to_vec()` in delta reconstruction | 4–16 KB | Per delta-page materialization |
| Row-level `Vec<Value>` allocations | variable | Per row during table materialization |

The engine currently mitigates this with:
- **Post-checkpoint `malloc_trim(0)`** (ADR 0138, `wal/platform.rs`) — returns freed glibc arenas to
  the OS, but does not prevent fragmentation during uptime.
- **`SmallVec<[WalVersion; 1]>`** (slice M7) — eliminates per-page heap allocation for the single-version
  common case.
- **Reusable checkpoint scratch buffer** (slice M5) — avoids allocation churn on the checkpoint path.

### Missing mitigation

No memory pooling or slab allocation for frequently-allocated sized classes. An object pool for page-sized
buffers (4K/8K/16K) would:
- Eliminate `zeroed_page` allocations for new page creation (recycle freed page buffers)
- Reduce `malloc`/`free` churn on the page cache insert path
- Reduce allocator fragmentation between WAL index tensure and page cache workloads

### Recommended action

1. **Short-term:** Add a simple `Vec<Vec<u8>>` pool in `PagerHandle` for page-sized buffers. On
   `free_page`, optionally push the freed buffer into the pool. On `allocate_page`, pop from pool
   before allocating fresh. Bound the pool size to a configurable maximum.

2. **Medium-term:** Plumb `EngineAlloc` (S7) through the page cache and WAL index so that per-engine
   allocation accounting enables embedders to set memory watermarks.

3. **Long-term:** Consider a slab allocator (e.g., `slab` crate or custom) for the row-level `Value`
   and small `Vec<u8>` allocations that dominate B+Tree decode and table materialization paths.

## S11 — EngineRuntime deep-clone on transaction/savepoint

**Severity:** MEDIUM  
**Files:** `crates/decentdb/src/db.rs:4549-4568`, `crates/decentdb/src/exec/mod.rs:1077-1110`

### Problem

`build_sql_txn_state()` (called from `begin_transaction()` and `execute_prepared_in_state()`) calls
`runtime.clone()` on the `EngineRuntime` held under the `engine: RwLock<EngineRuntime>`:

```rust
// db.rs:4552-4557
let mut runtime = self.inner.engine.read()...?.clone();
```

`create_savepoint()` similarly clones the runtime:

```rust
// db.rs:672
runtime: state.runtime.clone(),
```

While many `EngineRuntime` fields are `Arc`-wrapped (clone is a refcount bump), several collections are
deeply cloned:

| Field | Type | Clone cost |
|---|---|---|
| `catalog` | `Arc<CatalogState>` | O(1) refcount bump |
| `tables` | `Arc<BTreeMap<...>>` | O(1) refcount bump |
| `temp_table_data` | `Arc<BTreeMap<String, Arc<TableData>>>` | O(1) refcount bump |
| `indexes` | `Arc<BTreeMap<String, Arc<RuntimeIndex>>>` | O(1) refcount bump |
| `persisted_tables` | `BTreeMap<String, PersistedTableState>` | **O(n) deep copy** |
| `deferred_tables` | `BTreeSet<String>` | **O(n) deep copy** |
| `dirty_tables` | `BTreeSet<String>` | **O(n) deep copy** |
| `paged_mutations` | `BTreeMap<String, PagedMutationDelta>` | **O(n) deep copy** |
| `cached_payloads` | `HashMap<String, Arc<Vec<u8>>>` | **O(n) refcount bumps per entry** |
| `cached_payload_access_order` | `VecDeque<String>` | **O(n) string copies** |

The `cached_payloads` clone is the most expensive: a `HashMap` clone iterates all entries, cloning each
key (`String`) and bumping each `Arc<Vec<u8>>` refcount.

### Impact

With thousands of tables, large payload caches, or active deferred/dirty table sets, `begin_transaction()`
and `create_savepoint()` incur O(n) allocation overhead proportional to mutation state, not just the
working set. This is especially visible in workloads that create many savepoints within a single
transaction.

### Recommended action

- Wrap `persisted_tables`, `deferred_tables`, and `dirty_tables` in `Arc` with copy-on-write semantics,
  so cloning the runtime is an O(1) refcount bump. Mutations would clone the inner collection only
  when the refcount is > 1.
- For `cached_payloads`, consider a separate `Arc<Mutex<PayloadCache>>` that is shared across runtime
  clones rather than deep-copied on every snapshot. The payload cache is a session-level accelerator and
  does not need to be isolated per savepoint.

## S12 — O(N) VecDeque::retain() in cached_payload LRU

**Severity:** MEDIUM  
**Files:** `crates/decentdb/src/exec/mod.rs:1236-1241`

### Problem

`EngineRuntime::touch_cached_payload` uses `VecDeque::retain()` for LRU maintenance:

```rust
// exec/mod.rs:1236-1241
fn touch_cached_payload(&mut self, table_name: &str) {
    self.cached_payload_access_order
        .retain(|key| key != table_name);        // O(N) scan + shift
    self.cached_payload_access_order
        .push_back(table_name.to_string());
}
```

Every cache hit triggers a full `retain()` pass over the access-order deque. If
`cached_payloads_max_entries` is 512 (the default), every payload-cache hit pays a 512-element scan with
per-element string comparisons — and the `String` entries are compared by value, not pointer identity.

### Impact

This turns the payload cache from an accelerator into a potential drag at moderate cache sizes. The
overhead scales linearly with cache size, directly neutralizing the benefit of caching for frequent
access patterns.

### Recommended action

Replace the `HashMap<String, Arc<Vec<u8>>>` + `VecDeque<String>` pair with a true LRU structure (e.g.,
the `lru` crate, or a `LinkedHashMap`). Both provide O(1) touch operations instead of O(n).

This is a separate concern from S2 (page cache LRU) — it affects the per-session table-payload cache,
not the page cache.

## S13 — O(rows × columns) instrumentation via approximate_heap_bytes

**Severity:** LOW  
**Files:** `crates/decentdb/src/exec/mod.rs:214-225`

### Problem

`TableData::approximate_heap_bytes` walks every row and every column value:

```rust
// exec/mod.rs:214-225
pub(crate) fn approximate_heap_bytes(&self) -> usize {
    let row_struct = std::mem::size_of::<StoredRow>();
    let value_struct = std::mem::size_of::<Value>();
    let mut total = self.rows.capacity() * row_struct;
    for row in &self.rows {                          // O(rows)
        total += row.values.capacity() * value_struct;
        for value in &row.values {                   // O(columns)
            total += value.approximate_heap_bytes();
        }
    }
    total
}
```

This function is called by `table_memory_breakdown` and `table_memory_totals` (ADR 0143 instrumentation
paths). For fully resident tables with millions of rows, this blocks the calling thread for an O(rows ×
columns) traversal.

### Impact

Calling `table_memory_totals()` on a database with large resident tables can stall the caller for
seconds. This function is exposed as a public API for monitoring and introspection, making it vulnerable
to accidental misuse on large datasets.

### Recommended action

Maintain a running `heap_bytes: AtomicUsize` (or `Cell<usize>` within a `&mut self` context) on
`TableData`. Increment/decrement on INSERT, UPDATE, and DELETE mutations. The `approximate_heap_bytes`
query then becomes an O(1) read of the cached counter.

## S14 — Blocking checkpoint in DbInner::drop

**Severity:** LOW  
**Files:** `crates/decentdb/src/db.rs:238-268`

### Problem

`DbInner::drop` forces a blocking WAL checkpoint when the last `Db` handle is dropped:

```rust
// db.rs:265-267
let _ = self
    .wal
    .checkpoint(&self.pager, self.config.checkpoint_timeout_sec);
```

The `checkpoint_timeout_sec` config value controls the timeout, but the default is `DEFAULT_CHECKPOINT_TIMEOUT_SEC` (exact value TBD). If the WAL contains accumulated data (possible when `AsyncCommit` is active), the checkpoint performs blocking disk I/O inside `Drop`.

### Impact

If the `Db` handle is dropped on a latency-sensitive thread (e.g., a web server request handler,
an async executor thread, or an FFI caller's thread), the blocking I/O adds unpredictable tail latency
to what should be a trivial deallocation.

### Recommended action

1. Spawn the final checkpoint on a background thread inside `Drop`, with a `join`/timeout.
2. Alternatively, document that `Db::close()` (a dedicated shutdown method) should be called
   explicitly for a durable shutdown, while `Drop` performs a best-effort checkpoint that does
   not block beyond a configurable short timeout. The `checkpoint_timeout_sec` config already
   provides the timeout; ensure it defaults to a small value suitable for the Drop path.

## S15 — Large monolithic db.rs (18K+ lines)

**Severity:** LOW (maintenance)  
**Files:** `crates/decentdb/src/db.rs`

### Problem

`db.rs` is over 18,800 lines (~730 KB). It contains the `Db`, `DbInner`, `PreparedStatement`,
`SqlTransaction`, `SqlTxnSlot`, `WriteTxn`, `StatementCache`, `PreparedInsertCache`, and `TempSchemaState`
types together with all their method implementations — including the full SQL execution pipeline,
DDL dispatching, pragma handling, backup/restore logic, introspection metadata, and bulk load.

### Impact

- Compile time: any change to `db.rs` triggers a full rebuild of a ~730 KB translation unit.
- Lock-ordering audit difficulty: with 10+ `Mutex`/`RwLock` fields inside `DbInner` and methods
  spanning hundreds of lines, verifying that lock acquisition order is consistent across all
  code paths is error-prone.
- Navigability: locating specific logic within 18K lines is significantly harder than in
  well-factored modules.

### Recommended action

Break `db.rs` into at least these modules:
- `db/transaction.rs` — `SqlTransaction`, `SqlTxnSlot`, savepoint management
- `db/prepare.rs` — `PreparedStatement`, `StatementCache`, `PreparedInsertCache`
- `db/metadata.rs` — introspection, `HeaderInfo`, `TableInfo`, `ColumnInfo`, etc.
- `db/backup.rs` — `backup_to_path`, `restore_from_path`
- `db/bulk.rs` — `bulk_load_rows`

This is a mechanical refactor with no behavioral change. Together with S9 (splitting `exec/mod.rs`),
this represents the two largest files in the codebase.

---

## Items confirmed as well-designed

The following patterns were examined and found to be sound:

1. **WAL batch write** — Single `write_all_at` for all frames + commit frame using reusable
   `EngineByteBuf` scratch buffers. Efficient.

2. **Delta encoding** — 512-byte deltas instead of full pages for small edits. Reduces WAL I/O
   bandwidth.

3. **`SmallVec<[WalVersion; 1]>`** — Eliminates heap allocation for the common single-version-per-page
   case in the WAL index.

4. **`WalVersionPayload::OnDisk` demotion** — Cleanly drops resident heap data while preserving
   metadata for on-demand reconstruction. Configurable retention (`wal_resident_versions_per_page`).

5. **AsyncCommit group-commit** (ADR 0135) — Well-implemented with `AtomicU64` watermarks + background
   flusher thread + `Condvar` barrier.

6. **Post-checkpoint `malloc_trim(0)`** (ADR 0138) — Addresses glibc arena fragmentation on
   Linux/glibc platforms.

7. **Reusable scratch buffers** — `checkpoint_scratch`, `delta_scratch`, `page_batch` avoid allocation
   churn on hot paths.

8. **No `unsafe` in storage/WAL/btree** — Unsafe is confined to `alloc.rs` (raw pointer buffer),
   `c_api.rs` (FFI boundary), and `checksum.rs` (hardware CRC intrinsics). All `unsafe` blocks have
   `// SAFETY:` comments.

9. **Reader-aware checkpoint** — Correctly gates WAL truncation on active reader completion.

10. **FFI panic safety** — `ffi_boundary()` wraps every `extern "C"` function in `catch_unwind`, and
    double-free is explicitly safe (null-check + null-out pattern).

11. **Zero-copy row-view API** — `ddb_stmt_row_view` and friends eliminate per-row allocation for
    read-heavy FFI workloads. Bindings like Python exploit this via a C extension (`_fastdecode.c`).

12. **LRU eviction pin-count safety** — The cache refuses to evict when all pages are pinned, returning
    a `DbError::Transaction` error instead of deadlocking or crashing.
