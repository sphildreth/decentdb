# DecentDB Comprehensive Review — 2026-06-18

**Author:** Automated engineering review (Kilo)
**Scope:** General project health (code quality, architecture, hygiene) **and** bindings review (performance + feature richness vs. the engine core).
**Method:** Static inspection of the repository at `main` (commit `dffce10`, clean tree). Findings are evidence-backed with `file:line` references throughout.

## Part I — General Project Review

### 1.1 Executive Summary

DecentDB is a mature, feature-rich Rust-native embedded relational database. The engineering foundations are strong: clean `cargo clippy` (`-D warnings`), clean `cargo fmt --check`, a well-organized Cargo workspace, 2,908 tests, 201 ADRs, proptest/property tests, a dedicated WAL fuzz target, and differential testing against `rusqlite` and `duckdb`. The architecture follows the documented bottom-up layering (VFS → pager → WAL → page cache → B+Tree → planner/executor → SQL/bindings) and preserves the one-writer/many-readers snapshot model.

The concerns that surfaced are **maintainability, binding parity, and evidence hygiene**, not correctness-of-design:

1. Two source files (`exec/mod.rs`, `db.rs`) are extraordinarily large (41,120 and 36,384 lines respectively) and dominate the engine.
2. Raw panic/unwrap counts are high, but they include tests. In the two largest files, the production slice is much smaller than the raw total; the right next step is a non-test audit, not treating every count as a production durability hazard.
3. `exec/mod.rs` still has a small number of production `unreachable!()` arms where a typed `DbError::internal(...)` would be safer.
4. Binding feature parity is uneven: Dart already exposes sync changesets, while Python currently exposes watches but not sync changeset wrappers.
5. Repository hygiene artifacts at the root (`derby.log`, `*.coord`, stale tracked `plan.md`, placeholder `CHANGELOG.md`).

None of these are architectural regressions; all are addressable incrementally without restructuring the engine.

### 1.2 What Is Working Well

- **Lint/format discipline:** `cargo clippy --all-targets --all-features` is clean under `-D warnings`; `cargo fmt --check` is clean. This is non-trivial at this codebase size.
- **Workspace layout:** Five crates (`decentdb`, `decentdb-cli`, `decentdb-migrate`, `decentdb-benchmark`, `libpg_query_sys`) with the benchmark `rust-baseline` correctly excluded from the workspace. `cdylib` + `staticlib` + `rlib` lib targets are correct for the FFI/embedding story.
- **Test infrastructure:** 2,908 `#[test]` functions; property tests (`proptest`), a `wal_fuzz` binary target, differential tests vs. `rusqlite` 0.31 and `duckdb` 1.0, and `insta` snapshot tests. Criterion + iai-callgrind benchmarks with native `embedded_compare`, `release_metrics`, `micro_hot_paths`, and `plan_cache` benches.
- **ADR discipline:** 201 ADRs under `design/adr/`. This is exceptional for an open-source database and gives strong decision provenance.
- **Documented agent contract:** `AGENTS.md` is unusually precise about mission, priorities, command aliases, validation expectations, ADR-required decisions, and commit/PR hygiene. The RTK rules further reduce context cost.
- **Pre-commit staging:** `scripts/do-pre-commit-checks.py` with `fast`/`paranoid` modes and graceful skipping of missing binding toolchains.
- **Cross-platform release matrix:** Linux x86_64/arm64 (incl. Raspberry Pi OS), macOS, Windows, plus JDBC/DBeaver artifact bundles.

### 1.3 Major Findings

#### 1.3.1 File size is a serious maintainability hazard

The engine is ~158,508 lines of Rust. Two files dominate:

| File | Lines | Share |
|---|---|---|
| `crates/decentdb/src/exec/mod.rs` | 41,120 | ~26% |
| `crates/decentdb/src/db.rs` | 36,384 | ~23% |
| `crates/decentdb/src/exec/dml.rs` | 6,845 | ~4% |
| `crates/decentdb/src/c_api.rs` | 5,784 | ~4% |
| `crates/decentdb/src/sql/normalize.rs` | 4,414 | ~3% |

Together `exec/mod.rs` and `db.rs` are **~77,500 lines — roughly 49% of the engine in two files**. This has concrete costs:

- **Review and diff friction:** Any change to the executor or DB façade produces a diff against a 40k-line file, making PR review and blame navigation slow and error-prone.
- **Compile times:** Large modules defeat incremental compilation granularity; a touch to `exec/mod.rs` forces recompilation of the whole executor unit.
- **Cognitive load / bug density:** Large files correlate with ownership/lifetime bugs the borrow checker cannot catch at design time; AGENTS.md §5 explicitly says "If the compiler reports ownership or lifetime errors, fix them systematically; do not guess," which is harder to honor in a 41k-line file.

The pattern for splitting already exists: `exec/` has siblings (`dml.rs`, `ddl.rs`, `cte.rs`, `operators.rs`, `views.rs`, `triggers.rs`, `constraints.rs`, `bulk_load.rs`, `txn.rs`, `row.rs`). The `mod.rs` has simply never been carved down to a coordinator + re-exports. The same is true of `db.rs`, which mixes transaction lifecycle, query dispatch, recovery, runtime indexes, and coordination state.

**Recommendation:** Aggressively split `exec/mod.rs` by statement/expression kind (SELECT projection, JOIN execution, aggregates, window functions, subqueries, scalar functions, type coercion, expression evaluation, error mapping) and `db.rs` by concern (transaction lifecycle, query dispatch, recovery integration, runtime index cache, paged-row-locator cache, coordination/sidecar). Keep `mod.rs` as a thin re-export shim. Start by extracting the large test-helper region near the end of `exec/mod.rs` (the `panic!("expected query")` cluster at lines 35472–35892) into `*_tests.rs` modules — this shrinks the library file and prevents future raw panic/unwrap audits from confusing test assertions with production invariants.

2. Move the large `#[cfg(test)]` regions in `exec/mod.rs` and `db.rs` into focused test modules when doing the broader module split. This shrinks production files and prevents future raw-count audits from misclassifying test assertions.
3. Reserve `unreachable!` for matches where the compiler or type structure makes the arm unreachable, and document the invariant when it is retained.

## Part II — Bindings Review (Performance & Feature Richness vs. Engine Core)

### 2.1 Engine-Core ABI Foundation

The C ABI (`include/decentdb.h`, 575 lines, 97 exported `ddb_*` functions) is the single shared boundary per AGENTS.md §6 ("The Rust engine is authoritative; the C ABI is the single shared boundary"). It is well-designed for performance, exposing a layered read model and a rich set of streaming/feature APIs.

#### 2.1.1 Read path — layered fast/slow design

- **Slow path (owned, allocating):**
  - `ddb_result_value_copy(result, row, col, out_value)` — per-cell owned copy; may allocate per cell.
  - `ddb_stmt_value_copy(stmt, col, out_value)` — per-cell owned copy from a stepped statement.
  - Documented performance note (`decentdb.h:533-539`): *"for streaming read-heavy paths, prefer statement row-view APIs."*
- **Fast path (borrowed, zero-alloc):**
  - `ddb_stmt_row_view(stmt, &out_values, &out_columns)` — borrowed pointers into the current row.
  - `ddb_stmt_step_row_view(stmt, &out_values, &out_columns, &out_has_row)` — **fused step + row view**, a single FFI crossing instead of two (`step` then `row_view`).
  - `ddb_stmt_fetch_row_views(stmt, include_current, max_rows, &out_values, &out_rows, &out_columns)` — batch fetch of multiple rows in one call.
- **Typed narrow shortcuts (highest throughput for known schemas):**
  - `ddb_stmt_fetch_rows_i64_text_f64(stmt, include_current, max_rows, &out_rows_ptr, &out_rows)` — AoS `ddb_row_i64_text_f64_view_t` rows.
  - `ddb_stmt_bind_int64_step_row_view(stmt, idx, value, &out_values, &out_columns, &out_has_row)` — fused bind + step + view.
  - `ddb_stmt_bind_int64_step_i64_text_f64(...)` — fused bind + step + typed narrow row.

#### 2.1.2 Write fast paths

- `ddb_stmt_execute_batch_i64(stmt, row_count, values_i64, &out_affected)` — batch int64 inserts.
- `ddb_stmt_execute_batch_i64_text_f64(stmt, ...)` — batch mixed-type inserts (parallel arrays).
- `ddb_stmt_execute_batch_typed(stmt, row_count, signature, ...)` — signature-driven typed batch.
- Fused rebind-execute: `ddb_stmt_rebind_int64_execute`, `ddb_stmt_rebind_text_int64_execute`, `ddb_stmt_rebind_int64_text_execute` — rebind one parameter and execute in a single call (avoids bind + execute round-trip).

#### 2.1.3 Streaming / advanced feature APIs

- **Reactive subscriptions:** `ddb_db_watch_table_json`, `ddb_db_watch_range_json`, `ddb_db_watch_query_json`, `ddb_db_change_stream_json`, `ddb_watch_next_json`, `ddb_watch_close`.
- **Local-first sync:** `ddb_sync_changeset_create_json`, `..._apply_json`, `..._inspect_json`, `..._invert_json`, `ddb_db_sync_execute_json`.
- **Branches / time travel:** `ddb_db_execute_on_branch(db, branch_name, sql, params, &out_result)`, `ddb_db_branch_execute_json`.
- **Lua extensions:** `ddb_extension_validate_json`, `..._install_json`, `..._enable_json`, `..._disable_json`, `..._list_json`, `..._dependencies_json`, `..._rebuild_json`, `..._purge_json`.
- **Write queue:** `ddb_db_execute_queued(db, sql, params, timeout_ms, &out_result)`, `ddb_db_write_queue_metrics(db, &out_metrics)`.
- **Plan cache diagnostics:** `ddb_plan_cache_summary`, `ddb_plan_cache_flush`.
- **Runtime tracing:** `ddb_runtime_tracing_snapshot(db, kind, &out_json)`, `ddb_runtime_tracing_reset`.
- **Schema introspection:** `ddb_db_list_tables_json`, `ddb_db_describe_table_json`, `ddb_db_get_table_ddl`, `ddb_db_list_indexes_json`, `ddb_db_list_views_json`, `ddb_db_get_view_ddl`, `ddb_db_list_triggers_json`, `ddb_db_get_schema_snapshot_json`, `ddb_db_get_tooling_metadata_json`, `ddb_db_describe_query_json`, `ddb_db_inspect_storage_state_json`.
- **Geospatial:** `ddb_stmt_bind_geometry_wkb`, `ddb_stmt_bind_geography_wkb`, value-view `DDB_VALUE_GEOMETRY`/`DDB_VALUE_GEOGRAPHY` tags.

This is a genuinely strong surface. The variation across bindings is in **how much of it they actually use**, not in ABI limitations.

#### 2.2.1 Python — Best in class

`bindings/python/decentdb/__init__.py` (4,033 lines, 120 top-level defs) is the gold standard of the bindings:

- **Every fast-path ABI is wired:** `ddb_stmt_row_view`, `ddb_stmt_step_row_view`, `ddb_stmt_fetch_row_views`, `ddb_stmt_fetch_rows_i64_text_f64`, `ddb_stmt_bind_int64_step_row_view`, `ddb_stmt_bind_int64_step_i64_text_f64`.
- **C-accelerated fast decode:** A dedicated native extension `bindings/python/decentdb/_fastdecode.c` (compiled to `_fastdecode.cpython-314-x86_64-linux-gnu.so`) exposes specialized high-throughput decoders:
  - `bind_int64_step_row_view`
  - `bind_text_step_row_view`
  - `bind_int64_fetch_all_row_views`
  - `reset_bind_int64_fetch_all_row_views`
  - `step_fetch_all_row_views`
  - `reset_step_fetch_all_row_views`
  - `bind_text_fetch_all_row_views`
  - `bind_float64_float64_fetch_all_row_views`
  - `bind_int64_step_i64_text_f64`
  - `fetch_rows_i64_text_f64`
- **Per-SQL capability caching:** `_native_*_sql_support` dicts (`__init__.py:983-1023`) cache whether a given fast path is supported for a specific SQL string, falling back gracefully on first failure. This avoids repeated probe overhead and lets the binding discover at runtime which fast paths the engine accepts for a given plan.
- **Env-gated fallback:** `_use_row_view`/`_use_step_row_view`/`_use_fetch_row_views` flags (`__init__.py:733-743`) honor `DDB_*` env vars and `hasattr` checks, so the binding degrades safely on older engine builds.
- **Feature breadth:** Exposes watches, write queue, Lua extensions, and fulltext/trigram helpers (via SQL). It does **not** currently expose sync changeset wrappers.

The 4,033-line `__init__.py` is large, but the breadth of fast paths and the capability-caching machinery justify it. A future refactor into submodules (`statement.py`, `cursor.py`, `fast_paths.py`) would help maintainability without changing behavior.

#### 2.2.2 .NET — Excellent, deepest ecosystem integration

`bindings/dotnet/src/DecentDB.Native/NativeMethods.cs` (518 lines) imports the ABI; `DecentDB.cs` (2,293 lines) is the high-level client.

- **Fast paths wired:** `ddb_stmt_step_row_view` (`DecentDB.cs:941`), `ddb_stmt_bind_int64_step_row_view` (`DecentDB.cs:961`), `ddb_stmt_fetch_row_views` (`NativeMethods.cs:345`), `ddb_stmt_value_copy` for the general accessor (`DecentDB.cs:1594`).
- **Write fast paths:** batch execute, rebind-execute.
- **Streaming/feature APIs:** `ddb_db_sync_execute_json` (`DecentDB.cs:351`), `ddb_db_execute_queued` (`DecentDB.cs:371`), `ddb_db_write_queue_metrics` (`DecentDB.cs:405`), watches (11 files), changesets via `DecentDBSyncClient.cs`.
- **Ecosystem depth unmatched:** ADO.NET provider, Dapper-compatible, MicroOrm with LINQ (`ExpressionSqlBuilder.cs`, `LinqProvider.cs`, `FastMaterializer.cs`), EF Core provider with migrations + diagnostics + query SQL generator + window-function expression visitor, EF Core NodaTime provider, EF Core Design/Scaffolding. This is the richest binding ecosystem of the seven.
- **Write-queue options surfaced:** `DecentDBConnection.cs:573-603` maps `write_queue_enabled`/`capacity`/`default_timeout_ms`/`strict_group_commit`/`max_batch`/`max_group_delay_us` into native open options.

#### 2.2.3 Go — Good hot path, mixed convenience path

`bindings/go/decentdb-go/driver.go` (2,073 lines).

- **Hot path is correct:** `rows.Next` (`driver.go:1974-2062`) uses the fused `C.ddb_stmt_step_row_view` — a single cgo crossing — then decodes the borrowed `ddb_value_view_t` slice via `unsafe.Slice` into `dest []driver.Value`. This is the right design and the path used by every `database/sql` query.
- **Convenience/helper path is slow:** `driver.go:1344-1360` (the result-materialization helper used by `Exec`-returning-rows convenience APIs) uses a nested `for r, for c` loop calling `C.ddb_result_value_copy` + `C.ddb_value_init` + `C.ddb_value_dispose` per cell. For a result of R rows × C columns this is R×C FFI crossings and R×C allocations + disposals. This is the single most impactful Go perf finding (see §2.4.1).
- **Fast paths exposed but not on the `database/sql` path:** `BindAndStepRowView` (`driver.go:1901`), `FetchRowViews` (`driver.go:1927`), `ValueCopy` (`driver.go:1919`) are available as public methods but `database/sql`'s `Rows.Next` only uses `step_row_view`, not the batch `fetch_row_views`. A batch-prefetch `Rows` implementation could amortize cgo crossings further for bulk reads.
- **Feature exposure:** watches (`WatchQueryJson`, `driver.go:545`), write queue (`ExecQueued`, `driver.go:471`), `DescribeQueryJson`. No sync changesets, no branches, no extensions.

#### 2.2.4 Node — Good, typed + view paths at JS layer

`bindings/node/decentdb/src/addon.c` (2,677 lines) + `native_lib.c` + `index.js` (JS wrapper).

- **ABI fast paths wrapped:** `stmt_step_row_view`, `stmt_fetch_rows_i64_text_f64` (+ a `Number` variant `stmt_fetch_rows_i64_text_f64_number`), `stmt_row_view`, batch execute (`execute_batch_i64_text_f64`, `execute_batch_typed`), rebind-execute.
- **JS layer exposes them:** `index.js` exposes `stepRowView()` (`:610`), `fetchRowsI64TextF64(maxRows)` (`:592`), `fetchRowsI64TextF64Number(maxRows)` (`:599`). The Knex client (`knex-decentdb/src/client.js:136,141,215,220`) uses `fetchRowsI64TextF64Number` for bulk reads — the right fast path.
- **Graceful degradation:** `addon.c:2203` returns `DECENTDB_UNSUPPORTED` when `ddb_stmt_step_row_view` is absent, so the addon works against older engine builds.
- **Feature gaps:** No watches, no sync changesets, no branches, no extensions exposed at the JS layer (the ABI symbols aren't even loaded for these). Node is a natural fit for `EventEmitter`-style change streams; this is a missed differentiator.

#### 2.2.5 Dart — Mixed; fast paths loaded but not the default

`bindings/dart/dart/lib/src/native_bindings.dart` (1,224 lines) + `statement.dart` (1,114 lines) + `branch.dart` (824 lines).

- **All fast-path symbols are looked up:** `native_bindings.dart:1019-1044` resolves `ddb_stmt_row_view`, `ddb_stmt_step_row_view`, `ddb_stmt_fetch_row_views`, `ddb_stmt_fetch_rows_i64_text_f64`, `ddb_stmt_bind_int64_step_row_view`, `ddb_stmt_bind_int64_step_i64_text_f64`.
- **But the default `step()` uses the slow path:** `statement.dart:872` `bool step()` calls `_bindings.stmtStep` (plain `ddb_stmt_step`), then per-column accessors go through `stmt_value_copy`. The fast-path methods (`stmtStepRowView`, `stmtFetchRowViews`, `stmtBindInt64StepRowView`, `stmtBindInt64StepI64TextF64`) exist as separate methods (`statement.dart:794,835,884,921`) but are not the default. Callers must opt in.
- **Branch support — the only binding that exposes it:** `branch.dart` (824 lines) wraps `ddb_db_execute_on_branch` and `ddb_db_branch_execute_json` (`native_bindings.dart:886,931`). See §2.3.2.
- **Sync + write queue exposed:** `Database.sync` wraps `ddb_db_sync_execute_json` and the public changeset create/apply/inspect/invert APIs; `Database.executeQueued(sql)` and `Database.writeQueueMetrics()` expose the write queue.
- **Feature gaps:** No watches, no extensions.

#### 2.2.6 Java/JDBC — Adequate, no batch fetch

`bindings/java/native/decentdb_jni.c` (JNI shim) + `DecentDBResultSet.java` (1,009) + `DecentDBPreparedStatement.java` (744) + `DecentDBNative.java` (native decls).

- **Row-view caching per step:** `decentdb_jni.c:67 cache_row_view(...)` caches the borrowed `ddb_value_view_t*` array after each `stmt_step_row_view` (`:602`) or `stmt_bind_int64_step_row_view` (`:629`). Per-column accessors (`colInt64`, `colFloat64`, `colText`, `colBlob`, `colDecimalUnscaled`, `colDatetime`, `colIsNull`, `colType`) read from the cached view via `row_view_at` (`:1214`) — so **within a row, per-column reads are cheap (no extra JNI crossing)**. This is a sensible design for JDBC.
- **No batch fetch JNI binding:** There is no JNI binding for `ddb_stmt_fetch_row_views` or `ddb_stmt_fetch_rows_i64_text_f64`. Bulk JDBC reads therefore do one JNI crossing per row (`stmtStepRowView` per `ResultSet.next()`). For OLTP this is fine; for high-throughput bulk reads it lags Python/Node/Dart which expose batch fetch.
- **Write fast paths present:** `stmtExecuteBatchI64`, `stmtExecuteBatchI64TextF64`, `stmtExecuteBatchTyped`, `stmtRebindInt64Execute`, `stmtRebindTextInt64Execute`, `stmtRebindInt64TextExecute`.
- **Metadata rich:** `DecentDBDatabaseMetaData.java` (1,574 lines) exposes JDBC metadata via the JSON introspection APIs.
- **Feature gaps:** No watches, no sync changesets, no branches, no extensions. Java is a natural fit for `RowSetListener`-style change notification; absent.

#### 2.2.7 Web/WASM — Good binary transport, no row-view exposure

`bindings/web/src/` (binary.ts 219, opfs-host.ts 235, index.ts 1,586, probe.ts 270, protocol.ts 442, worker.ts 1,052).

- **Binary result transport:** `binary.ts` decodes a binary result format (magic + version validated, `:36`, `:40`) preferred over JSON. `protocol.ts:2` `ResultTransport = "binary" | "json"`; default is `"binary"` (`worker.ts:532`, `index.ts:1274`). This avoids JSON parse overhead for result sets — a real win in the browser.
- **OPFS-backed VFS:** `opfs-host.ts` provides the browser-persistent VFS — a genuine differentiator over SQLite/WASM in many configurations.
- **No row-view / fetch-row-views exposure:** The worker protocol surfaces `statement_step` (`worker.ts:699`, `index.ts:790`) returning a row, and the binary result transport for `exec`-style calls, but does **not** expose `stmt_row_view` / `stmt_fetch_row_views` / the typed narrow shortcuts through the worker. Large browser reads therefore go through the binary (or JSON) materialized result. A faster path should be designed around explicit lifetime/copy semantics for borrowed row views; `WebAssembly.Memory` is already tracked at `worker.ts:893-895`, but borrowed native pointers should not be handed across the worker boundary casually.
- **Sync changeset helpers present:** `web/src/worker.ts` exposes sync apply helpers (`:925-936`), and `index.ts` exposes relay fetch helpers (`fetchRelay`, `:1093`). Web has a partial relay/apply surface; .NET and Dart expose the fuller native changeset create/apply/inspect/invert surface.
- **Feature gaps:** No watches, no extensions, no write queue, no branches exposed.

#### 2.3.2 Branches — exposed only in Dart (highest-impact gap)

The README headlines "🌿 Branch, Diff, Restore, And Time Travel" as a flagship feature. The ABI exposes it cleanly:

```c
ddb_db_execute_on_branch(db, branch_name, sql, params, params_len, &out_result);
ddb_db_branch_execute_json(db, request_json, &out_json);
```

Yet **only Dart** wraps these (`branch.dart`, 824 lines; `native_bindings.dart:886,931`). Six of seven bindings cannot use a flagship capability. This is the most surprising gap in the matrix and the largest README-vs-reality credibility issue. Dart's `branch.dart` demonstrates the ABI is clean to wrap; the gap is purely binding-side effort.

**Recommendation:** Add `execute_on_branch` / `branch_execute_json` wrappers to at least .NET, Python, Go, and Node (the four most-used embedded bindings). Each is a thin wrapper over an existing ABI call.

#### 2.3.3 Spatial/geospatial — no typed binding surface (second-highest gap)

The README headlines "🗺️ Native Geospatial" with `GEOMETRY`/`GEOGRAPHY`, `ST_*` functions, and `USING spatial` indexes. The ABI supports it:

```c
ddb_stmt_bind_geometry_wkb(stmt, idx, data, len);
ddb_stmt_bind_geography_wkb(stmt, idx, data, len);
// value-view: DDB_VALUE_GEOMETRY, DDB_VALUE_GEOGRAPHY (with data/len as WKB)
```

Yet **no binding** surfaces geometry/geography as a first-class typed value. Callers must hand-encode WKB and read raw bytes. Go (`driver.go:2024-2029`) decodes `DDB_VALUE_GEOMETRY`/`GEOGRAPHY` to `[]byte` — usable but untyped. For a feature this prominent, the primary bindings should offer typed geometry objects (or at minimum documented WKB encode/decode helpers + `ST_GeomFromText`/`ST_GeogFromText` integration):

- .NET: `DbGeometry`/`DbGeography`-style or Microsoft.Spatial types.
- Python: a WKB helper module + integration with `shapely` if optional.
- Go: a `Geometry` struct + WKB marshal/unmarshal.
- Java: JTS `Geometry` interop (optional dependency).

**Recommendation:** Add typed geometry helpers to the primary bindings; at minimum document the WKB contract in each binding's user guide.

#### 2.3.4 Watches / change streams — missing in Java, Node, Dart, Web

The ABI exposes a complete watch API (`ddb_db_watch_table_json`, `..._range_json`, `..._query_json`, `ddb_db_change_stream_json`, `ddb_watch_next_json`, `ddb_watch_close`). Only .NET, Go, and Python wrap it. Java and Node are conspicuous absences:

- **Java/JDBC** is a natural fit for `RowSetListener`/event-driven patterns; a `DecentDBWatch` wrapper exposing `next(timeoutMs)` would be straightforward.
- **Node** is a natural fit for `EventEmitter`-based change streams; the JS layer already wraps streaming-style calls.

**Recommendation:** Add watch wrappers to Java and Node (and Dart/Web as demand warrants).

#### 2.3.5 Lua extensions — only Python

Only Python exposes `ddb_extension_*_json`. The README headlines "🧩 Lua Extensions" as a feature. For embedded bindings (.NET, Node) where users may want to ship manifest-declared extension packages, the absence is notable. Lower priority than branches/spatial but worth noting.

#### 2.3.6 Sync changesets — .NET and Dart strongest; Web partial

Sync is a headline feature ("🔁 Local-First Sync"). .NET (`DecentDBSyncClient.cs`) and Dart (`Database.sync`) expose the native sync JSON and public changeset create/apply/inspect/invert APIs. Web exposes a partial relay/apply surface. Go/Java/Node/Python have no high-level sync changeset wrappers. For a "local-first" database targeting multiple embedded languages, those missing bindings remain a parity gap, but Dart should be counted as a strength here rather than a gap.

### 2.4 Specific Actionable Findings (Bindings)

#### 2.4.1 Go `result_value_copy` per-cell loop (HIGH)

`bindings/go/decentdb-go/driver.go:1344-1360`:

```go
for r := 0; r < int(rowCnt); r++ {
    row := make(map[string]interface{})
    for c := 0; c < int(colCnt); c++ {
        var val C.ddb_value_t
        C.ddb_value_init(&val)
        rc := C.ddb_result_value_copy(result, C.size_t(r), C.size_t(c), &val)
        if rc == C.DDB_OK { ... row[colName] = valueToGo(val) }
        C.ddb_value_dispose(&val)
    }
    rows = append(rows, row)
}
```

This is O(rows × cols) cgo crossings + allocations + disposals. The hot `Rows.Next` path correctly uses `step_row_view`, so `database/sql` users are fine — but any caller using the convenience result-materialization helper pays this cost.

**Fix:** Convert to a prepared statement + `fetch_row_views` (or `fetch_rows_i64_text_f64` for typed schemas) batch fetch, decoding the borrowed view slice once per batch. This removes per-cell init/copy/dispose and amortizes cgo crossings.

#### 2.4.2 Branch API exposure (HIGH — credibility)

Wire `ddb_db_execute_on_branch` / `ddb_db_branch_execute_json` into .NET, Python, Go, Node. Dart's `branch.dart` is the reference implementation. This closes the single largest README-vs-reality gap.

#### 2.4.3 Geometry/geography typed helpers (MEDIUM-HIGH)

Add typed wrappers in the primary bindings (see §2.3.3). The ABI already provides `bind_geometry_wkb`/`bind_geography_wkb` and value-view tags; only binding-side typing is missing.

#### 2.4.4 Web/WASM row-view path (MEDIUM)

Expose a row-view or batch-prefetch path through the worker protocol. Because `ddb_stmt_*_row_view` returns borrowed pointers, this should be designed with explicit lifetime/copy semantics across the worker boundary rather than as a naive pointer handoff. `WebAssembly.Memory` is already tracked at `worker.ts:893-895`; the binary transport is good, but a carefully designed batch path would reduce overhead for large result sets.

#### 2.4.5 Java JNI batch fetch (MEDIUM)

Add JNI bindings for `ddb_stmt_fetch_row_views` and `ddb_stmt_fetch_rows_i64_text_f64`, plus a `ResultSet` batch-prefetch mode. Currently bulk JDBC reads do one JNI crossing per row.

#### 2.4.6 Dart default `step()` path (LOW effort, real win)

`bindings/dart/dart/lib/src/statement.dart:872` `bool step()` should route through `stmtStepRowView` (symbol already loaded at `native_bindings.dart:1024`) so the common path gets the fused fast path instead of `stmtStep` + per-column `value_copy`. Low effort, real win for every Dart caller using the default step API.

#### 2.4.7 Binding feature-parity matrix in docs (MEDIUM — credibility)

The README implies uniform feature support across all seven languages; the reality is a sparse matrix (§2.3.1). Add a "binding feature support" table (like the existing Languages table but per-feature) so users know what is actually available per language. This is a docs-credibility fix, not code.

### 2.5 Bindings — What Is Working Well

- **Python** is a gold standard: C-accelerated fast decode, per-SQL capability caching, every fast-path ABI wired.
- **.NET** has the deepest ecosystem integration of any binding (ADO.NET + Dapper + MicroOrm/LINQ + EF Core + NodaTime + Design/Scaffolding).
- **Go** `rows.Next` hot path is correct (fused `step_row_view`); the slow convenience path is the only real issue.
- **Node** exposes both typed and row-view paths at the JS layer; Knex client uses the typed fast path.
- **Java** row-view caching per step makes per-column reads cheap within a row; metadata support is rich.
- **Web** binary result transport + OPFS VFS are genuine differentiators.
- **The ABI's layered slow/fast design** is the right foundation; the gaps are binding utilization, not ABI limitations. Extending the ABI is rarely needed — wrapping existing symbols is.

### 2.6 Bindings — Suggested Next Steps (highest leverage first)

1. **Wire branch API into .NET, Python, Go, Node** (closes biggest README-vs-reality gap; reference: Dart `branch.dart`).
2. **Fix Go `result_value_copy` per-cell loop** → `fetch_row_views` (driver.go:1344-1360).
3. **Add typed geometry/geography helpers** to the primary bindings.
4. **Add Java JNI batch fetch + design a Web WASM batch/row-view path.**
5. **Route Dart default `step()` through `step_row_view`.**
6. **Add watch/change-stream wrappers to Java and Node.**
7. **Publish a binding feature-parity matrix** in the docs.

### 4.4 Property & Fuzz Testing Gaps

- **Only 3 `proptest!` blocks** across the entire engine. For a database with byte-level encoding (`record/`, `wal/format.rs`, `btree/`, `spatial/ewkb.rs`), property tests are the highest-leverage testing investment available. Round-trip properties (encode→decode == identity), monotonicity (key ordering), and boundary properties (page-size edge cases) are natural fits and largely absent.
- **`wal_fuzz`** exists (`crates/decentdb/src/bin/wal_fuzz.rs`, 196 LOC) — good. But there is **no SQL planner/executor fuzz harness**: no fuzz target for the parser (`sql/parser.rs`, 1,040 LOC, 39 tests), the normalizer (`sql/normalize.rs`, 4,414 LOC), or expression evaluation. SQL fuzzing (random valid/invalid statements) is standard practice for SQL engines and would catch production parser/normalizer/executor panics that integration tests may miss.
- **`miri_smoke`** exists — good for `unsafe` audit (51 `unsafe` occurrences across 7 files).

### 4.5 Specific Unit Test Opportunities (Concrete)

Ordered by leverage (durability/safety first, then correctness, then perf).

#### 4.5.1 WAL coordination — HIGH (Priority #1 surface, 3 tests for 1,199 LOC)

`wal/coordination.rs` currently tests only header/slot serialization. Add:

- **Writer ownership lifecycle:** acquire writer lock when free → succeeds; acquire when held by another PID/token → fails with `DbError::Busy`/`Locked`; release → frees; re-acquire after release → succeeds.
- **Stale-PID reclaim:** writer slot with a PID that no longer exists (or token mismatch) can be reclaimed; live-PID slot cannot.
- **Reader-slot lifecycle:** register reader with snapshot LSN → active; multiple concurrent readers with distinct slots; reader unregister frees slot; slot reuse after free.
- **`.coord` sidecar rebuild:** corrupt or missing sidecar is rebuilt from the primary file without data loss; rebuild is idempotent.
- **Timeout/expiry:** `process_coordination_timeout_ms` enforcement — a stale writer exceeding timeout is reclaimable.
- **Concurrent writer contention:** two threads/processes racing for the writer slot — exactly one wins, the other gets a busy error (no deadlock, no double-acquire).
- **Checksum corruption beyond the header:** corrupt a reader-slot record and assert `Corruption` error (the header test does this for the header only).

This is the single highest-value unit test addition in the codebase: it directly defends Priority #1 (durable ACID) and the cross-process coordination feature highlighted in the README.

#### 4.5.2 Lua extensions / signature trust — HIGH (Safety, 0 in-file tests for 2,646 LOC)

`extensions.rs` `ExtensionManager`:

- **Signature verification:** valid signed package → installs; tampered content (hash mismatch) → rejected; wrong key → rejected; unsigned package with `unsigned_development()` → allowed; unsigned package with strict trust anchor → rejected.
- **Manifest parsing:** well-formed manifest → parsed; missing required field → error; unknown SQL type → error; malformed dependency graph → error; cyclic dependencies → detected.
- **Content-hash trust:** content hash matches → trusted; mismatch → untrusted.
- **Lifecycle:** install → list shows it → enable → enabled → disable → disabled → purge → gone; install twice → error or idempotent per contract; enable nonexistent → error.
- **Runtime limits / permissions:** limits enforced on function registration; permission denied paths.

Use fixture packages (a valid signed `.lua` zip, a tampered one, an unsigned one) committed under `tests/fixtures/` or generated in-test.

#### 4.5.3 Security command parser — HIGH (Safety, hand-rolled sub-parser, 0 in-file parser tests)

`security.rs` `parse_security_command` / `parse_set_audit_context` and the `TokenCursor`:

- **Valid:** `SET audit.context.tenant = 'acme'`, `SET audit.context.actor = 'bob'`, `CREATE POLICY ...`, `DROP POLICY ...`, `ALTER POLICY ...`, `CREATE MASK ...`, etc. → parsed to correct `SecurityCommand`.
- **Invalid:** malformed SQL, missing `IF EXISTS`, unquoted literals, trailing tokens, unknown policy/mask names → typed error, not panic.
- **Redaction:** ensure `parse_literal_value` and audit-context values do not leak unredacted secrets into diagnostics.
- **`is_security_internal_table`:** each internal table name returns true; user tables return false.

This is a hand-written parser inside a security module with integration coverage but no isolated parser tests — parser bugs here are security bugs.

#### 4.5.4 Error taxonomy + diagnostic path redaction — HIGH (Public API, safety)

`error.rs` and `unit_error_taxonomy_tests.rs`:

- **Path redaction (`DbDiagnosticPath::redacted` / `redacted_with_display`):** absolute paths, paths with username (`/home/steven/...`), Windows paths, relative paths, empty paths → redacted form does not contain the raw username/home prefix. This is safety-relevant: diagnostics must not leak filesystem structure. Snapshot tests (`insta` is already a dev-dep) are ideal here.
- **`DbDiagnosticOpenOptions::from_raw`:** parse `key=value` separated by whitespace/comma/semicolon (per `decentdb.h:181`); unknown keys, empty values, duplicate keys, missing `=` → defined behavior.
- **`DbDiagnosticAuditContext::from_pairs`:** empty, single, multi, duplicate keys.
- **Builder chaining:** `with_relation`/`with_column`/`with_constraint`/`with_index`/`with_policy`/`with_branch` produce the expected context; redaction applies to each.
- **Error code stability:** the existing `error_codes_are_stable` test should be expanded to assert every public `DbErrorCode` variant's numeric code against a frozen map (regression guard for the stable-code contract).

#### 4.5.5 TDE / encrypted VFS — HIGH (Safety, 3 tests total)

`vfs/encrypted.rs` + `tests/tde_security_tests.rs`:

- **Round-trip:** write N bytes at offset → read back identical across page boundaries, exact-page-size writes, and sub-page writes.
- **Wrong key:** open encrypted file with a different key → reads fail with a typed error (not corrupt plaintext, not panic).
- **Partial page / torn write:** simulate a partial last page (via `vfs/faulty.rs` fault injection) → recovery rejects or recovers correctly.
- **Metadata sync:** `sync_metadata` after encrypted write persists the encryption header.
- **`write_all_at_many`:** batched scattered writes encrypt each segment correctly and are readable.
- **Key zeroization:** `zeroize` is a dependency; assert key material is zeroized on drop (the crate advertises durable row policies / transparent encryption).

#### 4.5.6 Record value operations — MEDIUM-HIGH (Correctness, plan.md issue lives here)

`record/value.rs`:

- **Cross-type comparison (the open `plan.md` item):** `PartialOrd`/`cmp_values` between `Decimal` and `Float64`, `Int64` and `Decimal`, `Int64` and `Float64`, `Timestamp` and `Date`, etc. — assert defined ordering or typed error per the SQL spec. The `plan.md` Decimal-vs-Float64 comparison bug should be reproduced and fixed with a regression test here.
- **`approximate_heap_bytes`:** snapshot per `Value` variant (Null, Int64, Float64, Bool, Text(empty/nonempty), Blob, Decimal, Uuid, Timestamp, Enum, IpAddr, Cidr, Geometry, Geography, Interval) — guards memory accounting regressions.
- **Decimal aggregation:** `SUM`/`AVG` over `Decimal` (the second `plan.md` item) — unit test the accumulator directly, not just via SQL.
- **Decimal overflow:** `SUM` overflow on `Decimal` → typed error, not panic.
- **Text/Blob boundary:** zero-length, max-inline, overflow-threshold sizes.

#### 4.5.7 Tracing sink ring buffers — MEDIUM (Off-by-one prone, 0 tests)

`tracing/sink.rs`:

- **Ring-buffer boundaries:** fill exactly to capacity, over-fill (oldest evicted), snapshot returns most-recent N in order.
- **`reset(family)`:** resets only the named family; other families untouched.
- **`record_slow_query` / `record_lock_wait` / `record_index_usage`:** records appear in the correct snapshot family; `any_enabled` gating works; disabled families do not record.
- **Session lifecycle:** `mark_active`/`mark_in_transaction`/`mark_closed` transitions produce correct `sessions_snapshot`.

#### 4.5.8 B+Tree read path — MEDIUM (Priority #2, 0 tests in `btree/read.rs`)

`btree/read.rs` (221 LOC, 0 tests) while `write.rs`/`table.rs` have a few:

- **Point lookup:** hit, miss, empty tree.
- **Range scan:** forward/reverse, inclusive/exclusive bounds, empty range, full scan.
- **Tree shapes:** single-leaf, shallow (root+leaves), multi-level; verify traversal order is sorted.
- **Page-boundary keys:** keys exactly at leaf boundaries.

#### 4.5.9 Reactive subscription JSON — MEDIUM (Headline feature, 0 in-file serialization tests)

`reactive.rs`:

- **`RowChange::to_json` / `TableChange::to_json` / `InvalidationEvent` / `ChangeStreamEvent` / `LaggedWatchEvent::to_json`:** snapshot with `insta` (already a dev-dep) for stable wire format — wire-format drift would silently break the C ABI watch consumers and all 7 bindings.
- **`WatchHandle` lifecycle:** `try_recv` before any event → `Ok(None)`; `recv_timeout` expiry → `Ok(None)`; `close` → subsequent recv errors; `next_json_timeout` produces valid JSON.
- **Lagged watch:** when the channel lags, a `LaggedWatchEvent` is delivered and subsequent events resume.

#### 4.5.10 C ABI error/edge paths — MEDIUM-HIGH (Single shared boundary, 28 tests for 100 fns)

`c_api.rs`: for the high-traffic FFI functions, add isolated tests for:

- **Null/bad handles:** `ddb_db_execute(NULL, ...)`, `ddb_stmt_step(NULL, ...)`, `ddb_result_free(NULL)` → defined `DDB_ERR_*`, no crash.
- **Out-of-range indices:** `ddb_stmt_bind_int64(stmt, 0, ...)` (0 is invalid, API is 1-based), `ddb_result_value_copy(result, row >= row_count, ...)` → error.
- **Double free / use-after-free:** `ddb_result_free` twice → second returns error, no double-free; `ddb_db_free` then use → error.
- **ABI version mismatch:** simulate an older `ddb_abi_version` and assert graceful unsupported errors (the bindings already degrade on missing symbols; the engine side should match).
- **Thread-safety of free:** the header says "Do not call ddb_result_free concurrently" — add a test documenting the contract and that concurrent *execute* (readers) is safe.

Binding smoke tests (1–4 files each) cannot substitute for these because they exercise the happy path through a language shim.

#### 4.5.11 Planner/physical plan choice — MEDIUM (Silent perf regressions, 0 tests)

`planner/physical.rs` (458 LOC, 0 tests):

- **Plan selection regression guards:** for a set of representative queries (point lookup on PK, range scan, join, aggregate), assert the chosen physical operator (index seek vs scan, join strategy). These are golden tests that fail when plan choice regresses — the cheapest way to catch the silent perf regressions the benchmark suite only catches at release time.

#### 4.5.12 Write queue — MEDIUM (Priority #1, 0 in-file tests)

`write_queue.rs`:

- **Enqueue/dequeue ordering, capacity bound, timeout enforcement, group-commit batching, metrics snapshot fields** (`group_commit_batches`, `group_commit_syncs`, `physical_syncs_saved`, `total_queue_wait_ns`) — unit-test the queue mechanics directly rather than only via `ddb_db_execute_queued` end-to-end.

### 4.6 Property/Fuzz Test Opportunities

- **`proptest` round-trips** (encode→decode == identity) for: `record/row.rs` row encoding, `wal/format.rs` WAL records, `btree/` key encoding, `spatial/ewkb.rs` WKB, `record/value.rs` all `Value` variants. The infrastructure (`proptest` 1.5 dev-dep) is present but barely used (3 blocks).

- **SQL fuzz harness:** a new fuzz target (`cargo fuzz` or a `#[fuzz_test]`-style binary) generating random SQL from the grammar and asserting no panic. Start with expression evaluation and the normalizer.
- **WAL fuzz expansion:** the existing `wal_fuzz` should cover torn-write-at-every-offset and partial-page recovery, not just the current scope.
