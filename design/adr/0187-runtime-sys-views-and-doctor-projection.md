# Runtime `sys.*` Views And Doctor Projection
**Date:** 2026-06-09
**Status:** Accepted

### Decision

DecentDB will expose runtime tracing and Doctor findings primarily through
read-only virtual SQL inspection views that follow the ADR 0163 `sys.*`
contract.

The initial runtime views are:

- `sys.sessions`
- `sys.slow_queries`
- `sys.lock_waits`
- `sys.index_usage`
- `sys.doctor_findings`

These views are virtual inspection surfaces. They do not create a durable `sys`
schema, do not create user-visible catalog objects, do not write rows, and do
not persist telemetry. They materialize snapshots from owned in-memory runtime
state, Doctor reports, existing operational metrics, and catalog metadata.

Only the documented `SELECT * FROM sys.name` shape is part of the initial
stability contract. Projections, predicates, joins, ordering, limits, and bind
parameters may work as implementation details, but they are not the first
compatibility promise unless documented separately.

`sys.sessions` exposes current and recent process-local connection/session
state. It does not imply a server process, global connection pool, or
cross-process session registry. Closed-session rows are retained only in a
bounded recent-session buffer.

`sys.slow_queries` exposes recent statement events that crossed the configured
slow-query threshold or were captured by an explicitly enabled statement
sampling profile. It must include enough metadata to distinguish statement
execution time from lock-wait time when that breakdown is known. Failed
statements may be included if they crossed the threshold, but their status and
structured error code must be explicit.

`sys.lock_waits` exposes recent waits that crossed the configured lock-wait
threshold or ended in timeout/cancellation. It must classify wait source only
when the engine can prove the source. The stable source vocabulary begins with:

- `write_queue`
- `sql_write_lock`
- `transaction_writer`
- `wal_append`
- `wal_checkpoint`
- `process_coordination`
- `process_reader`
- `schema_lock`
- `index_build`
- `index_verify`
- `storage_io`
- `vfs_file_lock`
- `unknown`

`unknown` is an acceptable source. Misclassification is worse than incomplete
classification.

`sys.index_usage` is aggregate-first. It exposes observed index use over the
current bounded observation window, including read uses, write maintenance uses,
constraint uses when known, last observed use, and trace-window metadata. It is
not proof that an index is globally unused. It is evidence for advisors.

`sys.doctor_findings` projects the latest explicit Doctor report cached in the
current process. A plain `SELECT * FROM sys.doctor_findings` must not
unexpectedly run deep Doctor checks, index verification, runtime advisors, or
expensive scans. If no report has been generated in the process, the view
returns zero findings and snapshot metadata must make the absence of a report
observable through the report fields or a companion summary surface.

Doctor refresh from SQL is explicit. The accepted direction is a `PRAGMA doctor`
family or equivalent table-valued function that runs a bounded Doctor collection
and updates the latest in-memory Doctor report. The implementation spec will
finalize syntax, but the semantic contract is:

- explicit refresh may perform Doctor work;
- simple selection from `sys.doctor_findings` is a projection;
- runtime advisors run only when requested by Doctor/advisor options or a named
  diagnostic profile;
- expensive index verification remains opt-in, consistent with current Doctor
  behavior.

`sys.doctor_findings` must project the existing Doctor finding model and runtime
advisor extensions with stable row fields for:

- report schema version;
- report status;
- finding ID;
- severity;
- category;
- title;
- message;
- evidence JSON;
- recommendation summary;
- recommendation command or SQL when safe;
- `safe_to_automate`;
- advisor ID when generated by an advisor;
- confidence when generated by an advisor;
- trace-window metadata when derived from tracing;
- path redaction mode.

Runtime views must label internal rows or exclude internal inspection work by
default. Materializing a `sys.*` view, running Doctor, or generating advisor
reports must not recursively create ordinary user-facing slow-query or advisor
evidence.

The implementation must snapshot runtime view data under short locks and return
owned row data to SQL execution. It must not hold internal WAL, pager,
coordination, session, trace-buffer, or Doctor locks while formatting SQL text,
serializing JSON, running advisors, or recursively executing SQL.

### Rationale

SQL views are the common inspection surface across Rust, C ABI callers,
bindings, CLI tooling, tests, and Decent Bench. ADR 0163 already established
the model for operational metrics: stable read-only views, no persistent schema,
and conservative compatibility for documented `SELECT *` forms.

Runtime tracing needs the same shape. A Python application, a Java service, the
CLI, and Decent Bench should not need different APIs to answer "which query was
slow?" or "which lock source caused the wait?"

Doctor projection as rows lets automation and bindings consume health findings
without shelling out. Keeping `sys.doctor_findings` as a projection avoids
surprising users with expensive work from an ordinary `SELECT`.

Index usage is aggregate-first because per-access events can be too noisy and
too expensive. Slow queries and lock waits remain event-first because their
value is historical evidence about specific operations.

### Alternatives Considered

1. **Expose runtime tracing only through Rust APIs.** Rejected. That would make
   every binding and tool build custom wrappers and would fragment the
   inspection contract.
2. **Expose runtime tracing through C ABI JSON only.** Rejected for the primary
   surface. JSON snapshots may be useful later, but SQL is easier for users and
   tests and already works across bindings.
3. **Make `sys.doctor_findings` run Doctor automatically on every query.**
   Rejected. Doctor can be expensive, especially with index verification and
   runtime advisors. Refresh must be explicit.
4. **Persist Doctor reports in hidden tables.** Rejected for this phase. It
   would create write paths and retention policy questions unrelated to runtime
   inspection.
5. **Expose per-index usage as one event per access.** Rejected for the first
   implementation. Aggregates are cheaper and more useful for advisors.
6. **Hide failed slow statements.** Rejected. Failed statements can be the most
   important slow events, as long as status and error code are explicit.

### Trade-offs

- `SELECT *` stability is narrower than a full virtual-table contract, but it
  preserves implementation freedom while the engine's system-view machinery
  evolves.
- A cached Doctor projection means SQL users need an explicit refresh step.
  That is preferable to hidden expensive work.
- Process-local session rows cannot explain another process's active sessions.
  Cross-process process coordination metrics remain the source for that
  category of facts.
- Aggregate index usage can hide individual query context. Slow-query rows and
  advisor evidence can link back to statement fingerprints and event IDs where
  available.

### Consequences

- `docs/api/sql-functions.md` must document every new view, column, nullability
  rule, redaction behavior, and reset behavior when implementation lands.
- Runtime view tests must cover disabled tracing, enabled tracing, reset,
  eviction, redaction, and concurrent snapshotting.
- Doctor tests must verify that `sys.doctor_findings` does not run expensive
  checks implicitly.
- The CLI and Decent Bench should consume the same underlying snapshot/report
  APIs used by the SQL views.
- Any decision to make `sys.doctor_findings` run Doctor on demand by default
  requires a follow-up ADR.
