## 2. Strengthen WAL Testing

### Phase 2a: Property-based tests for delta encoding

File: `crates/decentdb/src/wal/delta_tests.rs` (new, `#[cfg(test)]`)

Leverage the existing `proptest` dev-dependency (already in Cargo.toml). Add:

1. **Delta roundtrip**: For any two byte vectors of equal length (0..4096), `encode_page_delta(base, updated)` produces a delta payload, and applying that delta to `base` produces `updated`.
2. **No-op identity**: Encoding a delta where `base == updated` should produce `None` (or a minimal delta that roundtrips).
3. **Size bounds**: Delta payloads never exceed `DELTA_FRAME_PAYLOAD_SIZE` (512 bytes).
4. **Cross-platform determinism**: For fixed inputs, the encoded delta is byte-identical.

### Phase 2b: Property-based tests for WAL frame format

File: `crates/decentdb/src/wal/format_tests.rs` (new, `#[cfg(test)]`)

1. **Frame roundtrip**: For any `page_id: u32` and `payload: Vec<u8>` of length matching `page_size`, encode/decode produces the same data.
2. **Header roundtrip**: `WalHeader` encode/decode identity.
3. **Invalid frame types**: Random u8 values outside 0-3 produce `DbError::Corruption`.
4. **Frame size consistency**: `FrameType::payload_size()` matches the actual frame layout (header + payload + trailer).

### Phase 2c: Expanded crash-recovery scenarios

File: `crates/decentdb/tests/unit_wal_recovery_edge_cases_tests.rs` (extend existing)

1. **Crash mid-checkpoint**: Write pages, start checkpoint, fail mid-way, recover — verify all committed data is intact.
2. **Delta-encoded crash**: Commit delta frames, crash before commit record, verify uncommitted deltas are discarded.
3. **Reader-with-snapshot during checkpoint**: Hold snapshot, trigger checkpoint, verify reader still sees its snapshot and WAL retains needed frames.
4. **WAL growth + truncation cycle**: Many commits → checkpoint → verify WAL is header-only → new commits → verify data survives reopen.
5. **Concurrent writer + background checkpointer**: Stress the auto-checkpoint interaction with active writing.

### Phase 2d: Fuzz target for WAL recovery

File: `crates/decentdb/src/bin/wal_fuzz.rs` (new binary, not in default workspace builds)

A standalone binary that:
1. Generates a random valid DB + WAL file pair
2. Introduces random corruptions (flipped bits, truncated bytes, wrong sizes, bad magic)
3. Calls `initialize_or_recover` and asserts either clean recovery or a `DbError::Corruption` (never a panic or segfault)

Run periodically via CI (can be added to `memory-safety-nightly.yml` as a 5-minute fuzz job).

## 3. Resolve Known Test Gaps

### Issue 3a: Decimal ordering defect

**Root cause** (`crates/decentdb/src/record/key.rs:116-137`):  
The `sortable_decimal_bytes` function uses biased exponent encoding with an inversion for negative numbers. The inverted exponent ordering is incorrect because for negative values, larger magnitude (more negative) should sort first, but the current inversion causes smaller magnitudes to sort first.

Example: `-99999999.99` (more negative) should come before `-0.01`, but current encoding sorts `-0.01` first.

**Fix**: When `scaled < 0`, the exponent should NOT be inverted. Instead, the entire ordering should be reversed for negative numbers. The simplest correct approach: use the same exponent encoding for both signs but prepend a sign byte (0 for negative, 2 for positive) so that the lexicographic order matches numeric order. Alternatively, for negative numbers, NOT inverting the exponent and complementing the digits is sufficient — the issue is just the exponent inversion.

**Specific change**: In `sortable_decimal_bytes` at line 128, for negative numbers, emit the biased_exponent as-is (not inverted) and invert the digits. The current inverted exponent is the problem: `inv_exp = u16::MAX - biased_exp`, which reverses the natural ordering. Just emitting `biased_exp` directly (with digit inversion) gives correct ordering for negative numbers because larger magnitude → larger biased_exp → larger byte value → sorts later in ascending lexicographic order, which is exactly opposite of what negative numbers need. Wait, that doesn't work either because then positive and negative exponent byte values overlap in range.

**Correct fix**: Keep current approach but remove the exponent inversion. For negative numbers:
```rust
output.push(0);
output.extend_from_slice(&biased_exp.to_be_bytes());  // NOT inverted
output.extend(digits.bytes().map(|byte| u8::MAX - byte));
```
This works because:
- For negative numbers, larger magnitude → more digits → larger biased_exp. But we want larger negative magnitude to sort FIRST. Since the sign byte is 0 for negative, and 2 for positive, all negatives come before positives. Within negatives, a LARGER biased_exp makes the bytes LARGER, which means it sorts AFTER. But we want -99999999.99 BEFORE -0.01. So this doesn't work either.

The real issue is that the current encoding tries to use a single sortable byte sequence where the exponent and sign interact. The standard approach (used by SQLite's decimal encoding, among others) is:
- For negative numbers: negate the exponent, invert the mantissa digits
- The key insight: negative values need their entire encoding to sort in reverse

**Implementation**: For negative values, compute `inverted_exp = (2 * 1024 - 1) - biased_exp` (mirror around midpoint) and keep digit inversion. This correctly maps larger magnitude negative numbers to smaller byte values.

### Issue 3b: Zero-byte WAL test

**Root cause**: Not a bug — a design choice documented in `src/wal/recovery.rs:39-40`. A zero-byte WAL is treated as clean initialization, not corruption. The ignored test incorrectly asserts the opposite.

**Fix**: Convert the ignored test into a non-ignored test that asserts the **current correct behavior**: opening a DB with zero-byte WAL **succeeds** (the engine writes a fresh header). Keep the existing `zero_byte_wal_is_reinitialized_and_db_opens_successfully` test (line 197) and remove the ignored test, replacing it with a comment explaining the design choice.

### Issue 3c: INSERT DEFAULT VALUES

**Root cause**: Feature does not exist in the parser or executor. The engine returns "INSERT is missing its source rows" when encountering `INSERT INTO t DEFAULT VALUES`.

**Options**:
1. **Implement the feature** — Add support for `DEFAULT VALUES` in the SQL parser AST, normalizer, and executor. For each column with a `default_sql`, evaluate the default expression. Non-trivial but useful SQL compatibility.
2. **Remove the ignored test** — If the feature is not planned for the near term, delete the test to avoid perpetual ignored-test noise.

**Recommendation**: Remove the test for now and file a tracking ADR if/when this feature is desired. The test pollutes the test runner output without value since there's no implementation planned.
